# 변경 내역 (Changelog)

이 저장소(easyproduct 스킬 세트)의 버전별 변경 내역이다. 버전·날짜를 언제 확정하는지는 `VERSIONING.md`를 따른다.
형식은 [Keep a Changelog](https://keepachangelog.com/ko/) 관례를 참고하되 한국어로 정리한다.

- **버전 번호**는 작업 브랜치가 `beta`에 머지될 때 확정된다.
- **릴리즈 날짜**는 단일 필드로, 머지될 때마다 기록된다 — `beta` 머지 시 beta 머지일, `main` 머지 시 그 날짜로 갱신.

> `beta` 머지 시 버전 번호와 beta 머지일이 확정된다.

- `install.ps1`을 UTF-8 BOM으로 저장 + 버전 추출을 ASCII 정규식으로 변경 → Windows PowerShell 5.1에서도 한글 출력·버전 인식이 정상.

## [0.2.0] — 2026-07-15 (beta)

> 상태: `beta` 머지로 **버전 0.2.0 확정**, 릴리즈 날짜 = beta 머지일(2026-07-15).
> 릴리즈 날짜는 단일 필드이며, `main` 머지(정식 릴리즈) 시 그 날짜로 갱신된다.

### Added (추가)
- **data-model: 그룹별 인라인 JSON 미러** — 각 그룹 표 바로 아래에 그 구조를 담은 ```json 블록을 상시 포함(소프트웨어가 md 파싱 없이 구조를 읽도록). **표가 원본(SSOT), JSON은 재생성 미러**(손으로 고치지 않음).

### Changed (변경)
- **data-model: JSON 표기 통일** — 그룹/필드 오브젝트 스키마 + 영문 타입 enum(`string`/`number`/`datetime`…). 기존 "④ json 추출"을 **인라인 미러를 한 파일로 묶기**로 재정의하고, 결과 회신 구조체도 같은 스키마로 통일(이전의 상시-JSON 금지 원칙을 "표 SSOT + JSON 미러 재생성"으로 개정).
- **data-model 버전업** — 인라인 JSON이 없는 옛 문서에 표에서 **gap-fill 생성**하는 절차 추가, suite "오래된 구조" 트리거·데이터모델 재진입 라우팅에 반영.

## [0.1.0] — 2026-07-12 (beta)

> 상태: `beta` 머지로 **버전 0.1.0 확정**, 릴리즈 날짜 = beta 머지일(2026-07-12).
> 릴리즈 날짜는 단일 필드이며, `main` 머지(정식 릴리즈) 시 그 날짜로 갱신된다.

### Added (추가)
- **`easyproduct-scenario` 스킬 신설** — 사용자 여정 시나리오를 요구·설계·테스트를 잇는 "흐름의 축"으로 추가.
  - Given/When/Then + 스텝별 추적 링크 형식(`SCN.*` ID), 생애주기(스케치→이관→파생→승격→검증).
  - 두 거버넌스 규칙(결정 이관·링크-온리)으로 SSOT 보존.
  - 정합·추적 점검(죽은 링크는 확실히, 의미 모순은 후보만) 절차.
  - 필요 여정을 **기획서 핵심 행동·핵심 유스케이스에서 도출**하고, 커버리지 검사를 **필수**로(누락=테스트 구멍) 승격.
  - 시나리오 파일에 **mermaid flowchart 파생 뷰**(여정을 한눈에) 추가.
  - **게이트키퍼 역할**: 앞 게이트(Stage 1.5 스케치, 화면 전 조기 결함 발견)와 뒤 게이트(파생·검증)를 두고, 설계 결함은
    **회귀 후보와 함께 보고 후 사용자가 진행을 결정**(시나리오는 제기·라우팅 제안만, 자동 재설계 금지).
  - **작성 예시(`references/example-walkthrough.md`)** 추가: 채워진 시나리오(GWT+추적 링크+mermaid) + 게이트키퍼 보고의 실제 모습(참고 패턴 겸 확인용).
- **정책 규칙 참조용 안정 ID 체계 `POL.*`** — `easyproduct-policy-legal`에 도입.
  시나리오·화면 설계서·유스케이스가 정책 규칙을 이름이 아니라 ID로 참조 가능.
- **각 스킬 README.md** 추가.
- **버전·릴리즈 규칙 문서**(`VERSIONING.md`)와 이 `CHANGELOG.md` 추가.

### Fixed (검토 반영 — 논리적 구멍 6종)
- **유스케이스 안정 ID `UC.도메인.목표`** 도입(ia-designer) — 시나리오가 유스케이스를 이름이 아니라 ID로 참조(정책 `POL.*`·기능 `FEAT.*`와 대칭).
- **참조 표기 `이름(ID)`** 규칙 — 링크를 사람이 읽는 이름 + ID 함께(예: `메인 홈(FEAT.home.landing)`). 새로 만들거나 수정하는 문서에만 적용, 기존 문서 미적용은 결함 아님.
- **링크-온리 강화** — 예시·템플릿에서 정책 수치("24시간")를 문장에 복사하던 것 제거, 값은 원본만 두고 가리키기.
- **Stage 1.5 스케치 배선** — 스케치 산출물을 Stage 2(IA) 입력으로 흡수, 정책·데이터 결정은 뒤 Stage로(이관 타이밍 정정), handoff-guide 반영.
- **커버리지 "핵심 유스케이스" 정의 고정** — 기획 핵심행동(1순위) + IA 핵심흐름 파생 유스케이스.
- **문서 정합성** — 시나리오 도입 게이트를 "IA 이상"으로 확장, README 흐름도 순서(기능 관계도→시나리오)·명명 규칙(SCN.*/UC.*)·suite 설명 정정.

### Migration (기존 문서 업그레이드 경로)
- **옛 문서 세트 업그레이드 지원**: suite의 "오래된 구조" 감지에 `POL.*` ID 없음·시나리오 없음을 추가.
- policy-legal: 규칙 ID가 없는 기존 정책서에 **`POL.*` ID를 gap-fill로 부여**하는 버전업 절차 추가.
- scenario: 시나리오가 없던 옛 세트도 **기존 확정 문서에서 처음 파생**해 도입할 수 있음을 명시.

### Changed (변경)
- `easyproduct-suite`: 파생 시나리오 옵트인(3.8-D)과 Stage 4 시나리오 정합·추적 점검(3-1) 배선,
  색인·인계 템플릿에 시나리오 반영.
- `easyproduct-doc-builder`: GWT 배제 원칙이 비전문가 기획 단계 한정임을 명시(정형 시나리오는 scenario 스킬 소관).
- `easyproduct-ia-designer`: 유스케이스↔시나리오 경계, 읽기 전용 파생 원칙 공유 명시.
- `easyproduct-screen-design`: 시나리오가 있으면 화면 도출 드라이버로 사용(화면 SSOT는 글 정의).
- 루트 README: 스킬 7개 반영, 명명 규칙에 `POL.*` 추가.
