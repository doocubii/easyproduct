## 기계 블록을 설계·수정할 때 — 핵심 원칙 (빈 껍데기 금지)

문서에 기계 블록을 새로 만들거나 고칠 때, **그 블록의 존재 이유인 필드를 비어 있게 두지 말 것.**

- **스키마에서 필수로 강제한다.** 블록의 실제 값을 담는 필드(참조 목록·항목 목록 등)는 `required`에
  넣고, 배열이면 `minItems: 1`을 건다. 식별자(`id`)만 있고 내용이 빈 엔트리가 스키마를 통과하면 안 된다.
  (예: 화면 블록의 `components`, 시나리오의 `refs` — 둘 다 필수·최소 1개.)
- **스킬 지침에서 "산문을 빠짐없이 미러하라"를 명시한다.** "블록을 넣는다"로 끝내지 말고, 산문의 어느
  부분(예: 화면 "보이는 것"의 모든 `UI.*`, "데이터 정의"의 모든 변수)을 그 필드에 **전부** 담고 **비우지
  말라**고 못박는다.
- **왜:** 이 내용 필드들이 크로스도큐먼트 참조 점검의 근거다. 비면 점검할 게 없어 "있으나마나한 블록"이
  된다 (2026-07 실제 결함: 화면 엔트리가 `{id, feat}`만 담고 컴포넌트·데이터를 전부 버린 사례).

**판단 기준:** "이 엔트리에서 식별자를 빼면 남는 게 있는가?" 없으면 무의미하다 — 내용 필드를 필수화하라.

## 규약·근거 문서 (재파생하지 말고 이 문서들을 따를 것)

- **기계 층 계약** — frontmatter 필드 의미·`doc_type`/anchor 레지스트리·참조 라우팅·점검 알고리즘:
  `skills/easyproduct-suite/references/checker-guide.md`. 참조 점검기: `skills/easyproduct-suite/scripts/check-docs.mjs`
  (무의존 Node, suite가 실행. 프로젝트로 복사 안 함, 소스는 요청 시 제공).
- **버전·릴리즈 규칙**: `VERSIONING.md`. 세트 전체가 한 버전을 공유하고, 버전은 `beta` 머지 시 확정한다.
  **공개·자기보고 표기(루트 `README.md`·각 `SKILL.md` 메타)에는 스테이지(beta/main) 라벨·문구를 넣지 않는다 —
  버전·날짜 값만.** 스테이지·갱신 규칙은 `VERSIONING.md`에만.
- **브랜치**: 작업은 `beta`에서 분기(`feature/*`), 완료 시 `beta`로 머지. 커밋은 자유, **푸시는 확인 후.**
- **산출 문서 원칙**: 비전문가 언어, 초안 우선(화면 설계서만 처음부터 확정 명세), 항상 파일로.
  스킬 간 참조는 최소화 — 각 스킬은 독립 동작하며 다른 스킬의 존재를 전제하지 않는다.
  세부는 각 스킬 `SKILL.md`를 따른다.
