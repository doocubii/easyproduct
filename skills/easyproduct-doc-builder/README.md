# easyproduct-doc-builder

> 개발 지식이 없는 사람도 대화만으로, 사람이 읽기 쉬운 기획서·설계서를 완성하도록 돕는 스킬입니다. 어려운 형식 대신 읽기 편한 문서로 보여주되, 그 안에 spec-kit이 나중에 추출할 내용을 모두 담습니다.

[easyproduct 스킬 세트](../../README.md)의 일부입니다.

## 무엇을 하나
- 기술 용어 없이, 단계적 대화로 아이디어를 **읽기 쉬운 기획서**로 정리합니다.
- 확정된 기획서를 바탕으로 **읽기 쉬운 설계서**를 만듭니다. 기술 항목은 3~4개만 묻고 나머지는 추론해 채운 뒤 확인받습니다.
- 사용자가 가져온 메모·자료·기존 문서에서 먼저 내용을 **추출**하고, 빈 곳·얇은 곳만 채웁니다(풍부화).
- 필요할 때만 사업계획서·사용자 정의·제품 원칙·유저 스토리 같은 **선택 모듈**을 제안해 정리합니다.
- 마무리 전 체크리스트로 spec-kit이 추출할 내용이 빠지지 않았는지 조용히 점검합니다.

## 언제 쓰나
- "앱/웹사이트/서비스/프로그램을 만들고 싶다"고 하지만 기술 지식·경험이 없어 보일 때
- "기획서 만들어줘", "설계서 써줘", "뭐부터 시작하지?"처럼 아이디어를 구체화하려 할 때
- spec-kit, spec-driven development(SDD)로 프로젝트를 시작하려 할 때
- 미리 만든 메모·기획안을 가져왔거나, 이전에 만든 문서를 다시 풍부하게 만들고 싶을 때

## 산출물
- **읽기 쉬운 기획서** (`assets/기획서-템플릿.md` 구조)
- **읽기 쉬운 설계서** (`assets/설계서-템플릿.md` 구조)
- 필요 시 선택 모듈 문서: 사업계획서 · 사용자 정의 · 제품 원칙 · 유저 스토리

## 단독 실행 vs 세트
- **단독으로도** 동작합니다. 대화 모드에서 한 항목씩 인터뷰하며 문서를 함께 만들어 갑니다.
- 상위 오케스트레이터(easy-dev-suite)가 **자동(초안) 모드**로 호출하면, 인터뷰 없이 자료 + 합리적 가정으로 기획서를 먼저 완성한 뒤 요약 카드로 확인만 받습니다.
- 세트 안에서는 **가장 앞 단계** — 아이디어를 문서로 정리하는 출발점입니다. 형식 변환(spec.md / plan.md)과 이후 단계는 개발 단계에서 알아서 진행됩니다.

## 구성 파일
- `SKILL.md` — 스킬의 목적·원칙·전체 흐름과 인터뷰 단계 정의
- `assets/기획서-템플릿.md` — 최종 기획서의 읽기 쉬운 구조
- `assets/설계서-템플릿.md` — 최종 설계서의 읽기 쉬운 구조
- `assets/사업계획서-템플릿.md` — 사업계획서 선택 모듈 템플릿
- `assets/사용자정의-템플릿.md` — 사용자 정의 선택 모듈 템플릿
- `assets/제품원칙-템플릿.md` — 제품 원칙 선택 모듈 템플릿
- `assets/유저스토리-템플릿.md` — 유저 스토리 선택 모듈 템플릿
- `references/optional-modules.md` — 각 선택 모듈을 언제 제안하고 어떻게 묻는지(인력·예산 추정법 포함)
- `references/speckit-fields.md` — spec-kit이 추출할 내용이 빠졌는지 점검하는 체크리스트
- `references/spec-template.md` — spec-kit 원본 spec 구조(참고용)
- `references/plan-template.md` — spec-kit 원본 plan 구조(참고용)

## 세트 안에서의 위치
- 세트의 **시작점**입니다. 여기서 만든 기획서·설계서가 이후 스킬들의 재료가 됩니다.
- 여러 화면·기능을 관통하는 **여정(시나리오)**은 이 스킬이 아니라 `easyproduct-scenario`가 맡습니다. 이 스킬은 여정을 억지로 늘려 쓰지 않습니다.
- 유저 스토리는 단일 요구의 가치("누가·무엇을·왜")를 다루며, 요청이 있거나 오케스트레이터가 켤 때만 만듭니다.
