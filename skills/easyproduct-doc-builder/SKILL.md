# 개발 기획서 만들기

## 목적

기술 지식이 전혀 없는 사람도, AI와의 **단계적 대화**만으로 자기 아이디어를
**읽기 쉬운 기획서와 설계서**로 완성하게 돕는다.

최종 산출물은 spec-kit의 빡빡한 형식(영문 헤더, FR-001, Given/When/Then 등)이 **아니다.**
그렇게 만들면 정작 사용자가 자기 문서를 못 알아본다. 대신 **사람이 읽고 고개를 끄덕일 수 있는
문서**를 만들되, 그 안에 spec-kit이 필요로 하는 내용이 빠짐없이 담기게 한다.
형식 변환(spec.md / plan.md 생성)은 나중에 **spec-kit이 알아서** 한다.

## 핵심 원칙

1. **비전문가 언어.** 전문 용어(API, 프레임워크, 스키마 등)를 먼저 꺼내지 않는다. 불가피하면 한 줄로 풀어 쓴다.
2. **읽기 쉬운 산출물.** spec-kit의 표기법을 사용자에게 보여주지 않는다. 요구사항은 "이 서비스가 할 수 있어야 하는 것들"처럼 평범한 목록으로, 동작은 "이럴 때 이렇게 된다"는 문장으로 쓴다.
3. **spec는 묻고, plan은 적게 묻는다.** 기획 내용은 누구나 답할 수 있으니 평이하게 묻는다. 설계의 기술 항목은 사용자가 모르므로 **3~4개만 묻고 나머지는 추론해 채운 뒤 반드시 확인**받는다.
4. **한 번에 한 주제.** 질문을 한꺼번에 쏟지 않는다. 한 번에 한두 개씩, 답을 듣고 다음으로.
5. **게이트.** 기획서를 사용자가 "네, 맞아요"로 확정하기 전에는 설계서로 넘어가지 않는다.
6. **모르면 비워두지 말고 기록한다.** 결정 못 한 항목은 기획서의 "가정" 또는 설계서의 "아직 안 정한 것"에 적는다.
7. **있는 건 묻지 않는다.** 사용자가 메모·자료·기존 문서를 가져오면, 답을 거기서 **먼저 추출**하고 **빈 곳·얇은 곳만** 묻는다. 이미 적힌 걸 다시 묻지 않는다.
8. **빠짐 점검.** 마무리 직전 `references/speckit-fields.md`의 체크리스트로, spec-kit이 추출할 내용이 다 들어 있는지 확인한다.

## 실행 모드 (두 가지)

이 스킬은 두 모드로 돌아간다. **차이는 Phase A(기획서)를 인터뷰로 묻느냐, 초안부터 만들어 확인만 받느냐뿐**이고, 아래의 전체 흐름·기획서 게이트·핵심 원칙은 두 모드가 똑같이 지킨다. 단독으로 쓰든 세트에서 자동으로 불리든 같은 스킬이 일관되게 동작하게 하기 위한 장치다.

1. **대화 모드 (기본)** — Phase A를 한 항목씩 인터뷰로 진행한다(기존 동작 그대로). 자료가 거의 없고 사용자가 **함께 만들어 가고 싶어 할 때**, 또는 별다른 신호가 없을 때의 기본값.
2. **초안 모드 (자동)** — Phase A를 인터뷰로 묻지 않는다. 가져온 자료·대화에서 얻은 신호 + 합리적 가정으로 **기획서를 "초안부터 먼저" 완성**한 뒤, 기획서 게이트의 요약 카드로 확인받는다. 인터뷰로 되돌아가지 않는다. 자료가 얇아도 마찬가지다(있는 것 + 가정으로 채운다).
   - **이 모드로 들어가는 신호:** 사용자가 "**알아서 / 빠르게 / 다 해줘**"라고 하거나, 가져온 자료가 넉넉할 때. 그리고 상위 오케스트레이터 **easy-dev-suite가 자동 모드로 이 스킬을 호출**할 때.

**두 모드 공통으로 반드시 지킬 것:**

- **게이트는 두 모드 모두 유지한다.** 초안 모드에서도 요약 카드(한 줄 정체성 / 핵심 행동 3개 / 성공 기준 / 가정)를 **반드시 한 번 보여주고** 사용자가 바로잡을 틈을 준다(침묵하거나 "맞아요"면 통과). 이 게이트가 초안 모드의 안전장치다.
- **초안 모드는 "대충 만드는 것"이 아니라 "묻지 않는 것"이다.** Phase A 7번의 기능별 풍부화 항목(무엇을 하나 / 누가·언제 / 입력→결과 / 규칙·조건 / 안 되는 것)을 자료가 없으면 **가정으로라도** 채운다. 임의로 정한 것은 전부 "가정"에 표시한다.
- 비전문가 언어, 읽기 쉬운 산출물, 있는 건 안 묻기, `speckit-fields` 점검 등 나머지 핵심 원칙과 **Phase B(설계서)·선택 모듈·마무리는 두 모드가 똑같이** 적용한다.

## 전체 흐름

```
0. 오리엔테이션 + 가져온 자료 확인
1. (자료가 있으면) 자료에서 추출 → 빈 곳·얇은 곳만 골라내기
A. 기획서 인터뷰  → 읽기 쉬운 기획서   (핵심)  ※ 추출로 채운 항목은 건너뜀
   ── 게이트: 사용자 확정 ──
B. 설계서 인터뷰  → 읽기 쉬운 설계서   (핵심)
+  선택 모듈 (단서가 보이거나 사용자가 원할 때만):
     · 사업계획서   · 사용자 정의   · 제품 원칙
C. 마무리 (문서 완성 안내)
```

핵심은 항상 기획서·설계서다. 선택 모듈은 강제하지 않는다 — 아래 "선택 모듈" 참고.

---

## Step 0: 오리엔테이션

처음 한 번만, 짧게(길게 늘어놓지 않는다):

- "지금부터 몇 가지 여쭤볼게요. 기술 용어는 몰라도 됩니다. 평소 말로 답해 주시면 제가 정리합니다."
- "두 가지를 만들 거예요. ① 무엇을·왜 만들지 정리한 **기획서**, ② 그걸 어떻게 만들지 정리한 **설계서**. 둘 다 읽기 편한 문서로 드립니다."
- **"혹시 미리 적어두신 메모나 문서, 자료가 있으세요? 있으면 먼저 보여주세요. 거기서 채울 수 있는 건 제가 채우고, 모자란 부분만 여쭤볼게요."**

자료가 없으면 곧장 Phase A 첫 질문으로 들어간다. 자료가 있으면 Step 1로 간다.

---

## Step 1: 가져온 자료 흡수 (자료가 있을 때만)

사용자가 메모·기획안·요구사항·기존 문서 등을 가져오면, **묻기 전에 먼저 읽는다.**

1. **읽고 매핑한다.** 자료 내용을 기획서·설계서(및 해당되면 선택 모듈) 항목에 맞춰 분류한다. `references/speckit-fields.md` 체크리스트로 **이미 채워진 항목 / 빈 항목 / 적혔지만 얇은 항목**을 가른다.
2. **추출 결과를 확인받는다.** "이렇게 이해했는데 맞을까요?"라며 짧게 요약해 보여준다. 추출이 틀릴 수 있으니 사용자가 바로잡을 기회를 준다. **자료에 적힌 내용을 임의로 바꾸지 않는다.**
3. **빈 곳·얇은 곳만 묻는다.** 이미 충분한 항목은 다시 묻지 않고, 비었거나 모호한 부분만 Phase A·B의 해당 질문으로 채운다.

### 우리가 만든 문서를 다시 가져온 경우 (풍부화 모드)

가져온 자료가 이전에 만든 기획서·설계서(또는 그 비슷한 형식)이면, 그것을 **기준선**으로 삼고 **더 깊게 파는 데 집중**한다.

- 얇은 지점을 찾는다: 기능에 "입력 → 결과·규칙·금지"가 빠졌거나, 성공 기준이 모호하거나, 예외 상황·사용자 정의가 한 줄뿐이거나, "아직 안 정한 것/가정"으로 남아 있던 항목 등.
- 그 지점을 겨냥한 **새 질문**으로 구체화한다. (기능은 Phase A 7번의 풍부화 항목 기준으로 보강.)
- 기존에 잘 적힌 부분은 건드리지 않는다. 무엇을 더 채웠는지 끝에 한 줄로 요약해 알려준다.

---

## Phase A: 기획서 인터뷰

`assets/기획서-템플릿.md`를 채운다. 아래 순서로 평이하게 묻는다.
**초안 모드일 때는 아래 1~12번을 하나씩 묻지 않는다.** 자료 + 합리적 가정으로 채운 뒤 곧장 "기획서 게이트"로 간다. (모드 판단은 위 "실행 모드" 참고.)
**Step 1에서 자료로 이미 채운 항목은 다시 묻지 않고 넘어간다.** 빈 곳·얇은 곳만 묻는다.
**괄호 안은 내부 메모(어떤 내용을 확보하는지)이며 사용자에게 말하지 않는다.**

1. **한 줄 정체성** — "한 문장으로, **누구를 위한 무엇**을 만들고 싶으세요?"
2. **가장 중요한 행동** — "이걸 쓰는 사람이 **가장 먼저, 가장 자주** 하게 될 행동은 뭔가요?" (→ 1순위 행동)
3. **그다음 행동** — "그다음으로 꼭 필요한 행동은요? 또 있을까요?" (→ 2,3순위)
4. **우선순위 이유** — "방금 그게 왜 제일 중요할까요?"
5. **MVP 확인** — "다른 건 없어도 **이것만 되면** 일단 쓸 만할까요?"
6. **꼭 있어야 하는 기능 받기** — "방금 그 일들을 하려면 **꼭 있어야 하는 기능**은 뭘까요? '있으면 좋은' 기능과 **구분**해서 말씀해 주세요." 핵심 기능과 부가 기능을 나눠 적는다.
7. **기능별 풍부화 (한 번에 한 기능씩)** — 핵심 기능 각각을 평이하게 캐물어 **자세히 채운다.** 한꺼번에 묻지 말고 기능 하나씩:
   - "이 기능은 정확히 **뭘** 하나요?"
   - "**누가, 언제** 쓰나요?"
   - "뭘 **넣으면** 뭐가 **나오나요**? (입력 → 결과)"
   - "지켜야 할 **규칙이나 조건**이 있나요? (예: 금액은 0보다 커야 한다)"
   - "이건 **안 돼야 한다**는 게 있나요?"
   받은 내용을 기능별로 **자세하고 읽기 쉽게** 정리한다. (FR-001 같은 번호는 붙이지 않는다.) **이 풍부화된 기능 묶음이 기획서의 알맹이다.**
   - **기능이 많으면**(대략 5개 초과) 모두를 똑같이 깊게 캐묻지 않는다. **핵심 기능 2~3개만** 위 항목으로 깊게 파고, 나머지는 "무엇을 하나" 한두 줄로 가볍게 적은 뒤 "이 기능들은 일단 이 정도로 두고 나중에 더 자세히 해도 될까요?"라고 확인한다. 사용자가 지치지 않게 하는 것이 우선이다.
8. **동작 되읽기** — 수용 시나리오를 따로 묻지 않는다. 핵심 기능 위주로 "그러면 *이럴 때 이렇게 하면 이렇게 되어야* 하는 거죠?"로 **자연스러운 문장으로 되읽어 확인**한다. (Given/When/Then 표기를 쓰지 않는다 — 이 비전문가용 기획 단계에서만의 원칙이다. GWT를 쓰는 정형 시나리오는 뒤의 파생·개발 인계 단계에서 `easyproduct-scenario` 스킬이 맡는다.)
9. **예외 상황** — "사용자가 실수하거나, 칸을 비워두거나, 인터넷이 끊기면 어떻게 되어야 할까요?" 예시를 들어 유도한다.
10. **핵심 정보 (정보 덩어리)** — "이 서비스가 **저장해 두고 다뤄야 할 정보의 큰 묶음**이 뭘까요? (예: 회원, 모임, 회비 내역)" **묶음(덩어리)만** 평이하게 받는다. 개별 항목(필드)은 캐묻지 않는다. 유형상 당연히 있을 덩어리(회원제면 `user`, 결제가 있으면 `payment` 등)는 사용자가 말하지 않아도 제안해 넣는다. 각 덩어리는 **그룹명(영문 소문자·단수) · 한글명 · 한 줄 설명** 셋으로 정리하되, **그룹명은 AI가 제안**하고 사용자는 한글명·설명만 확인하게 한다(사용자가 영어를 직접 짓지 않도록). 개별 필드는 비워 둔다 — "모든 묶음에는 고유번호·생성시각·수정시각 같은 기본 관리 정보가 자동 포함된다"만 한 줄로 안내하고 나열하지 않는다. 사용자가 자연스럽게 흘린 명백한 항목이 있으면 "여기 들어갈 만한 정보(예시)"에 참고용으로 한둘만 적고 "자세한 항목은 이후 단계에서 채웁니다"라고 표시한다. (세부 필드는 데이터 모델·화면 설계 단계 몫이다.)
11. **성공 기준** — "이게 성공했다는 걸 **무엇을 보고** 알 수 있을까요?" 못 정하면 결과·체감 기준을 제안하고 동의받는다(예: "가입까지 2분 이내"). 기술이 아니라 결과로 표현한다.
12. **가정 정리** — 사용자가 명시하지 않아 우리가 임의로 정한 것은 "이렇게 가정하고 갈게요"로 적는다.

### 기획서 게이트 (반드시 통과)

채워지면 **요약 카드**를 보여준다: 한 줄 정체성 / 핵심 행동 3개 / 성공 기준 / 가정.
"이대로 맞을까요? 고치고 싶은 곳 있으세요?"라고 묻는다.
확정되면 `assets/기획서-템플릿.md`로 **읽기 쉬운 기획서**를 완성한다.
**확정 전에는 설계서로 넘어가지 않는다.**

---

## Phase B: 설계서 인터뷰

`assets/설계서-템플릿.md`를 채운다. **직접 묻는 건 아래뿐**이고, 나머지는 추론해 채운 뒤 확인만 받는다.

**직접 묻기 (평이하게):**
- **종류** — "**웹사이트**인가요, **폰 앱**인가요, **컴퓨터 프로그램**인가요?"
- **규모** — "처음에 **몇 명쯤** 쓸 것 같으세요? (수십 / 수천 / 그 이상)"

**번역해서 묻기:**
- "느려도 괜찮나요, **빨라야** 하나요?" / "**인터넷 없이도** 써야 하나요?"
- "**무료로 시작**하고 싶으세요, **유료 운영**도 괜찮으세요?"

**묻지 않고 추론하기:**
- 프로그래밍 언어·라이브러리·저장 방식·테스트 도구는 위 답을 근거로 **합리적 기본 스택을 추천**하고 한 번만 확인한다: "이 기술들로 진행하려는데 특별히 원하시는 게 있으세요?" 대개 "알아서"라고 하므로 채택하고, **각 선택의 이유를 한 줄씩** 설계서에 남긴다.
- 정할 수 없는 항목은 설계서의 "아직 안 정한 것"에 `[확인 필요: …]`로 남긴다.

## 선택 모듈 (필요할 때만)

핵심 두 문서 외에, 아래 단서가 보이면 **"~도 정리해 드릴까요?"라고 한 번 제안**하고
사용자가 원할 때만 진행한다. 강제하지 않는다. 상세 질문·작성법은 `references/optional-modules.md`를 읽는다.

- **사업계획서** — 돈·예산·투자·수익·팀·마감·"만들 가치가 있나" 신호가 있을 때. 기획서의 기능 범위에서 인력·기간·예산을 **범위로** 추정한다(단일 숫자 금지, 견적·재무 조언 아님). → `assets/사업계획서-템플릿.md`
- **사용자 정의** — 대상이 제품을 좌우하거나, 사용자 그룹이 여럿이거나, 특별한 대상(노인·어린이·장애·외국어 사용자 등)이 나올 때. 대상이 단순하면 기획서의 "누가 쓰나요"로 충분. → `assets/사용자정의-템플릿.md`
- **제품 원칙** — 분위기·색감·UI·"쉬워야 한다"·접근성 같은 신호가 있을 때. 여기서 정한 "꼭 지킬 원칙"은 개발 내내 타협하지 않을 기준이 된다. → `assets/제품원칙-템플릿.md`
- **유저 스토리** — "유저 스토리도 만들어줘/뽑아줘" 요청이 있거나, 상위 오케스트레이터(easy-dev-suite)가 이 옵션을 켜서 호출할 때. **기본값은 만들지 않음** — 켜졌을 때만 만든다. 기획서의 대상·핵심 행동·핵심 기능·규칙을 **재료로 추출·변환**하는 성격이라 새 인터뷰를 벌이지 않는다. 각 스토리는 "누가 / 무엇을 원하나 / 왜"와 (분량에 따라) 수용 기준을 담는다. → `assets/유저스토리-템플릿.md`
  - **분량 3단계(사용자에게 한 번 묻고, 별말 없으면 "적당히"로 만들고 그 사실을 한 줄 알린다):** 간단히 = 핵심 기능당 한 줄 스토리 + 우선순위만 / 적당히(기본) = 핵심 기능 전부 + 스토리마다 수용 기준 1~2개 / 자세히 = 부가 기능까지 + 수용 기준 촘촘히.
  - **추출 우선:** 기획서에 있는 내용에서 먼저 뽑고, 모자란 부분만 합리적 가정으로 채운 뒤 전부 "가정"에 표시한다. Given/When/Then 같은 표기 없이 평범한 문장으로 쓴다.
  - **시나리오와의 구분:** 유저 스토리는 **단일 요구의 가치**("누가·무엇을·왜")를 다룬다. 여러 기능·화면을 관통하는 **여정**은 유저 스토리가 아니라 시나리오(`easyproduct-scenario`)의 몫이다. 여기서 여정을 억지로 늘려 쓰지 않는다.

순서 팁: **사용자 정의 → 제품 원칙**은 자연스럽게 이어진다(대상 특성이 원칙을 정한다). **사업계획서**는 기획서가 어느 정도 잡힌 뒤에 추정해야 정확하다.

**제안은 한 번에 하나만.** 여러 모듈이 동시에 해당돼도 한꺼번에 늘어놓지 않는다. **지금 맥락에서 가장 관련 깊은 것 하나만** 먼저 제안하고, 그게 끝나면 다음으로 관련 있는 것을 제안한다.

## Step C: 마무리

1. 마무리 직전 `references/speckit-fields.md` 체크리스트로 빠진 내용이 없는지 조용히 점검한다(사용자에게 보여주는 단계는 아님).
2. **읽기 쉬운 기획서·설계서**(필요 시 선택 모듈 문서)를 최종 산출물로 출력한다. 사용자 언어, 평범한 표현.
3. 사용자에게는 간단히 알린다: **"기획서와 설계서가 완성됐습니다. 이대로 개발에 활용하시면 됩니다."** 내용이 바뀌면 이 문서를 고치면 된다는 정도만 덧붙인다.
4. spec-kit 명령(`/speckit.specify` 등)을 사용자에게 안내하지 않는다. 형식 변환과 후속 단계는 개발 단계에서 알아서 진행된다.

## 비전문가 응대 규칙 (대화 내내)

- 한 답변에 질문은 **하나, 많아도 둘**. 폭탄 금지.
- "잘 모르겠어요"라고 하면 **구체적 예시 2~3개**를 제시해 고르게 한다. 예시 없이 다시 묻지 않는다.
- 전문 용어가 나오면 **즉시 한 줄로 풀어** 쓴다. (예: "데이터베이스(= 정보를 저장해 두는 창고)")
- 사용자의 핵심 의도를 임의로 줄이거나 부풀리지 않는다. 애매하면 가정으로 적고 확인한다.
- "이건 안 됩니다"보다 "**이렇게 하면 어떨까요?**"로 제안한다.
- 가끔 **한 줄 요약으로 되읽어** 사용자가 통제감을 느끼게 한다.
