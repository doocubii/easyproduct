# 기획서: [프로젝트 이름]

> 이 문서는 "무엇을, 누구를 위해, 왜 만드는지"를 사람이 읽기 쉽게 정리한 것입니다.
> 기술 용어는 쓰지 않습니다.

## 한 줄 소개
누구를 위한 무엇인지 한 문장으로.

## 왜 만드나요 (배경과 목적)
지금 어떤 불편이나 필요가 있고, 이 서비스가 무엇을 해결하거나 줄 것인지.

## 누가 쓰나요
주로 사용할 사람들. (한 종류가 아니면 나눠서)

## 사용자가 하는 일 (중요한 순서대로)
1. **(가장 중요)** … — 왜 이게 제일 중요한지 한 줄
2. … 
3. …

> 이 중 1번만 되어도 일단 쓸 만한가요? → (예/아니오, 한 줄 이유)

## 꼭 있어야 하는 기능 (자세히)
없으면 안 되는 기능부터 하나씩 자세히 적습니다.

### 기능: [기능 이름]
- **무엇을 하나**: …
- **누가·언제 쓰나**: …
- **입력 → 결과**: …을 넣으면 …이 됩니다
- **규칙·조건**: … (예: 금액은 0보다 커야 함)
- **안 되어야 하는 것**: …

### 기능: […]
(같은 항목으로 반복)

## 있으면 좋은 기능 (나중에)
지금은 없어도 되지만 나중에 있으면 좋은 것들.
- …

## 이렇게 동작하면 됩니다 (상황별)
핵심 기능이 실제로 어떻게 흘러가는지, 읽고 바로 이해되는 문장으로. 예:
- 사용자가 …할 때, …하면, …되어야 합니다.
- …인 경우에는 …합니다.

## 예외·돌발 상황
- 입력 칸을 비워두면: …
- 인터넷이 끊기면: …
- 사용자가 실수하면: …

## 기억해 둬야 할 정보 (정보 덩어리)
이 서비스가 저장해 두고 다뤄야 하는 정보의 큰 묶음입니다. 여기서는 **묶음(덩어리)만** 정합니다.
각 묶음 안에 어떤 세부 정보가 들어갈지는 이후 데이터·화면 설계 단계에서 채웁니다.

| 그룹명(영문) | 한글명 | 무엇의 묶음인가 |
|---|---|---|
| user | 사용자 | 회원(요청자·수행자) 정보 |
| request | 요청 | 올라온 심부름·재능 요청 글 |
| deal | 거래 | 확정된 요청의 진행·완료 |

> 그룹명(영문)은 이후 데이터·개발 설계에서 쓰는 이름이라 AI가 제안합니다. 한글명·설명만 확인해 주세요.
> 모든 묶음에는 **고유번호·생성시각·수정시각** 같은 기본 관리 정보가 자동으로 포함됩니다(따로 적지 않아도 됩니다).
> 세부 항목(예: 요청의 금액·시간)은 이후 데이터 모델·화면 설계 단계에서 각 묶음에 추가됩니다.

## 성공의 기준
"잘 만들어졌다"를 무엇을 보고 알 수 있는지. 결과·체감으로. 예:
- 처음 써보는 사람도 2분 안에 가입을 끝낼 수 있다.
- 회비 정산에 드는 시간이 절반으로 준다.
