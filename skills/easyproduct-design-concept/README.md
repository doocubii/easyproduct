# easyproduct-design-concept

> 개발·디자인 지식이 없는 사람이 몇 가지 질문에만 답하면, 색상 팔레트·글꼴·모양·컴포넌트 스타일·톤앤매너·다크모드·접근성 기준을 **구체적인 값(색상 코드 등)까지** 담은 디자인 컨셉 초안을 통째로 잡아 주는 스킬입니다. 매번 묻지 않고 서로 다른 3안을 먼저 만든 뒤 사용자가 고르고 고치게 합니다.

[easyproduct 스킬 세트](../../README.md)의 일부입니다.

## 무엇을 하나
- 분위기·좋아하는 사이트 등 **핵심 3~4가지만** 묻고, 나머지는 추론해 채운다.
- 성격·형태·톤까지 확실히 다른 **컨셉 3안을 자동 생성**한다(1안 = AI 추천안).
- 각 안에 색상 코드(hex)·글꼴·크기 위계·버튼/입력창/카드 스타일·아이콘·톤앤매너·다크모드·접근성 기준까지 **그대로 쓸 수 있는 값**을 채운다.
- 색 코드만으로는 감이 안 오므로, 각 안을 **UI 키트 + 샘플 페이지로 렌더링(HTML·PNG)**해 눈으로 비교하게 한다.
- 선택을 위해 대화를 멈추지 않고, **1안으로 곧장 컨셉 문서를 확정**한 뒤 나중에 바꾸라고 하면 교체한다.

## 언제 쓰나
- "디자인 컨셉/방향", "톤앤매너", "색상·색감", "폰트·글꼴", "무드·분위기", "스타일 가이드", "브랜드 느낌"을 정하려 할 때.
- "무슨 색으로 하지?", "어떤 느낌으로 만들지?"처럼 서비스 겉모습을 막막해할 때.
- 기획서나 화면 구조(IA)는 있는데 "이걸 어떤 색·글꼴·느낌으로 보이게 할지"로 넘어갈 때.
- easyproduct-doc-builder(기획서)나 easyproduct-ia-designer(화면 구조) 뒤 디자인 단계로 이어질 때.

## 산출물
- **디자인 컨셉 문서(필수, 딱 하나)** — `디자인컨셉-[프로젝트이름].md`. 1·2·3안을 한 문서에 나란히 담고, 맨 앞 "선택된 안"에 채택 번호(기본 1안)와 추천 이유를 적는다. 원하면 .docx로도.
- **3안 미리보기(필수, 임시 참고물)** — 각 안의 `안{n}-*-키트.html/.png`, `안{n}-*-샘플.html/.png` + 세 안을 나란히 비교하는 `디자인안-비교.html`.

## 단독 실행 vs 세트
- 기본은 **독립 실행**이다. 분위기·좋아하는 사이트만 알면 혼자서도 컨셉을 만든다.
- 같이 설치돼 있으면 서로 이어 쓴다: 기획서·화면 구조(IA)가 있으면 먼저 읽어 정체성·대상·유형·이미 정해진 색을 뽑고 관련 질문은 건너뛴다. 만든 컨셉은 각 화면을 그릴 때와 설계서의 "겉모습·느낌" 자리로 이어진다.

## 구성 파일
- `SKILL.md` — 스킬 본체(일하는 방식·3안 생성·미리보기·마무리 절차).
- `assets/concept-template.md` — 최종 디자인 컨셉 문서의 읽기 쉬운 구조(이대로 채운다).
- `references/mood-design-map.md` — 분위기를 색·글꼴·모양 등 구체 값으로 번역하는 지식베이스 + 질문법·"모르겠어요" 대응.
- `references/preview-guide.md` — UI 키트·샘플 페이지 HTML 구조와 PNG 변환 방법.
- `scripts/render_png.py` — HTML을 PNG로 변환하는 스크립트(배치·Claude Code용).

## 세트 안에서의 위치
- 전체 흐름상 **데이터 모델 다음, 정책·약관 앞**의 디자인 단계를 맡는다.
- easyproduct-doc-builder의 "제품 원칙"이 남긴 분위기·원칙 한 줄을 받아 **색상 코드·글꼴·컴포넌트 스타일까지 구체적인 컨셉으로 확장**한다.
- 확정된 컨셉은 easyproduct-screen-design(화면 설계서)이 각 화면을 그릴 때의 겉모습·느낌 기준이 된다.
