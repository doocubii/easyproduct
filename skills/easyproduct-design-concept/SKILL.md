# 디자인 컨셉 정하기

## 목적

디자인 지식이 전혀 없는 사람도, **핵심 몇 가지만 답하면** AI가 서비스의
**디자인 컨셉을 하나의 일관된 초안으로 통째로 잡아준다.** 그다음 사용자가 고친다.

디자인은 비전문가가 말로 표현하기 가장 어려운 영역이다. "예쁘게요", "깔끔하게요" 말고는
떠올리기 힘들다. 그래서 이 스킬은 **먼저 물어 채우지 않고, 먼저 만들어 보여준다.**
분위기와 좋아하는 사이트 몇 개만 알면, 색·글꼴·모양·버튼 스타일까지 **구체적인 값으로**
초안을 만들고, 사용자는 그걸 보고 "이건 좋고 저건 별로"라고 반응만 하면 된다.

## 일하는 방식 — 3안 자동 생성 + 무중단 확정 (중요)

**중간에 물어보고 멈추지 않는다.** 이 스킬은 다른 개발 문서들과 함께 **한 번에 쭉 실행되는 배치 흐름**의
일부로 돌아갈 수 있다. 그래서 사용자에게 "어느 안이 좋으세요?"를 물어 **멈추지 않는다.**
대신 여러 안을 만들고 **추천안(1안)으로 곧장 문서를 확정**한 뒤, 사용자가 나중에 바꾸라고 하면 그때 바꾼다.

```
초반에 꼭 필요한 3~4가지만 묻는다 (자료가 있으면 그것도 안 묻는다)
        ↓
서로 확실히 다른 컨셉 3안을 만든다 (1안 = AI 추천안)
        ↓
각 안을 눈으로 볼 수 있게: 컴포넌트 UI 키트 + 샘플 페이지를 HTML·PNG로
        ↓
1안(추천안)으로 최종 컨셉 문서를 곧장 확정한다 — 선택을 기다리지 않는다
        ↓
(나중에) 사용자가 "2안/3안으로 바꿔줘" 하면 그때 그 안으로 교체
```

- **선택을 위해 대화를 멈추지 않는다.** 3안을 나란히 보여주되, 그중 1안으로 이미 최종 문서를 완성해 둔다. 사용자는 그대로 써도 되고, 원하면 다른 안을 고르면 된다.
- 모르는 건 합리적으로 정해 채우고, **무엇을 가정했는지 문서에 표시**한다.
- 초안이 완벽할 필요는 없다. 사용자가 보고 고칠 수 있는 **출발점**을 빠르게 주는 게 목표다.

### 3안을 어떻게 다르게 만드나
세 안은 색만 다른 게 아니라 **성격이 확실히 다른 방향**이어야 비교가 의미 있다. 서비스 성격에 맞춰
`references/mood-design-map.md`의 프리셋에서 서로 대비되는 셋을 고른다. 기본 조합 예:
- **1안 (추천) — 안정·정공법**: 서비스 유형·대상에 가장 무난하고 신뢰가 가는 방향. 특별한 이유가 없으면 이걸 추천안으로.
- **2안 — 개성·활기**: 조금 더 과감하고 눈에 띄는 방향(선명한 색·큰 대비·둥근 형태 등).
- **3안 — 절제·미니멀**: 여백과 흑백 위주로 담백하게 가는 방향.
- 추천 근거(왜 1안인지)를 컨셉 문서 맨 앞에 **한두 줄**로 남긴다. 대상·유형·이미 정해진 색을 근거로.

## 핵심 원칙

1. **비전문가 언어.** 디자인 용어(팔레트, 서체, 대비, hex 등)를 먼저 꺼내지 않는다. 불가피하면 한 줄로 풀어 쓴다. ("hex = 색을 나타내는 코드, 예: #2563EB")
2. **말이 아니라 느낌으로 묻는다.** "무슨 색?" 대신 "어떤 느낌?"으로, 형용사·예시·비교로 묻는다.
3. **구체적인 값까지 채운다.** 사용자에게는 읽기 쉽게 보여주되, 실제 색상 코드·글꼴 이름·크기 위계 등 **만들 사람(또는 AI)이 그대로 쓸 수 있는 값**을 남긴다. 이 스킬의 산출물은 나중에 구현의 근거가 된다.
4. **하나의 일관된 컨셉.** 색·글꼴·모양·톤이 **따로 놀지 않고 하나의 성격**을 이루게 한다. 서로 안 어울리는 요소를 섞지 않는다.
5. **자동으로 채우고 표시한다.** 사용자가 정하지 않은 건 분위기에서 추론해 채우고 "가정"에 모아 둔다.
6. **접근성은 기본으로 챙긴다.** 글자·배경 명도 대비, 색만으로 정보를 구분하지 않기 등은 사용자가 말하지 않아도 기본값으로 넣는다.
7. **결과는 파일로 준다.** 최종 컨셉은 채팅 텍스트가 아니라 **문서 파일**로 만든다. 색은 코드만 적으면 감이 안 오므로 **눈으로 보이는 미리보기 파일**을 함께 만든다. (자세한 건 Step 4.)

## 다른 스킬과의 관계

기본은 **독립 실행**이다. 하지만 같이 설치돼 있으면 서로 이어 쓴다.
- **입력으로**: 사용자가 이미 만든 기획서(무엇을·누구를 위해)나 화면 구조(IA)가 있으면 먼저 읽는다. 한 줄 정체성·대상 사용자·서비스 유형·특별한 대상(노인·어린이 등)을 거기서 뽑아, 그 질문들은 건너뛰고 디자인 방향 추론의 근거로 쓴다.
- **easy-dev-doc-builder의 "제품 원칙"과의 차이**: 그 모듈은 분위기·원칙을 **가볍게 한 줄씩** 남기는 정도다. 이 스킬은 그걸 받아 **색상 코드·글꼴·컴포넌트 스타일까지 구체적인 컨셉으로 확장**한다. 제품 원칙이 이미 있으면 그것을 컨셉의 출발점으로 삼는다.
- **출력으로**: 이 컨셉은 화면 구조(IA)의 각 화면을 실제로 그릴 때, 그리고 설계서의 "겉모습·느낌" 자리에 그대로 들어간다.

## Step 0: 시작 (짧게)

길게 늘어놓지 않는다:

- "몇 가지만 여쭤보고, 서비스의 **색·글꼴·전체 느낌**을 제가 먼저 하나로 잡아드릴게요. 그다음 마음껏 고치시면 됩니다. 디자인 용어는 몰라도 됩니다."
- **기획서·화면 구조·참고 자료(예: 로고, 참고 이미지, 좋아하는 사이트 링크)가 있으면 먼저 달라고 한다.** 있으면 읽고 대상·유형·이미 정해진 것을 뽑아 아래 질문 중 답이 되는 건 건너뛴다.

## Step 1: 핵심 질문 (3~4개만)

한 번에 하나씩, 짧게. 자료에서 이미 답이 나온 건 묻지 않는다.
자세한 질문법과 "모르겠어요" 대응은 `references/mood-design-map.md`를 참고한다.

1. **분위기·느낌** — "이 서비스가 어떤 **느낌**이었으면 하세요?" 감을 못 잡으면 상반된 예시로 고르게 한다:
   - "차분하고 신뢰감 있는 / 밝고 활기찬 / 따뜻하고 친근한 / 고급스럽고 세련된 / 재미있고 귀여운 중에 가까운 게 있을까요?" (여러 개 골라도 됨)
2. **좋아하는 사이트·앱** — "평소에 '이 앱이나 사이트 보기 좋다, 쓰기 편하다' 싶은 곳 있으세요? 있으면 이름이나 링크를 알려주세요." (비전문가에게 가장 강력한 단서다. 없으면 넘어간다.)
3. **이미 정해진 것** — "이미 정해진 **색이나 로고, 이름 스타일**이 있으세요? 있으면 그걸 중심으로 잡을게요." (없으면 분위기로 정한다.)
4. **밝게 / 어둡게** — "화면이 **밝은 바탕**이 좋으세요, **어두운 바탕(다크)**이 좋으세요? 아니면 둘 다 되게 할까요?" (감이 없으면 밝은 바탕 기본 + 다크 지원으로 잡는다.)

> 대상 사용자(노인·어린이·시각약자 등)는 **따로 묻지 않는다.** 기획서·IA에서 뽑거나, 없으면 일반 성인 기준으로 잡되 접근성 기본값을 넣는다.
> 분위기가 여러 개거나 애매하면, 가장 가까운 하나를 1안(추천안)의 중심으로 잡고 나머지는 2·3안으로 갈래를 낸다.
> **배치 흐름으로 실행 중이라 사용자가 답할 수 없는 상황이면, 위 질문도 자료·유형에서 추론해 채우고 곧장 3안 생성으로 넘어간다. 추론한 것은 "가정"에 남긴다.**

## Step 2: 컨셉 3안 자동 생성 (묻지 않고 채운다)

`references/mood-design-map.md`를 열어, 서로 확실히 다른 **3안**을 만든다(1안=추천). 각 안마다 아래 11개 항목을
**한 번에 다 채운다.** 세 안이 색만 다르지 않게 — 성격·형태·톤까지 다르게 잡는다.

각 안에 채우는 항목:

1. **브랜드 성격** — 분위기를 형용사 3~5개로 못박고, 한 줄로 "이런 사람 같은 서비스"라고 표현.
2. **색상 팔레트** — 주색 / 보조색 / 강조색 / 배경 / 글자색 / 상태색(성공·경고·오류)을 **실제 색상 코드(hex)까지**. 이미 정해진 색이 있으면 그걸 주색으로. 각 색을 "어디에 쓰는 색인지" 한 줄로.
3. **글꼴(타이포그래피)** — 제목용·본문용 글꼴 방향(한글이면 무료로 널리 쓰이는 것 우선: Pretendard, 나눔 계열 등)과 크기 위계. 왜 이 글꼴이 분위기에 맞는지 한 줄.
4. **모양·형태** — 모서리 둥글기(각짐↔둥긂), 그림자(없음↔뚜렷), 여백·정보 밀도(빽빽↔여유). 값 + "그래서 어떤 느낌인지".
5. **컴포넌트 스타일** — 버튼(기본·보조·비활성), 입력창, 카드가 어떻게 보이는지. 만들 사람이 그대로 그릴 수 있게 구체적으로.
6. **아이콘·이미지 스타일** — 아이콘 종류(선형↔채움), 사진↔일러스트, 모서리 처리 등 방향.
7. **톤앤매너(글 어조)** — 버튼·안내문·오류 메시지의 말투. 예시 문구 1~2개.
8. **다크모드** — 지원 여부와, 지원하면 어두운 바탕 색 처리 방향.
9. **접근성 기준** — 글자·배경 명도 대비 확보, 색만으로 구분하지 않기, 터치 영역 최소 크기 등 기본으로 지킬 것. (특별한 대상이 있으면 강화.)
10. **참고 레퍼런스** — 사용자가 든 사이트 + AI가 덧붙인 비슷한 성격의 예시, "무엇을 참고했는지" 한 줄씩.
11. **하지 말 것 / 자동으로 넣은 것 / 가정 / 아직 안 정한 것** — 각각 모아 표시.

**세 안을 별도 문서 3개로 만들지 않는다. 하나의 컨셉 문서 안에 1·2·3안을 나란히 나열**하고, 각 안 아래에
그 안의 샘플 파일 이름을 적는다. 그리고 문서 맨 앞 "선택된 안"에 지금 채택된 안 번호만 적는다.
1안이 기본 선택이다. 각 안을 눈으로 볼 수 있게 렌더링하는 건 아래 Step 3.

## Step 3: 3안을 눈으로 보여주기 (HTML + PNG)

각 안마다 컴포넌트 UI 키트 + 샘플 페이지를 **HTML과 PNG 두 형태로** 만든다. 색상 코드만으로는 비전문가가
비교할 수 없으니, 실제로 그려진 걸 보게 한다. 만드는 법은 `references/preview-guide.md`를 반드시 참고한다.

각 안에 대해:
- **UI 키트(HTML+PNG)** — 색 견본(이름·코드·색칠 네모), 버튼(기본·보조·비활성), 입력창, 카드, 배지·알림, 글꼴 크기 위계를 한 화면에. 내용은 범용 자리표시(특정 화면 아님).
- **샘플 페이지(HTML+PNG)** — 그 컨셉이 실제 화면에 입혀지면 어떤 느낌인지 보여주는 대표 화면 한 장(헤더 + 대표 콘텐츠 + 주요 버튼). **IA의 실제 메뉴·기능을 조립하지는 않는다** — 어디까지나 "느낌 샘플"이다. IA가 있으면 라벨 정도만 참고해 사실감을 준다.
- HTML은 값과 **똑같은 색·글꼴**로 그린다. PNG는 그 HTML을 그대로 이미지로 변환한 것(`scripts/render_png.py` 사용).

세 안을 나란히 보여주되, **어느 안이 좋은지 물어 멈추지 않는다.** "1안(추천)으로 문서를 완성해 뒀고, 2·3안도 보시라"는 식으로 안내만 한다.

> **샘플 페이지·미리보기는 선택을 돕기 위한 임시 참고물이다.** 개발의 표준·근거가 되는 것은 컨셉 문서의 값(색·글꼴·컴포넌트 규칙)이지, 샘플 화면의 레이아웃이 아니다. 이 사실을 컨셉 문서에도 명시한다(Step 4).

## Step 3.5: 나중에 바꾸기 — 두 경우를 구분한다

사용자가 나중에 바꿔 달라고 할 때, **두 가지 경우를 구분**한다. 잘못 판단하면 멀쩡한 걸 지우거나, 지워야 할 걸 남긴다.

**(가) 이미 만든 안 중에서 고르기** — "2안으로 해줘", "3안이 낫다" 처럼 **제시된 안 안에서 선택**하는 경우.
- 문서 맨 앞 **"선택된 안" 표시만 그 번호로 바꾼다.** 1·2·3안 내용은 그대로 두고, 문서 전체를 다시 쓰지 않는다.
- 샘플도 이미 만들어 둔 그 안의 것을 쓰면 되므로 새로 만들지 않는다.
- 이게 3안을 한 문서에 나열해 두는 이유다 — **손대는 곳은 선택 표시 한 줄뿐.**

**(나) 새로운 방향을 구체적으로 지시** — "더 어둡고 고급스럽게", "초록 말고 파랑 계열로", "이 로고 색에 맞춰 다시" 처럼
**기존 3안에 없는 새 컨셉을 요구**하는 경우. 이건 선택이 아니라 새 출발이다.
- **기존 1·2·3안과 그 샘플·미리보기 파일을 전부 폐기한다.** (남겨두면 무엇이 유효한지 헷갈린다.)
- 새 지시를 반영해 **새 3안을 다시 만들고(또는 지시가 뚜렷하면 그 방향 1안 + 변주 2안), 샘플·PNG도 새로 그려** 보여준다.
- 폐기했다는 사실과 새로 만든 것을 한 줄로 알린다: "말씀하신 방향으로 다시 잡았고, 이전 안·샘플은 정리했습니다."

애매하면 (가)로 기울지 말고 사용자 표현을 본다: **안 번호를 말하면 (가), 색·느낌·방향을 말하면 (나).**

## Step 4: 마무리 — 반드시 파일로 만든다

**이 스킬의 산출물은 채팅창 텍스트가 아니라 "파일"이다.** 사용자가 "파일로 만들어 달라"고
말하지 않아도 **항상 파일로 만든다.** 배치 흐름에서는 아래 파일들을 다 만들어 두고, 1안을 최종으로 확정한다.

만드는 파일:

1. **디자인 컨셉 문서 (필수, 딱 하나)** — `assets/concept-template.md`를 채운 **읽기 쉬운 문서**. 안마다 문서를 나누지 않는다.
   - **1·2·3안을 한 문서 안에 나란히** 담는다. 각 안 아래에 그 안의 **샘플 파일 이름**을 적어, 어떤 이미지를 보면 되는지 연결한다.
   - 맨 앞 **"선택된 안"**에 지금 채택된 안 번호(기본 1안)와 추천 이유 한두 줄을 적는다. **선택은 이 한 곳에만 기록**한다 — 나중에 안을 바꾸면 여기만 고치면 된다.
   - 문서에 **"샘플 페이지는 선택을 돕기 위한 임시 참고물이며, 개발의 표준·근거는 이 컨셉 문서의 값이지 샘플 화면의 레이아웃이 아니다"**를 명시한다. (템플릿에 이 문구 자리가 있다.)
   - 기본은 **마크다운(.md)**. 원하면 Word(.docx)로도. 파일 이름 예: `디자인컨셉-[프로젝트이름].md`
2. **3안 미리보기 (필수, 임시 참고물)** — 세 안 각각에 대해 **UI 키트 + 샘플 페이지**를 만든다. 코드만으로는 비전문가가 비교 못 하므로 실제로 그려진 걸 준다.
   - 각 안: `안{n}-{이름}-키트.html` / `안{n}-{이름}-샘플.html` + 각 HTML을 변환한 **PNG**.
   - 세 안을 한눈에 비교하는 **표지 HTML**(`디자인안-비교.html`)도 만들어 PNG·핵심 색을 나란히 건다.
   - 만드는 법·구조·PNG 변환은 `references/preview-guide.md`와 `scripts/render_png.py`를 따른다.
   - 이 파일들은 **선택을 돕는 임시물**이다. 개발 표준으로 삼지 않는다. 안이 새 지시로 갈아엎이면(Step 3.5-(나)) 이 파일들은 폐기된다.

환경별 저장:
- **Claude Code / 배치**: 모든 파일을 프로젝트(또는 출력) 폴더에 저장하고 경로를 알려준다. PNG는 `scripts/render_png.py`로 HTML을 변환해 생성.
- **claude.ai**: 문서·HTML을 만들어 보여주고 내려받게 한다. PNG 변환기가 없으면 HTML 미리보기로 대체하고, 그 사실을 한 줄로 알린다.

마무리 안내:
- 배치가 아니라 대화 중이면 간단히: **"디자인 컨셉을 3안으로 만들고, 색·버튼·샘플 화면이 어떻게 보이는지 이미지로도 만들었습니다. 1안으로 문서를 확정해 뒀고, 다른 안이 좋으면 말씀만 주세요."**
- 기획서·화면 구조 흐름을 쓰는 사람이면, 이 컨셉이 각 화면을 그릴 때와 설계서의 겉모습 부분으로 이어진다고 한 줄만 덧붙인다.

## 비전문가 응대 규칙 (대화 내내)

- 초반 질문은 **3~4개**. 그 뒤로는 묻지 말고 초안으로 답한다.
- "잘 모르겠어요"라고 하면 **상반된 예시 2~3개**를 제시해 고르게 한다. ("각진 느낌 vs 둥근 느낌" 처럼.)
- 색을 코드로만 말하지 않는다. "차분한 남색(#2563EB)"처럼 **이름 + 코드**로.
- 전문 용어가 나오면 **즉시 한 줄로 풀어** 쓴다.
- "그 색은 안 됩니다"보다 "이렇게 잡아뒀는데 무거우면 더 밝게 바꿀게요"로.
- 사용자가 든 좋아하는 사이트가 있으면, 그 성격을 존중하되 **그대로 베끼지 않고** 이 서비스에 맞게 방향만 참고한다.
