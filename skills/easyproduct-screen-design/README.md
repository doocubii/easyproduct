# easyproduct-screen-design

> 개발 지식이 없는 사람도 대화만으로 서비스의 **화면 설계서(스토리보드)**를 갖추게 돕는 스킬입니다. 화면·기능 구조(IA)·확정 디자인 컨셉·데이터 모델을 재료로, 각 화면을 "글 정의(SSOT) + 데이터 정의 + (선택) HTML 목업"으로 설계합니다. 다른 문서와 달리 이 스킬만은 **초안 우선이 아니라, 컴포넌트·배치·문구가 처음부터 확정된 정확한 명세**를 만들어(같은 화면을 다시 만들어도 같은 결과) 곧바로 구현으로 이어지게 합니다.

[easyproduct 스킬 세트](../../README.md)의 일부입니다.

## 무엇을 하나

- IA의 화면 정의를 **발판(최소 스펙)**으로 삼아, 빈 상태·로딩·에러·권한 분기·정렬·필터·검색·페이지네이션 같은 실제 화면에 필요한 것을 능동적으로 채워 **풍부하게** 설계합니다.
- 각 화면을 **글 정의(SSOT) + 데이터 정의 + (선택) HTML 목업** 세 조각으로 만들며, 개발이 따르는 표준은 글 정의이고 목업은 눈으로 확인하는 참고물임을 명문화합니다.
- 화면마다 반복되는 **공통 UI 정의**(네비게이션·화면 전역 정책)와 **UI 컴포넌트 인벤토리**를 먼저 확정해, 이후 모든 화면이 이를 참조하도록 UI 드리프트를 막습니다.
- 데이터는 새로 정의하지 않고 **데이터 모델 문서의 변수를 참조**하며, 새 데이터가 필요하면 의도만 넘겨 정식 등록을 요청합니다.
- **한 번의 호출에서 한 범위**(사용자 앱 또는 백오피스)만 다루고, 산출물을 범위별·도메인별 파일로 분리합니다.

## 언제 쓰나

- "화면 설계서", "스토리보드", "화면 기획", "화면 상세 설계", "각 화면을 자세히"를 만들려 할 때.
- "이 화면에 뭐가 보이고, 뭘 누르면 어떻게 되는지"까지 화면별로 구체화하려 할 때.
- "화면 목업", "화면 시안(HTML)", "화면을 눈으로 보게" 만들려 할 때.
- 화면 구조(IA)와 디자인 컨셉은 있는데 이제 각 화면을 실제로 설계하려 할 때(easy-ia-designer·easy-design-concept·easy-data-model 뒤 단계).

## 산출물

- 각 화면을 **글 정의(SSOT) + 데이터 정의 + (선택) HTML 목업**으로 담은 화면 설계서. 글 정의는 컴포넌트·배치·문구가 확정된 명세이고, 데이터 정의는 데이터 모델 문서의 변수를 바인딩해 표기합니다.
- **UI 컴포넌트 인벤토리**(컴포넌트의 SSOT) — 2개 이상 화면에서 쓰이는 재사용 컴포넌트는 중앙 인벤토리(`ssot/ui-components-{scope}.md`)에, 화면 전용은 로컬로 두며 모든 컴포넌트에 ID를 부여합니다.
- **범위별·도메인별로 분리된 파일 집합** — `screens/{scope}/`에 공통 UI 정의를 담은 index 문서 + FEAT 도메인마다 도메인 파일(IA 사이트맵 순서). 목업은 재생성 가능한 소모품으로 `temp/mockups/{scope}/`에만 저장합니다.

## 단독 실행 vs 세트

- **세트에서는 옵트인 단계**입니다. 사용자가 화면 설계를 원할 때만 오케스트레이터가 이 스킬을 호출하며, 디자인 컨셉의 확정은 오케스트레이터가 담당합니다.
- **단독 실행에서는** IA·디자인 컨셉·데이터 모델 문서가 있으면 읽어 쓰고, 없으면 최소 확인 후 진행합니다(디자인 컨셉은 있으면 쓰고 없으면 한 번 확인). IA가 없으면 IA부터 필요함을 알립니다.

## 구성 파일

- `SKILL.md` — 스킬 본문(목적·범위·진행 방식·명세 형식·산출물 규칙·하지 말 것).
- `assets/screen-design-index-template.md` — index 문서 구조(맨 앞 SSOT·재생성 명시 + 공통 UI 정의 + 도메인 링크).
- `assets/screen-design-domain-template.md` — 도메인 파일 구조(전용 로컬 컴포넌트 절 + 화면들의 글 정의·데이터, 사이트맵 순서, 명세 형식).
- `assets/ui-components-template.md` — UI 컴포넌트 인벤토리 형식(중앙, 다국어 확장 주석 포함).
- `assets/screen-mockup-template.html` — 목업 HTML 뼈대(인벤토리 값 대입 자리 + `data-bind` 예시 + 시스템 폰트 fallback + 공통 요소).
- `references/screen-derivation-guide.md` — IA에서 화면 목록·사이트맵 순서 뽑는 법, 도메인 분리, 메인 선정과 공통 UI·인벤토리 anchor 확정, 목록형 분해, 명세 풍부화 체크리스트.

## 세트 안에서의 위치

- **입력**은 IA(기능 ID `FEAT.도메인.기능명`·사이트맵 순서), 확정 디자인 컨셉(토큰·여백 척도·타이포 위계), 데이터 모델 문서입니다. 이들을 재료로만 쓰고, 기능·범위를 IA 밖으로 넓히지 않으며 디자인 컨셉을 새로 확정하지 않습니다.
- **출력**인 화면 설계서와 인벤토리는 API 설계·실제 화면 구현의 근거가 됩니다.
- `easyproduct-scenario`의 **시나리오가 있으면 화면 도출의 드라이버**로 삼아 "이 여정에 어떤 화면이 필요한가"를 도출합니다. 단 화면의 SSOT는 여전히 이 스킬의 글 정의이고, 시나리오는 무엇을 만들지 이끄는 참고입니다.
