# 화면 설계서(스토리보드) 만들기

## 목적

기술 지식이 없는 사람도, IA(화면·기능 구조)·확정된 디자인 컨셉·데이터 모델을 재료로
서비스의 **화면 설계서(스토리보드)를 통째로** 갖추게 돕는다.

한 번의 호출에서는 **한 범위**(사용자 앱 또는 백오피스)의 IA 화면을 전부 만든다(범위는 아래 참고).
각 화면은 세 조각으로 이뤄진다.

1. **글 정의(SSOT)** — 이 화면이 무엇이고, 무엇이 보이고, 무엇을 하면 어떻게 되는지. **개발이 따르는 표준.**
2. **데이터 정의** — 이 화면이 보여주고 주고받는 정보를, 데이터 모델 문서(md)의 **변수를 참조**해 적는다.
3. **(선택) HTML 목업** — 디자인·UI를 눈으로 확인하는 **참고물**. 표준이 아니다.

산출물은 **항상 파일**이다. 파일은 하나가 아니라 **범위별·도메인별로 분리**된 여러 개다(아래 "산출물" 참고).

> **시나리오가 있으면 화면 도출의 드라이버로 삼는다.** `easyproduct-scenario`의 시나리오(사용자 여정)가
> 있으면, 그 흐름을 따라 "이 여정에 어떤 화면이 필요한가"를 도출하는 근거로 쓴다(흐름 → 화면).
> 단 화면의 **SSOT는 여전히 이 화면 설계서의 글 정의**다 — 시나리오는 무엇을 만들지 이끄는 참고이지,
> 화면 정의의 원본이 아니다. 시나리오는 화면·기능을 `FEAT.*`로 참조하므로, 여기서 확정한 ID와 어긋나지 않게 한다.

## 정확한 명세 우선 — 이 스킬만은 "초안 우선"이 아니다

기획서·IA와 달리, 화면설계서는 **사람이 훑어보는 초안이 아니라 곧 구현으로 이어지는 명세**다.
"보이는 것"에 "시작하기 CTA"처럼 뭉뚱그린 명사만 있으면, 같은 화면을 다시 만들 때마다 다른 결과가
나온다. 그래서 이 스킬은 **초안 우선을 버리고, 처음부터 컴포넌트·배치·문구가 확정된 명세**를 만든다.

- 글 정의는 "빠르게 대충 쓰고 나중에 다듬는" 대상이 **아니다.** 컴포넌트·배치·문구가 **확정된 상태로 한 번에** 만든다.
- **메인 화면 컨펌 → 나머지 화면 진행**이라는 리듬은 유지한다. 공통 UI 정의와 컴포넌트 인벤토리를 먼저
  확정해야 나머지 화면이 일관되게 나오기 때문이다. 바뀌는 건 "나머지를 대충 쓴다"가 아니라 **"나머지도
  똑같이 정확하게, 다만 목업(HTML 렌더)만 선택적"**이라는 점이다.
- 목업이 선택적인 원칙은 유지한다. 다만 **글 정의 자체가 목업 없이도 그대로 구현 가능한 수준**이어야 한다.

## 범위 — 사용자 앱 기본, 백오피스는 별도 호출

이 스킬은 **한 번의 호출에서 한 가지 범위만** 다룬다. IA는 프론트(사용자)와 백오피스(관리자)
화면을 함께 담지만, 사용자 앱과 관리자 백오피스는 실제로는 **인증 체계·배포 대상·개발 리포지토리가
다른 별개 프로젝트**인 경우가 대부분이다. 그래서 문서 단계부터 명확히 분리한다.

범위는 둘 중 하나:
- **사용자 앱 (기본값)** — 아무 지정 없이 부르면 이 범위. IA의 **사용자(프론트) 화면**만 대상. scope=`user`.
- **백오피스 (관리자)** — 사용자가 "백오피스도", "관리자 화면도 설계해줘"처럼 **명시적으로 요청할 때만.**
  IA의 **관리자(백오피스) 화면**만 대상. **별도 호출**로 처리한다. scope=`backoffice`.

두 범위는 서로 **독립적으로** 아래를 각자 갖는다(공유하지 않는다):
- 별도의 **메인/기준(anchor) 화면과 컨펌** (사용자 앱: 홈 등 / 백오피스: 관리자 대시보드)
- 별도의 **공통 UI 정의**(사용자 앱: GNB·하단 탭바 / 백오피스: LNB 중심)
- 별도의 **UI 컴포넌트 인벤토리**(`ssot/ui-components-user.md` / `ssot/ui-components-backoffice.md`)
- 별도의 **산출 파일 집합**(도메인별로 분리) — 절대 한 파일에 합치지 않는다.

### 백오피스 범위일 때만 — 디자인 "절제된 변형"
백오피스는 디자인 컨셉을 **새로 확정하지 않는다**(사용자 앱과 **동일한 확정 컨셉을 재료로** 쓴다).
다만 아래 **절제된 변형 규칙**으로 관리자 화면다운 톤으로 조정한다:
- 주색·브랜드 색상은 **유지**하되, 배경·표면 색은 더 무채색에 가깝게(채도를 낮춘다).
- 그림자·모서리 둥근 정도를 줄여 장식성을 낮추고 데이터 중심 톤으로.
- 여백을 줄이고 정보 밀도를 높인다(테이블·목록·필터가 조밀하게).
- 강조색(accent)은 상태 표시(성공/경고/오류/배지) 용도로만 절제해서 쓰고, 장식적으로 남용하지 않는다.
- 이 변형 규칙을 **공통 UI 정의(백오피스 index) 상단**과 **백오피스 인벤토리 상단**에 "사용자 앱 디자인
  컨셉의 절제된 변형"이라고 명시하고, **어떤 값을 어떻게 조정했는지 한 줄씩** 적는다.
- **백오피스의 로컬 컴포넌트도** 이 변형을 동일하게 적용한다(중앙 인벤토리뿐 아니라 로컬도 범위를 벗어나지 않는다).

> **두 범위 공통(변하지 않음):** IA 발판·풍부한 설계(기능·범위는 IA 안), 글 정의=SSOT·목업=참고물,
> 데이터 정의는 **같은 데이터 모델 문서**를 참조(데이터는 공유). 새 데이터 절차(의도만 넘기고 확정 변수 회신).

## 핵심 성격 — 옮겨적기가 아니라 풍부한 설계

이 스킬은 IA를 **그대로 옮겨 적는 스킬이 아니다.** IA의 화면 정의는 "발판(최소 스펙)"일 뿐이고,
이 스킬은 각 화면을 **실제로 쓸 만한 화면**으로 살 붙여 풍부하게, 그리고 **정확하게** 설계한다.

**능동적으로 채우는 것** (IA가 대개 언급하지 않음):
빈 상태·로딩·에러·권한별 분기, 정렬·필터·검색·페이지네이션, 빈 목록일 때 유도 문구,
입력 유효성 피드백, 확인/취소·되돌리기 흐름, 접근성 요소 등.

**창의성의 한계선** (반드시 지킴):
- **기능·규칙·범위는 기획서·IA를 벗어나지 않는다.** 살 붙이는 것은 "그 기능을 화면에서 어떻게 잘
  구현하느냐(상태·상호작용·표현)"이지 **새 기능 추가가 아니다.**
- **새 화면·새 기능이 필요해 보이면 지어내지 말고 제안·확인한다** (IA에 반영이 필요함을 알린다).
- **새 데이터가 필요하면** 아래 규칙대로 데이터 모델 스킬에 **의도만 넘겨 정식 등록**을 요청한다.

## SSOT 원칙 (반드시 문서에 명문화)

- 화면의 **표준·근거는 "글 정의"**다. HTML 목업은 참고물일 뿐이며, **목업의 레이아웃·배치를 표준으로
  삼지 않는다.** 이 문장을 index 문서 맨 앞에 명시한다.
- **데이터는 이 스킬이 정의하지 않는다.** 데이터의 SSOT는 **데이터 모델(easy-data-model이 만든 문서)**이다.
  화면의 데이터 정의는 그 변수를 **참조**하는 것이지, 여기서 새로 만드는 게 아니다.
- **컴포넌트의 SSOT는 UI 컴포넌트 인벤토리**다(아래). 목업은 그 값을 가져다 쓸 뿐, 목업에서 스타일을 새로 짓지 않는다.

## 공통 UI 정의 — 그 범위의 화면들보다 앞에 한 번(사용자 앱과 백오피스는 각자 별도로 가짐), 전 화면이 참조

개별 화면을 그리기 전에, 화면마다 반복되는 공통 요소와 앱 전역 정책을 **한 곳(index 문서)에** 정의한다.
화면마다 다시 정의하지 않고 이 섹션을 **참조**하게 한다(UI 드리프트 방지). 두 묶음으로 나눈다.

> **이 섹션은 화면이 아니라 문서 섹션이다. `FEAT` 화면 ID를 부여하지 않는다**(특정 기능에 속하지 않는
> 횡단 관심사이기 때문). IA가 정한 **네비게이션 구조**(항목 → 목적지(`FEAT` ID) → 노출 조건,
> GNB/LNB/Footer 구성)를 **재료로** 받아, 여기서는 그 구조가 **실제로 어떻게 배치·반응형·상태로
> 구현되는지**를 확정한다.

**(1) 공통 컴포넌트·네비게이션**
- GNB(상단 전역 네비), LNB(측면 네비 — 관리자 등), header, footer, 하단 탭바.
- 공통 버튼/입력/카드/배지, 토스트·모달·다이얼로그 등 반복 요소.
- 이들은 **디자인 컨셉의 컴포넌트 값을 받아** 정의한다(색·글꼴은 디자인 컨셉을 참조). **실제 컴포넌트 정의는
  UI 컴포넌트 인벤토리에 두고**(아래), 여기서는 배치·네비 구조를 확정한다.
- **배치**: 좌/중/우 어디에 무엇이 있는지 확정한다(예: 좌=로고, 중=메뉴, 우=검색·프로필).
- **고정 여부(스티키)**: 헤더·탭바가 스크롤 시 고정되는지.
- **모바일 접힘 방식**: 반응형으로 좁아질 때 메뉴가 어떻게 접히는지(햄버거 메뉴, 하단 시트 등).
- 위 배치·스티키·접힘 값은 **IA가 정의한 항목·목적지(`FEAT` ID)·노출 조건을 그대로 입혀** 확정한다.
  백오피스 범위면 LNB 중심 네비에 같은 원리를 동일하게 적용한다.

**(2) 화면 전역 정책** (앱 전체 공통 결정)
- 기준 화면폭(예: 모바일 375px 기준), 반응형 여부·방식, **브레이크포인트**(있으면), 터치 타깃 최소 크기, 다크모드 범위.
- 디자인 컨셉이 이미 준 값(색·다크모드·접근성)은 **참조**하고, **레이아웃·반응형·기준폭은 여기서 확정**한다.
  이 정책의 집은 **이 문서**다.
- **로딩 상태**의 표준 처리(예: 스켈레톤 UI).
- **빈 상태**의 표준 처리(예: 빈 일러스트 + 안내 문구 + 유도 버튼).
- **권한 없음(로그인 필요) 상태**의 표준 처리 — 아래 넷을 **반드시 포함**: 막는 시점(진입 즉시/특정 동작에서만) /
  처리 방식(리다이렉트/모달) / 복귀 동작(로그인 후 원래 화면 복귀, 대부분 그래야 함) / 문구 톤(디자인 컨셉 톤앤매너 참조).
- **긴 텍스트 처리 표준** — 말줄임(…)·줄바꿈·최대 줄 수 등을 **한 곳에서** 정한다. 개별 화면은 참조만 한다.

  이 상태·처리 표준들을 한 번 정해두면, 개별 화면 정의의 "상태·예외"·"보이는 것"은 이 표준을 **참조만** 하고
  반복 서술하지 않는다(공통 요소와 같은 참조 원리).

  > **이 표준들은 매번 묻지 않고 스킬이 서비스 성격을 보고 먼저 정한다.** (기획서의 정체성·핵심 기능,
  > 로그인 필수형인지 게스트 열람형인지, 서비스 유형.) 예: 게스트 열람 허용형이면 "특정 동작에서 모달로
  > 막고 복귀", 로그인 없이는 아무것도 못 보는 서비스면 "즉시 리다이렉트". 정한 값은 **"자동으로 정한 것"으로
  > 표시**하고, **메인 화면 컨펌 때** 함께 보여준다. 사용자가 바꾸라고 하면 그때 반영한다.

이 공통 UI 정의는 **메인 화면 컨펌 때 함께 확정**한다. 확정된 것이 이후 모든 화면의 **기준(anchor)**이 된다.
각 화면 정의는 "공통 GNB + footer 사용"처럼 **참조만** 하고, 그 화면 **고유 영역만** 서술한다.

## UI 컴포넌트 인벤토리 — 중앙 재사용 + 화면 전용 로컬 (컴포넌트의 SSOT)

컴포넌트를 화면마다 새로 그리면 같은 버튼·카드가 화면마다 달라진다. 그래서 **모든 컴포넌트는 ID를 갖고**,
한 번 정의한 뒤 화면은 그 ID를 **참조**한다. 인벤토리는 md로 유지한다(별도 CSS 파일을 만들지 않는다).

**공통 형식** (중앙·로컬 동일):
```
## UI.카테고리.컴포넌트명[.variant]
- 기본 문구(default): 화면에서 값을 안 넘기면 쓰이는 기본값.
- 파라미터: 호출 시 넘길 수 있는 값(예: label, title, desc). 넘기면 그 값, 안 넘기면 기본값.
- Variants / States: 버튼의 primary/ghost, hover/disabled 등.
- 규격: 크기·radius 등 실제 값(디자인 컨셉의 토큰을 참조해 구체화).
```
- **ID 형식**: `UI.카테고리.컴포넌트명[.variant]`. `UI`는 **대문자 고정**(FEAT·데이터 변수와 안 겹치게).
- **다국어 확장 대비**: 지금은 문구를 문자열 하나로 두되, 나중에 `{ko:"…", en:"…"}` 같은 언어별 키로
  확장 가능한 구조임을 **인벤토리 서두에 한 줄 명시**한다.

### (A) 중앙 인벤토리 — 재사용 원자·분자 단위만
파일: `ssot/ui-components-user.md` (백오피스 범위면 `ssot/ui-components-backoffice.md`, 절제된 변형 적용).
- **재사용 원자·분자 단위만** 등록한다: 버튼·입력창·카드·배지·모달·GNB·footer·탭바 등, **2개 이상 화면에서
  쓰이거나 쓰일 게 뻔한 것.** 특정 화면 하나에만 있는 고유 조합은 등록하지 않는다(→ 로컬).
- **조회 먼저 → 기존 흡수**: 새 컴포넌트가 필요하면 먼저 인벤토리를 조회해 비슷한 게 있으면 재사용하고,
  진짜 다를 때만 신규 등록한다(데이터 모델의 "조회 먼저" 원칙과 동일).

**(A-1) 인벤토리 파일의 시작 입력**
- **없으면**: 메인 화면 컨펌 때 **자동으로 새로 만든다.** 그 화면에서 등장하는 재사용 컴포넌트로 최초 등록.
- **있으면**: 그 파일을 **기준선**으로 삼는다(처음부터 다시 만들지 않는다). 새 컴포넌트가 필요할 때마다 먼저
  이 파일을 조회해 비슷한 게 있으면 재사용, 없으면 그 파일에 **추가**한다.

### (B) 로컬 컴포넌트 — 화면 전용
그 화면(또는 같은 도메인 몇몇 화면)에서만 쓰이고 재사용 가능성이 낮은 컴포넌트는 **중앙에 등록하지 않는다.**
- **정의 위치**: 그 화면이 속한 **도메인 파일**(`screen-design-{scope}-{domain}.md`) 안, 해당 화면의 글 정의
  바로 앞(또는 파일 상단 "이 파일 전용 컴포넌트" 절).
- **ID 형식**: `UI.FEAT.도메인.기능명[.화면종류].컴포넌트명` — 그 화면의 FEAT ID를 그대로 이어붙인다.
  예: 화면 ID가 `FEAT.product.hub`이면 그 전용 컴포넌트는 `UI.FEAT.product.hub.introHero`. ID만 보고
  **어느 화면 전용인지, 중앙 재사용 컴포넌트(`UI.카테고리.컴포넌트`)와 다른지** 바로 구분된다.
- 중앙과 **똑같이** 기본 문구·파라미터·규격을 적는다. **로컬이라도 ID·파라미터화 원칙은 동일**(모든 컴포넌트는
  ID를 갖는다 — 예외 없음).
- 백오피스 범위의 로컬 컴포넌트도 그 범위의 **절제된 변형**을 동일하게 적용한다.

**판단 기준**: "2개 이상 화면에서 재사용/재사용 예정 → 중앙", "그 화면(또는 도메인) 전용 → 로컬".
**승격 경로**: 로컬로 뒀다가 다른 화면·도메인에서도 같은 게 필요해지면 → 중앙으로 옮기고 ID를
`UI.카테고리.컴포넌트`로 바꾼 뒤, **로컬 정의는 삭제하고 그 자리를 중앙 ID 참조로 바꾼다.** 화면을 만들다
"이거 다른 화면에서도 봤는데?" 싶으면 사용자에게 **승격을 제안**한다.

(인벤토리 템플릿: `assets/ui-components-template.md`.)

## 진행 방식 — 메인 먼저 확정, 나머지도 정확하게(목업만 선택)

**메인 컨펌 외에는 자주 멈추지 않는다.** 다만 나머지 화면도 **대충이 아니라 정확한 명세로** 만든다.

> **이 진행은 한 범위 안에서의 흐름이다.** 범위가 다르면(백오피스 요청 시) 처음부터 **별도로 다시** 진행한다.

**1단계 — 메인 화면 풀세트 + 공통 UI + 인벤토리 확정.**
메인 화면(핵심 진입 화면)을 **풀세트**(글 정의 + 데이터 정의 + HTML 목업)로 만든다. 이때 **공통 UI 정의**와
**UI 컴포넌트 인벤토리**(그 화면에 등장하는 재사용 컴포넌트)도 함께 확정한다. 이걸 보여 **컨펌받는다.**
여기서 확정된 것이 이후 모든 화면의 anchor가 된다.

**2단계 — 나머지 전 화면을 글 정의 + 데이터로 정확히 생성(목업은 뒤로).**
나머지 전 화면을 **글 정의 + 데이터 정의**로 만든다. **목업(HTML)만** 미루는 것이지, 글 정의는 컴포넌트·배치·
문구가 **확정된 명세**로 쓴다(3단계에서 목업이 없어도 그대로 구현 가능해야 한다).
- 화면이 나오는 **순서는 IA 사이트맵 순서**를 따른다(홈/허브 → 목록 → 상세 → 쓰기/편집). 임의 순서 금지.
- 목록형 기능은 **목록→상세→작성**으로 분해하고, 기능 ID에 표준 화면종류(`list`/`detail`/`write`/`edit`)를
  붙여 화면 ID를 파생시킨다(예: `FEAT.member.search` → `.list` / `.detail`).
- 새 컴포넌트는 **인벤토리 조회 먼저**(재사용/신규/로컬 판단). 새 데이터는 **모아 두었다가 배치로 한 번에**
  데이터 모델 스킬에 등록 요청하고 확정 변수를 받아 반영한다(화면마다 왕복하지 않는다).

**3단계 — HTML 목업 붙이기 (선택적).**
기본은 **사용자가 원하는 화면만** 골라 목업을 그린다. "전부 그려줘" 하면 순서대로 하나씩. 각 목업은 공통 UI
정의와 **인벤토리 값**을 물려받는다. 시간이 부족하면 **목업 없이 멈춰도 된다**(글 정의·데이터가 이미 완결).
**"설계가 부족하면 실제 개발 과정에서 수정한다"**는 태도를 index 문서에 한 줄 남긴다.

### 충실도 정책 (각 범위 안에서 각자 적용)
- **기본**: 메인만 풀세트, 나머지는 글정의+데이터(목업은 3단계에서 선택적).
- 사용자가 "전 화면 목업까지" 원하면 3단계에서 전 화면을 그린다.
- "요청상세는 목업까지"처럼 특정 화면만 콕 집어 목업을 그릴 수 있다.

화면 목록 뽑기·사이트맵 순서·메인 선정·목록형 분해·풍부화 체크리스트는 `references/screen-derivation-guide.md`.

## 화면 하나의 글 정의 — 구성

IA의 화면 정의를 **발판**으로, 컴포넌트·배치·문구가 확정된 명세로 구체화한다.
(템플릿: `assets/screen-design-domain-template.md`의 화면 항목.)

**화면 제목에 화면 ID를 함께 표기한다.** 예: `### 멤버 찾기 결과 (FEAT.member.search.list)`

- **구현하는 기능** — IA의 **기능 ID**를 그대로 가져온다: `` `FEAT.도메인.기능명` `` (이 스킬이 만들거나 바꾸지 않는다).
- **로그인 필요** — `필수` / `무관(콘텐츠 분리)` / `무관(동일)` 중 하나(IA의 화면 정의·네비 노출 조건에서 판단).
  GNB 메뉴 노출 여부(= IA 네비게이션 노출 조건)와는 **별개**(진입/콘텐츠 문제) — 섞지 않는다.
  - `필수`: 진입 차단. "권한 없음 상태 표준"을 "상태·예외"에서 **참조만**.
  - `무관(콘텐츠 분리)`: "보이는 것"·"할 수 있는 일"을 **로그인/비로그인 두 버전**으로(화면 내부 콘텐츠 분기임을 주석).
  - `무관(동일)`: 한 버전으로 서술.
- **목적** — 이 화면이 왜 있고 무엇을 위한 곳인지 한 문장.
- **진입 경로** — 어디서 이 화면으로 들어오나.
- **보이는 것 (컴포넌트 · 배치 · 문구 — 전부 고정, 순서 있는 번호 목록)** — 아래 형식으로 쓴다:
  ```
  1. [영역 이름]: UI.카테고리.컴포넌트(파라미터="값")  또는  데이터 변수
     - 배치: 위치·정렬을 평이한 말로 (예: "가로 3열 그리드(모바일 1열)", "히어로 아래 좌우 24px 여백")
     - 크기: 폭·높이를 평이한 말로 (예: "카드 균등분할, 높이 auto", "높이 320px")
     - 간격: 디자인 컨셉의 여백 척도를 참조 (숫자를 새로 짓지 않는다. 예: "gap: spacing.md")
     - 텍스트 위계: 디자인 컨셉 타이포 위계 이름으로 참조 (예: "히어로 제목 = 큰제목 tier")
     - 문구: 전환에 중요한 것만 리터럴 고정 (title="…", label="…")
  2. …
  ```
  - **컴포넌트 ID와 순서는 예외 없이 전부 고정한다** — 이게 일관성의 핵심이다.
  - **문구**는 **전환에 중요한 요소(CTA·헤드라인·섹션 제목)만** 리터럴로 고정한다. 부가 설명·툴팁까지 전부
    고정하지 않는다. 판단 기준("전환에 중요한가")을 남긴다.
  - **간격·텍스트 위계**는 새 숫자를 짓지 않고 **디자인 컨셉의 척도·위계 이름을 참조**한다. 디자인 컨셉에
    여백 척도가 없으면 **"디자인 컨셉에 여백 척도 추가 필요"라고 알린다**(직접 만들지 않는다 — 디자인 컨셉 스킬 몫).
  - **반응형 전환 지점**(모바일 1열 등)은 화면 전역 정책의 **브레이크포인트를 그대로 참조**한다.
  - **아이콘·이미지**: 실제 리소스를 만들지 않는다. **"아이콘 자리(placeholder)"**라고 위치만 명시하고, 실제
    리소스는 개발 단계에서 채운다고 못박는다. 자리를 비우거나 대충 넘어가지 않는다.
  - **긴 텍스트**(말줄임/줄바꿈)는 화면 전역 정책의 "긴 텍스트 처리 표준"을 **참조**한다.
  - **로그인 콘텐츠 분리**는 "(조건) → label=…로 교체" 형식으로 이 안에 명시한다.
- **여기서 할 수 있는 일** — 동작마다 "누르면 → 무슨 일이 일어나는지". **동작 대상이 어떤 컴포넌트(UI ID)인지 명시**한다.
- **상태·예외** — 로딩·빈 상태·오류·권한 없음. 전역 표준이 있으면 **참조만**, 이 화면 고유 분기만 서술.
- **목록형이면 필수** — **기본 정렬 기준 · 필터 · 페이지네이션 여부**를 반드시 명시한다(누락 금지).
- **이어지는 화면** — 여기서 갈 수 있는 다음 화면들.
- **레이아웃(영역 배치)** — 헤더·본문 영역·순서를 텍스트 와이어프레임 수준으로. 공통 요소는 공통 UI 정의를 참조, 고유 영역만 상세히.
- **규칙·금지** — 기획서/IA에서 **해당 기능 ID로 정확히 지목**해 인용(중복 서술 안 함). 예: "규칙: `FEAT.member.search`의 조건 참조".

### 기능 ID · 화면 ID · 컴포넌트 ID
- **기능 ID** (IA가 확정 — 만들지도 바꾸지도 않음): `FEAT.도메인.기능명`. 산문에서는 백틱으로 감싼다.
- **화면 ID** (이 스킬이 파생): `FEAT.도메인.기능명.화면종류`. 화면종류는 **표준 어휘만**(`list`/`detail`/`write`/`edit`),
  **정확히 4마디 고정.** 더 세분화가 필요하면 ID를 쪼개지 말고 **본문 텍스트로.** 부모(기능 ID) 없는 화면 ID는 만들 수 없다.
- **컴포넌트 ID**: 중앙 `UI.카테고리.컴포넌트[.variant]` / 로컬 `UI.FEAT.도메인.기능명[.화면종류].컴포넌트명`.
- 어느 기능 ID에도 속하지 않는 새 화면이 필요해 보이면 **지어내지 말고 제안·확인**하고, IA에 새 기능으로 먼저 반영되어야 함을 알린다.

## 화면의 데이터 정의 — 데이터 모델 문서(md) 참조

- **데이터 모델 문서(표)를 직접 읽어**, 화면에서 쓰는 변수를 참조한다(읽기는 파일 직접, 스킬 호출 불필요).
- 각 화면에 **"보여주는 정보(읽기)"**와 **"주고받는 정보(동작 시 입력→출력)"**를 변수명으로 적는다.
  - 예: 보여주는 정보 = `request.title`, `request.price` / 동작 "지원하기" → 보냄`{request.id, user.id}` 받음`{apply.success}`
- 화면 요소·컴포넌트 파라미터에 변수를 **바인딩해 표기**한다: `금액: 15,000원 ← request.price`.
- **HTML 목업에도** 요소에 `data-bind="request.price"`로 변수를 달아 **목업 ↔ 변수 ↔ 글 정의**가 추적되게 한다.

### 데이터가 없거나 애매할 때만 데이터 모델 스킬 호출
- **이미 있는 변수는 데이터 모델 문서에서 찾아 그대로 쓴다**(스킬 호출 안 함).
- 아래일 때만 요청: **새 데이터 필요 / 소속 덩어리 애매 / 중복·충돌 의심 / 새 덩어리 필요.**
- **요청 형식**: 완성된 변수명을 만들지 말고 **"의도"를 넘긴다** — `{ 덩어리 후보, 뜻(한 줄), 종류, 쓰는 화면 }`. 이름은 데이터 모델 스킬이 짓는다.
- 확정 변수 구조체(`variable`·`group`·`nameKo`·`type`·`required`·`description`·`resolution`)를 받으면 그 `variable`을 **그대로 참조**한다.
- **효율**: 2단계에서 새로 필요한 데이터를 **모아 두었다가 배치로 한 번에** 등록 요청한다.

## 디자인 컨셉 사용 — 확정은 이 스킬 밖

- **디자인 컨셉의 "확정"은 이 스킬이 하지 않는다.** 세트에서는 오케스트레이터가, 독립 실행에서는 있으면 쓰고 없으면 한 번 확인.
- 이 스킬은 **확정된 컨셉의 색·글꼴·컴포넌트 토큰·여백 척도·타이포 위계**를 공통 UI 정의·인벤토리·목업에 **입힌다.**
- **여백 척도·타이포 위계 이름을 참조**할 뿐 새 숫자를 짓지 않는다. 컨셉에 없으면 **컨셉에 추가가 필요하다고 알린다**(직접 만들지 않는다).

## 목업 = HTML만, 인벤토리를 참조, temp에 저장

- 목업은 **HTML로만** 만든다(PNG는 기본으로 안 만든다). 한 파일 안에 스타일까지 담고 **시스템 폰트 fallback**을 넣는다.
- **컴포넌트 스타일을 매번 새로 쓰지 않고, UI 컴포넌트 인벤토리(중앙+로컬)의 값을 그대로 가져다 쓴다.** 인벤토리는
  md 하나로 유지하고(별도 CSS 파일 안 만듦), 목업 생성 시 그 값을 대입한다. (템플릿: `assets/screen-mockup-template.html`.)
- **저장 경로**: `temp/mockups/{scope}/화면-[이름]-목업.html` (백오피스는 `화면-[이름]-관리자-목업.html`).
  `temp/`는 **확인 후 버려도 되는 소모품 폴더**다 — 글 정의+데이터 모델+컴포넌트 인벤토리만 있으면 **언제든 재생성 가능**하다.
  **이 폴더는 보관 대상이 아니며 `.gitignore` 대상**임을 index 문서에 명시한다.
- PNG가 꼭 필요하면 디자인 컨셉 스킬의 렌더 방식(`scripts/render_png.py`)을 참조해 뽑을 수 있다고 **안내만** 한다.

## 다른 스킬과의 관계 — 재료로만

- **입력**: IA(화면·기능 구조)와 **IA의 기능 ID(`FEAT.도메인.기능명`)·사이트맵 순서**, 확정 디자인 컨셉(토큰·여백 척도·타이포 위계),
  데이터 모델 문서(md). 있으면 읽어 쓰고, 없으면 최소 확인 후 진행. **IA가 없으면 IA부터 필요함을 알린다.**
- **출력**: 화면 설계서(도메인별 파일 + 인덱스)와 인벤토리는 **API 설계·실제 화면 구현의 근거**가 된다.
- **데이터 공유**: 사용자 앱·백오피스는 **같은 데이터 모델 문서**를 참조한다(프로젝트가 갈려도 데이터 모델은 나누지 않는다).

## 산출물 — 항상 파일로 (범위별·도메인별 분리)

사용자가 "파일로 만들어 달라"고 하지 않아도 **항상 파일로** 만든다. 화면설계서는 **하나의 파일이 아니다.**

```
screens/{scope}/
├── screen-design-{scope}-index.md       ← 공통 UI 정의(1부) + 도메인 파일 링크(IA 사이트맵 순서) + SSOT·재생성 명시
├── screen-design-{scope}-{domain1}.md   ← FEAT.{domain1}.* 화면 전부(사이트맵 순서) + 그 도메인 전용 로컬 컴포넌트(UI.FEAT.*)
├── screen-design-{scope}-{domain2}.md
└── …
ssot/
├── ui-components-user.md                ← 중앙 인벤토리(사용자 앱)
└── ui-components-backoffice.md          ← (백오피스 범위 진행 시)
temp/mockups/{scope}/
└── 화면-[이름]-목업.html                 ← 선택적·참고물 (백오피스는 "관리자" 표기, .gitignore 대상, 재생성 가능)
```

규칙:
- **도메인 단위 분리**: 화면설계서를 하나의 파일에 다 담지 않는다. FEAT 도메인마다 `screen-design-{scope}-{domain}.md`.
  scope는 `user`/`backoffice`. 개별 도메인 파일은 index를 가리키기만 하고 **공통 UI 정의를 복제하지 않는다.**
- **인덱스**: `screen-design-{scope}-index.md`에 공통 UI 정의와 도메인 파일 링크 목록을 담는다. 링크 순서는 IA
  사이트맵에서 그 도메인이 **처음 등장하는 순서.**
- **초대형 도메인**(대략 20개 초과)은 예외적으로 그 안에서 더 나눌 수 있다(예: `…-member-auth.md`).
- **범위 분리**: 사용자/백오피스 산출물을 **한 파일·한 폴더 집합에 합치지 않는다.**
- 런타임 산출물 파일명에만 프로젝트/화면 이름(한글 가능)을 넣는다. 패키지 안 파일명은 ASCII다.

## 비전문가 응대 규칙 (대화 내내)

- **메인 컨펌 외에는 자주 멈추지 않는다.** 나머지 화면은 정확한 글 정의+데이터로 이어서 만들어 보여준다.
- "잘 모르겠어요"라고 하면 **구체적 예시 2~3개**를 제시해 고르게 한다.
- 전문 용어가 나오면 **즉시 한 줄로 풀어** 쓴다. (예: "빈 상태 = 아직 아무 내용이 없을 때 보이는 화면")
- "이건 안 됩니다"보다 "이렇게 해뒀는데 필요 없으면 빼세요"로.

## 재진입 (이미 만든 화면설계서를 다시 받아 고치기)

이미 만든 화면설계서를 다시 가져오면 **처음부터 다시 만들지 않는다.** 두 경우로 나뉜다.

**언제 (가)/(나)인가.** "이 화면 이렇게 바꿔줘"처럼 **내용 변경**이면 **(가)**. "최신 버전으로 맞춰줘/업그레이드/새
항목 채워줘"처럼 **구조 동기화**면 **(나)**. 애매하면 확인한다.

### (가) 사용자 지정 수정
- 가져온 문서를 **기준선**으로, **지정된 화면·항목만** 고친다. 나머지는 그대로 둔다.
- **공통 UI 정의/인벤토리를 고치면**, 그걸 **참조**하는 화면들은 자동 반영되어 재작성 불필요(참조 구조). **단, 참조
  대신 내용을 직접 서술해 둔 경우**엔 그 부분도 고쳐야 함을 짚어준다.
- **FEAT ID·화면 ID·컴포넌트 ID는** 명시적 재배정 요청 없이는 바꾸지 않는다.
- 무엇을 고쳤는지 끝에 한두 줄로 요약한다.

### (나) 스킬 버전업 업그레이드 — 기존 문서를 최신 구조로
1. 가져온 기존 산출물을 **기준선**으로 삼는다(처음부터 다시 만들지 않는다).
2. 그 구조를 **현재 버전의 템플릿·구조와 항목 단위로 비교**한다 — index 템플릿(`assets/screen-design-index-template.md`),
   도메인/화면 템플릿(`assets/screen-design-domain-template.md`), 인벤토리 템플릿(`assets/ui-components-template.md`).
   **지금 구조엔 있는데 기존 문서엔 없는 것**을 **빠진 부분(gap)**으로 표시한다. 특정 필드에 국한하지 말고 **템플릿·구조와
   비교해 실제로 빠진 것을 찾는 일반 절차**로 수행한다(향후 또 바뀌어도 재사용). 이번 개편 기준 흔한 gap 예: 하나의 파일을
   **도메인별로 분리** + index, **UI 컴포넌트 인벤토리**(중앙/로컬), "보이는 것"의 **명세 형식**(컴포넌트·배치·문구 고정),
   **긴 텍스트 처리 표준**, 목업의 **temp 경로**, 화면·도메인의 **사이트맵 순서**, 화면별 "로그인 필요" 필드 등.
3. **빠진 부분만 채운다.** 채울 때 그 항목을 새로 만들 때 이 스킬이 쓰는 **자동 판단 기준**을 그대로 적용하고, 매번 묻지
   않고 **"자동으로 정한 것"으로 표시**한다.
4. **이미 있는 내용은 절대 다시 쓰거나 고치지 않는다.** 순수 "빠진 것만 추가". 기존 글 정의·데이터·목업·레이아웃·ID는 글자 하나 안 건드린다.
5. **사용자 앱·백오피스가 둘 다 있으면 각각 독립적으로** 같은 절차를 적용한다(한 파일/한 폴더 집합으로 합치지 않는다).
6. 끝나면 **무엇을 새로 채웠는지 화면별로 요약**해 알려준다.
7. 파일은 **제자리에 업데이트**한다(도메인 분리 gap을 채울 땐 기존 내용을 그대로 옮겨 나누되 재작성하지 않는다).

## 하지 말 것

- **IA를 얇게 옮겨적지 말 것**(발판으로 삼아 풍부하게). 단, **기능·범위를 IA 밖으로 넓히지 말 것**(새 기능은 제안·확인).
- **화면 정의의 "보이는 것"을 뭉뚱그린 명사 나열로 두지 말 것** — 컴포넌트·배치·문구·간격·텍스트 위계가 전부 고정되거나 명시적으로 참조된 명세여야 한다.
- **인벤토리에 화면 고유의 특수 조합까지 다 등록하지 말 것** — 재사용 단위만(고유 조합은 로컬).
- **모든 문구를 다 리터럴로 고정하려 하지 말 것** — 전환 핵심 요소만.
- **화면설계서를 도메인 구분 없이 하나의 파일에 다 담지 말 것.**
- **화면·도메인 나열 순서를 IA 사이트맵과 무관하게 임의로 정하지 말 것.**
- **여백·간격 숫자를 화면마다 새로 짓지 말 것** — 디자인 컨셉의 척도를 참조(없으면 컨셉에 추가 필요를 알림).
- **아이콘·이미지 리소스를 이 스킬이 직접 만들려 하지 말 것** — 자리(placeholder) 표시만.
- **목록형 화면에서 정렬·필터·페이지네이션을 빠뜨리지 말 것.**
- **목업을 `ssot/`나 `screens/` 등 보관 대상 폴더에 저장하지 말 것** — `temp/mockups/{scope}/`에만.
- **컴포넌트에 ID를 안 붙이지 말 것**(중앙·로컬 모두 ID 필수, 예외 없음).
- **목업에서 컴포넌트 스타일을 새로 짓지 말 것**(인벤토리 값을 가져다 씀. 컴포넌트 SSOT는 인벤토리).
- **목업을 표준으로 삼지 말 것**(SSOT는 글 정의).
- **데이터를 이 스킬이 새로 정의하지 말 것**(변수는 데이터 모델 문서 참조, 새 데이터는 의도만 넘겨 확정 변수 수신).
- **공통 요소·화면 전역 정책을 화면마다 다시 정의하지 말 것**(공통 UI 정의 참조).
- **디자인 컨셉을 이 스킬이 새로 확정하지 말 것**(입힐 뿐).
- **기능 ID를 새로 짓거나 IA와 다르게 바꾸지 말 것.** 화면종류 세그먼트에 표준 어휘 외 임의 단어 금지. 화면 ID 4마디 초과 금지.
- **한 호출에서 사용자 앱과 백오피스를 섞어 만들지 말 것**(기본값=사용자 앱, 요청 없으면 백오피스 안 만듦).
- **"로그인 필요" 필드와 GNB 노출 조건을 섞어 쓰지 말 것.** `무관(콘텐츠 분리)`을 다른 값으로 뭉뚱그리지 말 것.
- **권한 없음·긴 텍스트 등 전역 표준을 정할 때마다 먼저 묻지 말 것** — 스킬이 판단해 정하고 컨펌은 메인 컨펌 때.
- **재진입 시 문서를 처음부터 다시 만들지 말 것.** (나)에서 이미 있는 내용을 재작성하거나, 특정 필드 하나만 하드코딩해 처리하지 말 것(템플릿·구조 비교 일반 절차로).
- **두 화면설계서(사용자/백오피스)를 업그레이드하며 한 파일로 합치지 말 것.**
