# 화면 도출 가이드 (IA → 화면 목록·사이트맵 순서 · 도메인 분리 · 메인 선정 · 분해 · 명세 풍부화)

화면 설계서를 만들 때 이 파일을 참고한다. IA(화면·기능 구조)를 **발판**으로 삼아
화면을 빠짐없이 뽑고, 각 화면을 실제 쓸 만하게 살 붙인다. **기능·범위는 IA를 벗어나지 않는다.**

## 목차
1. IA에서 화면 목록·사이트맵 순서 뽑기 + 도메인 분리
2. 메인(기준 화면) 선정 + 공통 UI·컴포넌트 인벤토리를 anchor로 확정
3. 목록형 화면의 분해 (목록 → 상세 → 작성/편집)
4. 관리자(백오피스) 화면 포함 원칙
5. 명세 풍부화 체크리스트 (보이는 것 명세 · 상태·예외·상호작용·접근성)
6. IA 밖으로 나가야 할 때 (제안·확인)

---

## 1. IA에서 화면 목록·사이트맵 순서 뽑기 + 도메인 분리

- IA 문서의 **"화면 지도(사이트맵)"와 "화면 정의"**가 1차 출처다. 거기 적힌 화면을 전부 목록으로 옮긴다.
- IA의 "기능 목록"도 훑는다. **기능 하나가 화면 하나는 아니다** — 목록형 기능은 여러 화면으로 갈린다(§3).
- **순서는 IA 사이트맵 순서를 그대로 따른다**: 홈/허브 → 목록 → 상세 → 쓰기/편집. 임의 순서(알파벳·만든 순서) 금지.
  화면설계서를 IA와 나란히 놓고 검토할 때 순서가 어긋나지 않게 한다.
- **도메인 단위로 파일을 나눈다.** FEAT 도메인(`FEAT.{domain}.*`)마다 `screen-design-{scope}-{domain}.md` 하나.
  도메인 파일이 **인덱스에 나열되는 순서**도 사이트맵에서 그 도메인이 **처음 등장하는 순서**를 따른다.
  초대형 도메인(대략 20개 초과)만 예외적으로 그 안에서 더 나눈다(예: `…-member-auth.md`).
- **사용자 화면과 관리자 화면은 범위가 다르다** — 별도 호출·별도 파일 집합(§4). 한 호출은 한 범위만.
- IA가 없으면 화면 목록·순서의 출처가 없다 — **IA부터 필요함을 사용자에게 알린다.**

## 2. 메인(기준 화면) 선정 + 공통 UI·컴포넌트 인벤토리를 anchor로 확정

**메인 = 핵심 진입 화면.** 대개 홈, 또는 그 서비스의 가장 중요한 행동이 시작되는 화면.
(예: 배달앱=홈/가게목록, 커뮤니티=글 목록, 트래커=오늘 기록 화면.)

메인을 먼저 **풀세트**(글 정의 + 데이터 + 목업)로 만드는 이유:
- 메인은 **공통 요소(GNB·탭바·카드·버튼 등)가 처음 등장하는 화면**이다. 그래서 메인을 그리며
  **공통 UI 정의**(공통 컴포넌트·네비 + 화면 전역 정책)를 함께 확정한다.
- 여기서 확정된 색·글꼴·컴포넌트·기준폭·반응형·터치타깃·다크모드가 **이후 모든 화면의 anchor**가 된다.
- 메인을 그리며 그 화면에 등장하는 **재사용 컴포넌트를 UI 컴포넌트 인벤토리에 최초 등록**한다(없으면 새로 만들고,
  있으면 기준선으로 삼아 조회 먼저). 이 인벤토리도 메인 컨펌 때 함께 확정한다.
- 그래서 메인만 **사용자 컨펌**을 받는다. 나머지 화면은 이 anchor(공통 UI + 인벤토리)를 물려받아
  **정확한 명세로** 만든다(대충 쓰지 않는다 — 목업만 선택적).

확정할 공통 UI 정의(설계서 1부에 남김):
- (1) 공통 컴포넌트·네비: GNB / (폰이면)하단 탭바 / (관리자면)LNB / header·footer / 버튼·입력·카드·배지 / 토스트·모달
  \+ **배치(좌/중/우)·스티키·모바일 접힘** — IA의 항목·목적지(`FEAT` ID)·노출 조건을 입혀 확정.
- (2) 화면 전역 정책: 기준 화면폭 / 반응형·브레이크포인트 / 터치 타깃 최소 크기 / 다크모드 범위 / 접근성 기본
  \+ **로딩·빈·권한 없음·긴 텍스트 처리의 표준**(정해두면 각 화면 "상태·예외"·"보이는 것"은 참조만).
- 색·글꼴·다크·접근성·**여백 척도·타이포 위계 값**은 디자인 컨셉을 참조. **레이아웃·반응형·기준폭은 여기서 확정.**
- 공통 UI 정의는 **인덱스 파일**(`screen-design-{scope}-index.md`)에 남긴다 — 도메인 파일은 복제하지 않고 참조만.

## 3. 목록형 화면의 분해 (목록 → 상세 → 작성/편집)

목록형 기능(글·상품·요청·예약·모임 등)은 보통 **세 화면**으로 갈린다. IA에 한 줄로 있어도 여기서 나눈다.
분해할 때 그 **기능 ID**에 표준 화면종류를 붙여 **화면 ID**를 파생시킨다: `FEAT.도메인.기능명` +
`list`/`detail`/`write`/`edit` → 예 `FEAT.member.search.list`, `.detail`, `.write`. (4마디 고정, 부모 기능 ID 필수.)

- **목록 화면**: 여러 개를 카드/행으로. 정렬·필터·검색·페이지네이션, **빈 목록 상태**(유도 문구·버튼).
- **상세 화면**: 하나를 자세히. 주요 정보 + 핵심 행동(신청·구매·수정 등) + 이어지는 화면.
- **작성/편집 화면**: 입력 폼. 입력 유효성 피드백, 저장/취소, **되돌리기**, 미저장 이탈 경고.

각 화면의 데이터 정의는 데이터 모델 변수를 참조한다(목록=여러 건 읽기, 상세=한 건 읽기, 작성=입력→출력).

## 4. 관리자(백오피스) 화면 포함 원칙

- **사용자 화면만 있고 관리자 화면이 없는 설계서는 미완성이다.** 회원·콘텐츠·주문·문의가 생기면
  그걸 볼 사람이 필요하다. IA의 백오피스 목록을 그대로 화면으로 가져온다.
- 관리자 화면은 대개 **LNB(좌측 메뉴) + 목록/상세/처리** 패턴. 공통 UI 정의에 LNB를 넣는다.
- 규모가 아주 작은 개인 프로젝트면 IA를 따라 **"관리자 최소본"**(회원·문의·통계)만 가볍게 잡고 그렇게 표시.
- 관리자 화면도 §5 풍부화(권한 없음·빈 상태·처리 확인/취소)를 똑같이 적용한다.

## 5. 명세 풍부화 체크리스트 (보이는 것 명세 · 상태·예외·상호작용·접근성)

각 화면 글 정의를 쓸 때, 아래를 **빠짐없이** 살핀다. IA가 대개 언급하지 않으므로 **능동적으로 채운다.**
단, **새 기능을 만들지는 않는다** — "그 기능을 화면에서 어떻게 잘 구현하나"에 한정한다.

**보이는 것 = 명세 (뭉뚱그린 명사 나열 금지)**
- [ ] 각 요소가 **컴포넌트 ID**(중앙 `UI.카테고리.컴포넌트` 또는 로컬 `UI.FEAT.…`)로 지정됐나 — 없으면 인벤토리에 등록/로컬 정의
- [ ] **순서**(번호 목록)가 고정됐나
- [ ] 배치·크기·**간격(디자인 컨셉 여백 척도 참조)**·**텍스트 위계(타이포 위계 이름 참조)**가 적혔나 — 새 숫자 짓지 않기
- [ ] 전환 핵심 **문구(CTA·헤드라인·섹션 제목)**만 리터럴로 고정했나
- [ ] 반응형 전환 지점은 **전역 브레이크포인트 참조**, 아이콘·이미지는 **자리(placeholder)만** 표시
- [ ] 긴 텍스트는 전역 "긴 텍스트 처리 표준" 참조, 동작 대상은 **어떤 컴포넌트(UI ID)인지** 명시
- [ ] 로그인 콘텐츠 분리면 "(조건) → label=…로 교체" 명시

**상태**
- [ ] 로딩 중(불러오는 동안 무엇을 보이나)
- [ ] 빈 상태(내용 없음 → 무엇을 보이고 어떤 유도 문구·버튼으로 첫 행동을 이끄나)
- [ ] 정상(내용 있음)
- [ ] 부분 상태(일부만 있음, 더 불러오기)

**예외**
- [ ] 오류(불러오기 실패·인터넷 끊김 → 재시도 안내)
- [ ] 권한 없음(로그인 안 함·접근 불가 → 어디로 유도)
- [ ] 입력 오류(잘못된 값·빈 칸 → 어디에 어떤 피드백)
- [ ] 충돌·중복(이미 신청함·품절 등 규칙 위반 시)

**상호작용**
- [ ] **정렬(기본 정렬 기준)·필터·페이지네이션 — 목록형이면 반드시 명시(누락 금지)**
- [ ] 확인/취소, 되돌리기(실수 복구)
- [ ] 저장 결과 피드백(토스트·이동)
- [ ] 미저장 이탈 경고(작성/편집 화면)

**접근성** (디자인 컨셉의 값을 참조하되 화면에서 지킴)
- [ ] 글자·배경 대비 충분
- [ ] 색만으로 정보 구분하지 않기(아이콘·글자 병행)
- [ ] 터치 타깃 최소 크기(화면 전역 정책 값) 준수
- [ ] 포커스·라벨(입력창에 라벨, 버튼에 명확한 이름)

## 6. IA 밖으로 나가야 할 때 (제안·확인)

설계 중 "이 기능엔 이런 **새 화면·새 기능**이 더 필요해 보인다"고 판단되면:
- **지어내지 않는다.** 설계서 부록 "IA 반영이 필요한 제안"에 한 줄로 적고, 사용자에게 **제안·확인**한다.
  (화면 ID의 부모가 될 **기능 ID가 IA에 없으면 화면을 만들 수 없다** — 새 기능으로 먼저 IA에 반영돼야 한다.)
- 승인되면 IA에 반영이 필요함을 알린다(이 스킬은 IA를 직접 고치지 않는다).

새 **데이터**가 필요하면(화면에 보여줄 값이 데이터 모델에 없음):
- 데이터 모델 문서를 먼저 읽어 이미 있는지 확인한다(있으면 그대로 참조, 스킬 호출 안 함).
- 없으면 데이터 모델 스킬에 **의도만** 넘긴다: `{ 덩어리 후보, 뜻(한 줄), 종류, 쓰는 화면 }`.
  이름은 데이터 모델 스킬이 짓는다. 확정 변수(`variable`)를 받아 그대로 참조한다.
- **2단계(전 화면 글 정의)에서 모아 두었다가 배치로 한 번에** 등록 요청하면 왕복이 준다.
