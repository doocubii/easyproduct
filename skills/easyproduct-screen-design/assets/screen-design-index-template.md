# 화면 설계서 · 인덱스 ({scope}) : [프로젝트 이름]

> 범위(scope): `user`(사용자 앱) 또는 `backoffice`(관리자). 이 파일은 **그 범위 하나**의 인덱스입니다.
> 사용자 앱과 백오피스는 **각자 별도의 index·파일 집합**을 가지며, 한 파일/한 폴더로 합치지 않습니다.

## ⚠️ 이 문서를 읽는 규칙 (반드시 지킴)
- **개발이 따르는 표준·근거는 "글 정의"입니다.** 각 도메인 파일의 화면 글 정의가 SSOT이고, HTML 목업은 **참고물**입니다.
- **데이터 SSOT는 데이터 모델 문서**입니다. 화면의 데이터 정의는 그 변수를 참조할 뿐입니다.
- **컴포넌트 SSOT는 UI 컴포넌트 인벤토리**(`ssot/ui-components-{scope}.md`)입니다.
- **목업(`temp/mockups/{scope}/`)은 보관 대상이 아니며 `.gitignore` 대상입니다.** 글 정의+데이터 모델+인벤토리만 있으면 언제든 재생성됩니다.
- 설계가 부족한 곳은 실제 개발 과정에서 수정합니다.

## 한 줄 요약
누구를 위한 무엇이고, 어떤 종류의 서비스인지 한 문장.

## 재료
- 화면 구조(IA): (파일명) — 화면 목록·**사이트맵 순서**·기능 ID(`FEAT.*`)의 출처
- 디자인 컨셉: (파일명·선택된 안) — 색·글꼴·컴포넌트 토큰·여백 척도·타이포 위계의 출처
- 데이터 모델: (파일명) — 데이터 변수의 출처
- UI 컴포넌트 인벤토리: `ssot/ui-components-{scope}.md`

# 1부. 공통 UI 정의  (전 화면이 물려받는 기준 · anchor)

> **이 부는 화면이 아니라 문서 섹션입니다. `FEAT` 화면 ID를 붙이지 않습니다.** IA가 정한 네비게이션
> 구조(항목→목적지(`FEAT` ID)→노출 조건, GNB/LNB/Footer)를 재료로, 여기서 실제 배치·반응형·상태를 확정합니다.
> (백오피스 범위면 상단에 "사용자 앱 디자인 컨셉의 절제된 변형 — 무엇을 어떻게 조정했는지"를 한 줄씩 명시.)

## (1) 공통 컴포넌트·네비게이션
> 실제 컴포넌트 정의(문구·파라미터·규격)는 `ssot/ui-components-{scope}.md`에 있습니다. 여기서는 배치·네비 구조를 확정합니다.
- **GNB / (폰)하단 탭바 / (관리자)LNB / header / footer**: 무엇이 들어가나.
- **배치**: 좌/중/우 무엇이 있나 (예: 좌=로고, 중=메뉴, 우=검색·프로필).
- **고정 여부(스티키)**: 헤더·탭바가 스크롤 시 고정되나.
- **모바일 접힘 방식**: 좁아질 때 메뉴가 어떻게 접히나 (햄버거/하단 시트 등).
  → 위 배치·스티키·접힘은 **IA의 항목·목적지(`FEAT` ID)·노출 조건을 입혀** 확정합니다.

## (2) 화면 전역 정책 (이 정책의 집은 이 문서)
- **기준 화면폭 / 반응형·브레이크포인트 / 터치 타깃 최소 크기 / 다크모드 범위 / 접근성 기본** (색·다크·접근성 값은 디자인 컨셉 참조).
- **로딩 상태 표준**: (예: 스켈레톤 UI)
- **빈 상태 표준**: (예: 빈 일러스트 + 안내 문구 + 유도 버튼)
- **권한 없음(로그인 필요) 상태 표준** — 넷을 반드시 포함: 막는 시점 / 처리 방식(리다이렉트·모달) / 복귀 동작 / 문구 톤.
- **긴 텍스트 처리 표준**: 말줄임(…)·줄바꿈·최대 줄 수 등.
  → 이 표준들을 정해두면 각 화면의 "상태·예외"·"보이는 것"은 여기를 **참조만** 하고 반복 서술하지 않습니다.
  → *자동 모드에서는 서비스 성격을 보고 스킬이 먼저 정하고 "자동으로 정한 것"에 표시, 메인 컨펌 때 함께 확인받습니다.*
