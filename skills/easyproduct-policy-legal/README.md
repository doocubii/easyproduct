# easyproduct-policy-legal

> 개발·법률 지식이 없는 사람도 핵심 몇 가지만 답하면, 서비스 운영에 필요한 **정책서**와 **이용약관·개인정보처리방침**을 갖추도록 돕는 스킬입니다. 특히 법적 효력 문서는 완성된 법률 문서가 아니라 **빠짐없이 준비하도록 돕는 초안·점검표**이며, 실제 사용 전 **반드시 법률 전문가의 검토가 필요**합니다.

[easyproduct 스킬 세트](../../README.md)의 일부입니다.

## 무엇을 하나
- 서비스 운영 규칙을 담은 **정책서**를 실제 운영 규칙 문장으로 완성합니다(회원·콘텐츠·결제·환불·신고 등, 해당되는 것만).
- **이용약관 + 개인정보처리방침**을 **1부 완결 초안 → 2부 확인 항목 → 3부 점검표** 구조로 만들고, 문서 앞·뒤에 전문가 검토 경고를 넣습니다.
- 기획서·IA·설계서가 있으면 거기서 수집 항목·처리 목적·켜질 조항 등을 **최대한 추출**하고, 사용자·전문가만 아는 값(보유기간·수탁업체·관할 등)만 `[확인 필요]`로 남깁니다.
- 정책서의 각 규칙에 참조 가능한 **`POL.*` 안정 ID**(`POL.도메인.규칙명`)를 붙여, 시나리오·화면 설계서 등이 규칙을 이름이 아니라 ID로 참조하게 합니다.
- 법률 용어를 풀어 쓰고, 표준 조항 문안은 끝까지 작성하되 **고유값만 비우며**, 소비자 권리(청약철회·환불)를 불리하게 축소하지 않습니다.

## 언제 쓰나
- "정책서", "서비스 정책", "운영 규칙", "이용약관", "약관 만들어줘", "개인정보처리방침", "환불/취소 정책"을 만들거나 점검하려 할 때
- "이 서비스에 약관이 뭐가 필요하지?", "개인정보 뭘 넣어야 하지?"처럼 무엇을 갖춰야 할지 고민할 때
- 서비스를 런칭·오픈하기 전에 꼭 갖춰야 할 문서를 챙기려 할 때
- 기획서(`easyproduct-doc-builder`)·화면 구조(`easyproduct-ia-designer`)를 만든 뒤 정책·약관 단계로 넘어가려 할 때

## 산출물
- **서비스 정책서** (`정책서-[서비스이름].md`) — 운영 규칙을 실제 규칙 문장으로, 규칙별 `POL.*` ID 부착
- **이용약관 + 개인정보처리방침** (`약관-개인정보처리방침-[서비스이름].md`) — 1부 완결 초안 → 2부 확인 항목 → 3부 점검표를 한 파일로, 앞·뒤 검토 경고 포함
- 기본은 마크다운(.md), 원하면 Word(.docx)로도 제공

## 단독 실행 vs 세트
- **표준 정책·약관 스킬로 단독 사용이 1순위**입니다. 자료가 없어도 성격 파악 질문(4~5개)만으로 표준 뼈대를 완성합니다.
- 기획서·IA·설계서가 있으면 먼저 읽고 최대한 추출한 뒤, 빈 값만 최소로 확인합니다.
- 이미 만든 문서를 고칠 때는 처음부터 다시 만들지 않고, 기준선을 보존한 채 **지정된 곳만** 수정합니다(`#n` 양방향 연결·검토 경고·1·2·3부 구조 유지).

## 구성 파일
- `SKILL.md` — 스킬의 목적·원칙·전체 흐름(신규 생성 Step 0~4, 재진입·부분 수정 Step 5)
- `assets/정책서-템플릿.md` — 서비스 정책서(운영 규칙)의 읽기 쉬운 구조
- `assets/약관-개인정보-템플릿.md` — 이용약관 + 개인정보처리방침(1부 완결 초안 → 2부 확인 항목 → 3부 점검표, 완결 초안 예시 포함)
- `references/약관-표준조항.md` — 이용약관이 반드시 포함할 표준 조항 목록(누락·발췌 방지, 조항 켜기·끄기 재판정 기준)
- `references/개인정보처리방침-표준항목.md` — 개인정보처리방침이 반드시 포함할 표준 항목 + 문안 뼈대
- `references/개인정보-법정항목-체크리스트.md` — 법정 기재사항 + 해당 여부 판단 + "문서에서 채울 수 있나"
- `references/성격파악-질문가이드.md` — 독립 실행 시 질문 → 어떤 조항·문서가 켜지는지 매핑, "모르겠어요" 대응
- `references/정책-ID체계.md` — 정책 규칙의 참조 가능한 안정 ID(`POL.도메인.규칙명`) 체계·표준 도메인 매핑

## 세트 안에서의 위치
- 세트 안에서는 기획서·IA 이후의 **정책·약관 단계**를 맡습니다. 자료가 있으면 앞 단계 산출물에서 규칙·수집 항목·켜질 조항을 뽑아 씁니다.
- 정책서에 부여한 **`POL.*` ID**는 정책 규칙의 원본(SSOT)입니다. `easyproduct-scenario`의 시나리오와 화면 설계서·유스케이스가 이 ID로 규칙을 참조하므로, 한 번 정한 ID는 문구가 바뀌어도 유지합니다.
- 개인정보처리방침은 이용약관과 한 묶음으로 취급하며, 안전 원칙·검토 경고는 어떤 산출 조합에서도 동일하게 적용됩니다.
