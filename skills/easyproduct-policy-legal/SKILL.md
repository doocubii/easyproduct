# 정책서·약관 만들기

## 목적

법률 지식이 전혀 없는 사람도, **핵심 몇 가지만 답하면** 서비스 운영에 필요한 문서 두 종류를
**초안과 점검표**로 갖추게 돕는다.

1. **서비스 정책서** — "우리 서비스를 이렇게 운영한다"는 운영 규칙. (회원·콘텐츠·결제·환불·신고 등)
2. **이용약관 + 개인정보처리방침** — 법적 효력이 있는 문서. **완결 초안**(복사해 바로 쓰는 전문)을 맨 앞에 두고,
   그 안의 고유값을 모은 **확인 항목**, 빠짐을 보는 **점검표**를 뒤에 붙인 뒤, **전문가 검토가 필요함을 명시**한다.

> **이 스킬은 법률 자문을 하지 않는다.** 특히 (2)는 완성된 법률 문서가 아니라, 무엇을 갖춰야 하는지
> 빠짐없이 준비하도록 돕는 **초안·점검표**다. 실제 서비스에 쓰기 전 **반드시 법률 전문가의 검토**를 받아야 한다.
> 이 경고는 대화에서도, 만들어지는 법적 문서의 **맨 앞과 끝에도** 넣는다.

## 핵심 원칙

1. **비전문가 언어.** 법률 용어를 먼저 꺼내지 않는다. 불가피하면 한 줄로 풀어 쓴다. ("청약철회 = 주문을 취소하고 돈을 돌려받는 것")
2. **초안 우선.** 매번 묻고 멈추지 않는다. 성격을 파악한 뒤 **먼저 초안을 통째로 만들고**, 사용자가 보고 고치게 한다.
3. **해당되는 것만 넣는다.** 서비스에 없는 기능(결제·위치·신고 등)의 조항은 넣지 않는다. 자동으로 켜고 끄는 기준은 `references/성격파악-질문가이드.md`.
4. **문서에서 최대한 채우고, 못 채우는 값만 비운다.** 기획서·IA가 있으면 `[확인 필요]`를 뭉뚱그려 비우지 말고 **두 층으로 가른다.** (ㄱ) 문서에서 **추론 가능한 것**(수집 항목·처리 목적·켜질 조항·위탁의 존재/유형·대상 등)은 채우고 "문서에서 가져온 것/가정"으로 표시한다. (ㄴ) 문서가 있어도 **사용자·전문가만 아는 값**(보유기간·보호책임자·수탁 업체명·국외이전 상세·사업자정보·관할·청약철회 세부·실제 안전조치)만 `[확인 필요: …]`로 남긴다. 어느 쪽도 **지어내지 않는다.** 판별 기준은 `references/개인정보-법정항목-체크리스트.md`의 "문서에서 채울 수 있나" 열을 따른다.
5. **전문가 검토는 타협 없음.** 법적 효력 문서의 앞·끝에 검토 경고를 반드시 넣는다. 사용자가 "그냥 완성본으로 써도 되지?"라고 해도, 초안이라는 성격과 검토 필요성을 분명히 한다.
6. **있는 건 묻지 않는다.** 기획서·IA·설계서가 있으면 거기서 먼저 뽑고, 빈 곳만 최소로 확인한다.
7. **단독으로도 완성한다.** 자료가 없어도 성격 파악 질문만으로 표준 뼈대를 만든다. "기획서가 없어서 못 만든다"고 하지 않는다.
8. **결과는 항상 파일로.** 정책서, 약관·개인정보처리방침을 각각 **문서 파일**로 만든다. 채팅에만 늘어놓지 않는다.

## 무엇을 만들지 (부분 생성 가능 — 디폴트는 둘 다)

이 스킬은 두 산출물을 낸다. 사용자 요청에 따라 갈린다.

- **"정책서만"** → 서비스 정책서만 만든다.
- **"약관만" / "이용약관만" / "개인정보처리방침만"** → 약관·개인정보처리방침만 만든다.
  (개인정보처리방침은 이용약관과 **한 묶음**으로 취급한다 — 둘 중 하나만 콕 집어도 이 묶음으로 처리한다.)
- **별말이 없으면** → **둘 다** 만든다 (디폴트).

하나만 콕 집어 요청하면 나머지는 만들지 않는다. **어느 경우든 안전 원칙·검토 경고는 똑같이 적용**한다.

## 실행 방식 (독립 실행이 기본, 세트로도 물림)

이 스킬은 easy-dev 계열과 세트로 물려 돌 수도 있지만, **표준 정책·약관 스킬로 단독 사용이 1순위**다.

- **(연계)** 기획서·IA·설계서가 있으면 **먼저 읽고 최대한 추출**한다. 서비스 성격·규칙·금지, 회원/결제/신고/위치/마케팅 여부를 문서에서 뽑아, 아래 성격 파악 질문은 **건너뛴다.**
- **(독립)** 아무 자료가 없으면, 아래 **성격 파악 질문**만으로 서비스 성격을 파악해 표준 문서를 만든다. 기획서가 없어도 **표준 뼈대는 단독으로 완성**한다.

## Step 0: 시작 (짧게)

길게 늘어놓지 않는다.

- "서비스 운영에 필요한 **정책서**와, **이용약관·개인정보처리방침**을 준비하도록 도와드릴게요. 법률 용어는 몰라도 됩니다."
- **딱 한 번, 성격을 분명히 한다:** "약관·개인정보처리방침은 제가 **초안과 점검표**로 만들어 드리는 거예요. 실제로 쓰시기 전에는 **법률 전문가의 검토**가 꼭 필요합니다."
- **자료 확인:** "혹시 만들어 두신 기획서·화면 구조·설계서나 메모가 있으면 먼저 주세요. 있으면 거기서 최대한 뽑고, 모자란 것만 여쭤볼게요."
- **무엇을 만들지 확인(필요할 때만):** 사용자가 콕 집지 않았으면 둘 다 만든다. "정책서만/약관만" 같은 말이 있으면 그것만.

## Step 1: 성격 파악 — 추출 우선, 없으면 질문

**자료가 있으면**: 먼저 읽고 **최대한 뽑아낸다.** 성격 파악 질문가이드의 매핑에 맞춰 서비스명·운영주체·대상, 회원/결제/위치/민감정보/마케팅 여부를 판단하는 데서 그치지 말고, 체크리스트 항목의 **내용까지** 문서에서 도출한다:
- **개인정보 수집 항목** ← IA의 회원가입·프로필 화면 요소 + 기획서의 "핵심 정보 덩어리". (예: 회원가입 화면에 이메일·닉네임 → 그대로 수집 항목으로)
- **처리 목적** ← 각 기능의 목적. (회원관리·서비스 제공·결제처리·문의응대 등)
- **켜질 조항** ← 기능 목록. (결제 기능 있음 → 결제·환불·청약철회 켬 / 신고 기능 → 신고·제재 / 위치 → 위치기반 / 알림 → 마케팅)
- **위탁의 존재·유형** ← 기능에서. (결제 → 결제대행 위탁 존재, 문자·푸시 알림 → 발송 대행 존재. **업체명은 미정이므로 `[확인 필요]`**)
- **정책서의 규칙·금지** ← 기획서의 규칙·금지·성공 기준을 그대로.
이렇게 채운 건 "문서에서 가져온 것"으로 표시해 사용자가 바로잡게 한다. 이미 답이 나온 건 다시 묻지 않고, **여전히 못 채우는 값(보유기간·연락처·업체명 등)만** `[확인 필요]`로 남긴다.

**자료가 없으면**: `references/성격파악-질문가이드.md`의 질문을 **한 번에 하나씩, 짧게** 묻는다. (4~5개)

1. **서비스가 뭐고 누가 쓰나요?** (한 줄) — 서비스명·운영주체·대상
2. **회원가입이 있나요?** (있으면 개인정보 수집·회원 정책을 켠다)
3. **돈을 받나요?** (결제/구독/판매가 있으면 결제·환불·전자상거래 조항을 켠다)
4. **위치·사진·연락처 등 민감하거나 특별한 정보를 다루나요?** (위치기반 약관·개인정보 항목 조정)
5. **마케팅 알림(문자·메일·푸시)을 보내나요?** (마케팅 수신 동의를 켠다)

- "잘 모르겠다"면 **예시 2~3개**로 유도한다. 그래도 불명확하면 **가장 흔한 기본값으로 가정**하고 "가정"에 표시한다.
- 운영주체 이름·문의 연락처처럼 문서에 꼭 필요한데 답이 없으면 `[확인 필요: …]`로 비워 둔다. **지어내지 않는다.**
- 각 답이 **어떤 조항을 켜는지**는 `references/성격파악-질문가이드.md`의 매핑을 따른다.

## Step 2: (A) 서비스 정책서 만들기

`assets/정책서-템플릿.md`를 채운다. 이건 **우리가 정하는 운영 규칙**이라 초안 우선으로 채우고, 임의로 정한 건 "가정"에 표시한다.

- 재료: 기획서의 **규칙·금지·핵심 정보**, 또는 성격 파악 질문의 답.
- 항목은 **해당되는 것만** 넣는다(없는 항목은 아예 빼서 문서를 어지럽히지 않는다):
  회원 정책(가입·등급·탈퇴·정지), 콘텐츠·게시물 정책, 결제·환불·취소, 신고·제재, 노쇼/취소(예약·모임형), 위치정보 이용, 분쟁 처리 원칙 등.
- **넣는 항목은 실제 운영 규칙 문장으로 끝까지 쓴다. 요약본·한 줄 프롬프트로 두지 않는다.** 각 섹션은 그대로 게시·운영할 수 있을 만큼 구체적으로(누가·언제·어떻게·위반 시 어떻게) 작성한다. 약관만큼 자세할 필요는 없지만, "가입 자격이 있음" 같은 뼈대만 남기지 말고 "만 ○세 이상, ○○ 절차로 가입" 식 실제 규칙으로.
- 규칙 값(예: 환불 가능 기간, 정지 기간, 취소 마감 시점)을 사용자가 안 정했으면 합리적 기본을 문장 안에 `[확인 필요: 예시]`로 넣고 "가정"에도 표시한다. 조항(섹션) 자체는 생략하지 않는다. 법으로 정해진 소비자 권리(예: 청약철회)를 **불리하게** 축소하지 않는다.
- **참조 가능한 `POL.*` ID를 붙인다.** 넣는 각 규칙 제목 옆에 인라인 코드로 안정 ID를 단다(예: `**환불·취소 기준** \`POL.payment.refund\``, 섹션 제목엔 `POL.payment.*`). 이 ID는 정책 규칙의 원본(SSOT)으로, 시나리오·화면 설계서·유스케이스가 규칙을 **이름이 아니라 ID로** 참조하게 해 준다. 도메인·규칙명 표준과 규칙은 `references/정책-ID체계.md`를 따른다. **한 번 정한 ID는 문구가 바뀌어도 유지**하고, 넣지 않은 규칙에는 붙이지 않는다.
- **frontmatter + 규칙 인덱스 블록.** 정책서 맨 앞에 frontmatter(`doc_type: policy`·`version`·`ssot: prose`·`machine.{tag: policy.rules, schema, namespace: POL}`)를 두고, 문서 하단에 ` ```json policy.rules ` 블록으로 넣은 모든 규칙을 `{id, label, domain}`으로 담는다(규칙 문장이 원본, 이 블록은 ID 인덱스 미러). `schemas/policy.v1.schema.json`을 산출물 옆 `schemas/`에 복사한다. 이 블록으로 소프트웨어가 다른 문서의 `POL.*` 참조를 자동 조회한다.

## Step 3: (B) 이용약관 + 개인정보처리방침 — 완결 초안이 먼저

`assets/약관-개인정보-템플릿.md`를 채운다. 맨 앞에 frontmatter(frontmatter, `doc_type: terms-privacy`·`version`·`ssot: prose`)를 둔다(이 문서는 다른 문서가 ID로 참조하는 레지스트리가 아니므로 **기계 블록은 두지 않는다** — frontmatter만으로 식별). 산출 문서는 **초안이 맨 앞**에 오고 점검표는 뒤로 뺀다. **반드시 이 순서**:

> **1부. 완결 초안 → 2부. 확인이 필요한 항목 → 3부. 점검표.** 그리고 문서 맨 앞·맨 끝에 검토 경고.

### ① 1부 — 완결 초안 (가장 중요)
그대로 복사해 쓰고 빈 값만 채우면 되는 **완결 초안**을 만든다. **발췌·축약 금지.**
- 이용약관 전문 → 개인정보처리방침 전문 순으로, **모든 표준 조항을 실제 문장으로 빠짐없이** 쓴다. "…나머지 확장", "(발췌)", "이하 생략", "…" 같은 표현을 쓰지 않는다.
- 어떤 조항을 다 넣었는지는 `references/약관-표준조항.md`와 `references/개인정보처리방침-표준항목.md`의 목록을 **전부 돌며** 확인한다. 조항 누락·발췌를 이 목록으로 막는다.
- **"지어내지 않기"의 적용 범위:**
  · **표준 조항 문안**(목적·정의·계약 성립·서비스 제공/변경/중단·회원 의무·금지행위·게시물 권리·면책·손해배상·준거법·분쟁해결·약관 개정 등)은 **정형 문장이므로 끝까지 작성한다.**
  · **사업자·서비스 고유값**(회사명·대표·연락처·보유기간·보호책임자·수탁업체·수수료·관할 등)만 **문장 안에** `[확인 필요 #n: 예시]` 형태로 남긴다. **조항 자체를 생략하지 않는다.**
  · 예) "회원 개인정보 보유 기간은 [확인 필요 #3: 예—회원 탈퇴 시 즉시 파기]로 한다."
- **기획서·IA에서 도출 가능한 값**(수집 항목·처리 목적·켜질 조항·위탁의 존재)은 `[확인 필요]`로 두지 말고 채운다. (Step 1 참고)
- 해당 없는 상황별 조항(결제·위치·마케팅 등)은 초안에서 뺀다. 넣는 조항은 **완결**로.
- 개인정보처리방침은 법정 명칭 **"개인정보처리방침"** 을 그대로 쓴다.

### ② 2부 — 확인이 필요한 항목 (뒤로 뺌)
1부의 `[확인 필요 #n]`을 **번호별로 모아**, 각 항목을 무엇으로 채워야 하는지(예시 값 포함) 안내한다.
"앞 초안의 #n을 이 값으로 바꾸세요" 식으로 쓴다. **번호로 1부 본문과 2부를 연결**해, 사용자가 뒤를 보고 앞의 정확한 위치를 찾아 고치게 한다.

### ③ 3부 — 점검표 (맨 뒤)
이 서비스에 필요한 조항/법정 항목이 다 들어갔는지 확인용. 개인정보처리방침은 한국 개인정보보호법 법정 기재사항 기준(`references/개인정보-법정항목-체크리스트.md`).

### ④ 검토 경고 (앞·뒤 모두)
- **맨 앞**: 한 줄 요약("초안·점검용, 대한민국 법령 기준, 사용 전 전문가 검토 필요").
- **맨 끝(3부 뒤)**: 다음을 함께 명시한다.
  > "이 초안은 **대한민국 법령 기준**으로 작성되었습니다(개인정보보호법, 전자상거래법, 위치정보법 등). 다른 국가 서비스라면 해당 국가 기준으로 다시 검토해야 합니다."
  > "이 문서는 초안·점검용이며, 실제 사용 전 반드시 법률 전문가의 검토가 필요합니다."

## Step 4: 마무리 — 반드시 파일로

1. **읽기 쉬운 문서를 "파일"로** 만든다. 만들기로 한 것만:
   - 정책서 → `정책서-[서비스이름].md`
   - 약관·개인정보처리방침 → `약관-개인정보처리방침-[서비스이름].md` (1부 완결 초안 → 2부 확인 항목 → 3부 점검표, 한 파일)
   - 기본은 **마크다운(.md)**. 원하면 Word(.docx)로도.
   - (Claude Code/배치에서는 파일로 저장하고 경로를 알려주고, claude.ai에서는 문서로 만들어 보여준다.)
2. **분량 규칙**: 약관·개인정보처리방침의 기본은 **"완결"**이다. "간단히"라도 **조항을 빼지 않는다** — 각 조의 부연 설명만 짧게 한다. "자세히"는 부연·예시를 늘린다. **어느 분량에서도 표준 조항은 모두 존재해야 한다.** 정책서도 같은 원칙이다: 기본 "적당히"이되 **해당 섹션을 실제 규칙 문장으로 완성**하고, "간단히"라도 섹션을 빼거나 한 줄 요약으로 만들지 않는다(부연만 줄인다).
3. **끝나면 3~4줄로 요약**한다: 무엇을 만들었는지(파일 목록 포함), `[확인 필요]`로 남은 핵심 항목이 무엇인지, 그리고 **전문가 검토가 필요하다**는 점.

## Step 5: 재진입 — 부분 수정 (이미 만든 문서를 고칠 때)

이미 만든 정책서·약관을 **처음부터 다시 만들지 않고, 지정된 곳만** 고치는 경로다. 세트(suite)의 리뷰·컨펌 게이트가 라우팅해 부르거나, 사용자가 독립 실행으로 직접 부를 수 있다. **두 경우 모두 아래 규칙이 동일하게 적용된다.**

### 1. 진입 조건
사용자가 이전에 만든 정책서/약관(또는 그 형식의 문서)을 가져와 "이 조항/이 규칙만 바꿔줘", "○○ 부분 고쳐줘"라고 할 때 발동한다. **처음부터 다시 만들지 않는다** — 가져온 문서를 **기준선**으로 삼고 **지정된 곳만** 손댄다. (Step 0~4의 신규 생성 흐름으로 되돌아가지 않는다.)

### 2. 기준선 보존 원칙
지정되지 않은 조항·항목·문장은 **글자 그대로 유지**한다. 임의 재작성·재배열을 하지 않는다. (다른 세 스킬의 "잘 된 곳은 안 건드린다"와 동일.) 부분 수정을 빌미로 전체를 다시 쓰지 않는다.
- **`POL.*` ID 보존**: 정책서에 규칙별 `POL.*` ID가 있으면, 지정되지 않은 규칙의 ID는 그대로 둔다. 문구만 고칠 때 ID를 바꾸지 않는다(다른 문서가 이 ID로 링크하고 있을 수 있다). 규칙을 **삭제**하면 그 ID도 폐기하고, 규칙을 **추가**하면 `references/정책-ID체계.md` 도메인 안에서 새 ID를 짓는다.
- **`POL.*` ID 부여(버전업)**: 기존 정책서에 규칙 ID가 **아예 없으면**(옛 구조), `references/정책-ID체계.md`의 표준 도메인·규칙 매핑대로 **각 규칙 제목 옆에 ID를 새로 붙인다**. 이때 **규칙 문장은 건드리지 않고 ID만 추가**한다(gap-fill — 다른 내용은 그대로). 이 버전업은 세트(suite)가 "구조 동기화"로 부르거나 사용자가 "정책 ID 붙여줘"라고 할 때 수행한다.
- **frontmatter·규칙 인덱스·스키마 gap-fill(버전업)**: 옛 정책서에 frontmatter나 ` ```json policy.rules ` 블록이 없으면 표는 그대로 두고 기계층만 보강한다 — 맨 앞에 frontmatter를 얹고, 규칙들의 `POL.*`를 모아 인덱스 블록을 만들고, `schemas/policy.v1.schema.json`을 산출물 옆에 복사하고, "자동 점검할 수 있는 것" 설명 절이 없으면 채운다. 약관·개인정보 문서에는 frontmatter(`doc_type: terms-privacy`)만 없으면 얹는다. (규칙 문장·조항은 건드리지 않는다.)

### 3. `[확인 필요 #n]` 연결 유지 (가장 중요)
약관·개인정보처리방침은 `[확인 필요 #n]` 번호가 **1부 본문 ↔ 2부 안내를 양방향 연결**한다. 수정으로 이 번호가 어긋나면 안 된다.
- 수정으로 고유값 자리가 **늘거나 줄면** `#n`을 재정렬하되, **1부 본문과 2부 안내의 번호가 계속 일치하도록 양쪽을 함께 갱신**한다. 한쪽만 고쳐 번호가 어긋나면 안 된다.
- **조항을 삭제**하면 그 조항 안의 `#n`을 2부에서도 제거하고, **뒤 번호를 당겨 연속성을 맞춘다**(재정렬 방식으로 통일).
- **조항을 추가**하면 새 `#n`을 **끝번호로** 부여하고 2부에도 같은 번호로 추가한다.

### 4. 구조·안전장치 보존
- **1·2·3부 순서**(1부 완결 초안 → 2부 확인 항목 → 3부 점검표), **3부 점검표**, **문서 앞·뒤 검토 경고**는 부분 수정 후에도 그대로 있어야 한다. 사용자가 "경고 빼줘"라고 해도 기존 안전 원칙대로 **남긴다.**
- 소비자 권리(청약철회·환불)를 **불리하게 축소**하는 수정 요구는 그대로 반영하지 않는다. **무효 가능성을 한 줄로 알린 뒤, 법정 기준을 지키는 선에서** 반영한다.

### 5. 조항 켜기/끄기 재판정
수정이 기능 변화를 동반하면, 해당 상황별 조항을 `references/약관-표준조항.md` 기준으로 켜거나 끈다.
- 예) "결제 기능 뺐어" → 결제·청약철회 조항 제거 / "위치 추가됐어" → 위치기반 조항 추가.
- 조항을 켜고 끌 때 **그에 딸린 `#n`·2부 안내·3부 점검표도 함께 정리**한다(3번 규칙대로 번호 재정렬).

### 6. 부분 산출 (기존 부분 생성 규칙과 동일)
"정책서만 고쳐줘"면 정책서만, "약관만"이면 약관·개인정보처리방침만 손댄다. 콕 집으면 나머지 문서는 열지도 고치지도 않는다.

### 7. 마무리 통보
수정 후 **무엇을 어디서 고쳤는지 한두 줄로 요약**한다.
- 예) "제18조 청약철회 문구를 ○○로 바꾸고, 그에 맞춰 #7 안내를 갱신했습니다. 나머지 조항·검토 경고는 그대로입니다."

> **하지 말 것:** 부분 수정을 빌미로 전체를 다시 쓰기 / 지정 안 된 조항 건드리기 / `#n` 양방향 연결·검토 경고·1·2·3부 구조 깨뜨리기.

## 정합성 점검 — 3층 계약을 따른다 (누가·무엇을·언제)

> 여기서 말하는 "정합성 점검"은 **기계 층**(frontmatter·`policy.rules` 블록·크로스참조) 점검이다.
> 약관·개인정보의 **법정 항목 점검표(3부)**와는 다른 축이다 — 그건 법적 완비를 돕는 체크리스트이고, 여기는 문서의 기계 미러 정합이다.

이 문서의 점검은 세트의 **점검 3층 원칙**을 따른다. 정본은 `easyproduct-suite`의
`references/checker-guide.md`("점검 3층 원칙")이며, 여기서 재서술하지 않고 그 계약을 따른다. 갈림 축은 「의미판단 vs 결정론」이다.
정책서의 `ssot`는 **`prose`**(규칙 문장이 원본, `policy.rules` 블록은 ID 인덱스 미러)다. 약관·개인정보 문서는 기계 블록이 없는 **식별 전용**(frontmatter만)이라 기계 정합 대상이 아니다.

- **소프트웨어(점검기, 아무때나·세트 전체)** — ID 형식(`POL.도메인.규칙`) · 산문↔블록 매칭(coverage: 규칙 문장의 모든 `POL.*`가 블록에) · 블록 유효성(스키마+크로스참조 죽은 링크).
  세트 점검기 `check-docs.mjs`가 담당하고 **항상 세트 전체**로 돈다(시나리오·화면이 `POL.*`를 참조하므로 크로스참조는 전역). 이 스킬이 단독으로 재구현하지 않는다.
- **LLM(에이전트, 풀 리뷰 때만)** — 산문 충실도 · 미러 충실도(규칙의 뜻·도메인·라벨이 블록에 맞게 담겼나).
  풀 리뷰 시기: 최초 완성 / 문서 수정 후(부분 풀 리뷰=수정 항목+파장) / 사용자 요구 / 에이전트 판단.
- **이 스킬이 생성·수정 시 로컬로 보는 것**: 모든 규칙이 `POL.도메인.규칙` 형식을 갖고 `policy.rules` 블록에 빠짐없이 담겼는지, frontmatter·태그·스키마 사본이 있는지.
- **수정 방향**: 원본은 **규칙 문장(`ssot: prose`)**이다. 문장↔블록이 어긋나면 **문장 기준으로 블록을 다시 생성**한다. 점검기는 어긋남을 **보고**까지, 재생성은 이 스킬(또는 LLM)이 한다.

## 안전 원칙 (대화 내내 지킴)

- **법률 자문 아님.** 이 스킬은 변호사가 아니다. "이렇게 하면 법적으로 안전하다"고 단정하지 않는다. 준비를 돕고, 검토를 권한다.
- **검토 경고는 항상.** 약관·개인정보처리방침 문서의 앞·끝에 검토 경고를 넣는다. 사용자가 빼달라고 해도 성격을 설명하고 남긴다.
- **지어내지 않기(범위 명확).** **표준 조항 문안은 정형이므로 끝까지 작성**한다(발췌·생략 금지). 반면 **사업자·서비스 고유값**(보유기간·제3자 제공 대상·위탁 업체·보호책임자·회사정보·수수료·관할 등)은 지어내지 말고 문장 안에 `[확인 필요 #n]`으로 남긴다. 즉 "조항은 완성, 고유값만 비움".
- **소비자 권리 축소 금지.** 청약철회·환불처럼 법으로 보장된 권리를 초안에서 불리하게 줄이지 않는다. (그런 조항은 무효가 될 수 있음을 한 줄로 알린다.)

## 비전문가 응대 규칙

- 질문은 자료가 없을 때만, **4~5개**. 그 뒤로는 묻지 말고 초안으로 답한다.
- "잘 모르겠어요"라고 하면 **예시 2~3개**로 고르게 한다.
- 법률 용어가 나오면 **즉시 한 줄로 풀어** 쓴다.
- "이건 안 됩니다"보다 "이렇게 잡아뒀는데 맞는지 전문가와 확인하세요"로.

- `assets/정책서-템플릿.md` — 서비스 정책서(운영 규칙)의 읽기 쉬운 구조
- `assets/약관-개인정보-템플릿.md` — 이용약관 + 개인정보처리방침. **1부 완결 초안 → 2부 확인 항목 → 3부 점검표** 구조 (완결 초안 예시 포함)

참고 자산:
- `references/약관-표준조항.md` — 이용약관이 반드시 포함할 표준 조항 목록 (초안 작성 시 전부 돌아 누락·발췌 방지 / **Step 5 조항 켜기·끄기 재판정 기준**)
- `references/개인정보처리방침-표준항목.md` — 개인정보처리방침이 반드시 포함할 표준 항목 + 문안 뼈대
- `references/개인정보-법정항목-체크리스트.md` — 법정 기재사항 + 해당 여부 판단 + "문서에서 채울 수 있나" (3부 점검표·도출 판단용)
- `references/성격파악-질문가이드.md` — 독립 실행 시 질문 → 어떤 조항·문서가 켜지는지 매핑, "모르겠어요" 대응
- `references/정책-ID체계.md` — 정책 규칙의 참조 가능한 안정 ID(`POL.도메인.규칙명`) 체계·표준 도메인 매핑 (시나리오·화면 설계서 등이 규칙을 ID로 참조하기 위함)
- `schemas/policy.v1.schema.json` — 정책서 `policy.rules` 블록의 JSON Schema 계약(스킬 소유 고정 자산, 산출물 옆 `schemas/`로 복사).
