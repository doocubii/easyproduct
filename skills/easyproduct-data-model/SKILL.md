# 데이터 모델 만들기

## 목적

기술 지식이 전혀 없는 사람도, AI와의 대화만으로 서비스가 다룰 **데이터 모델(데이터 사전)**을
갖추게 돕는다. 이 스킬이 하는 일은 **넷이다.**

1. **생성** — 정보 덩어리(회원·요청·거래 등)에서 기본 데이터 모델을 추출해 **문서로** 만든다.
2. **수정** — 그 문서를 계속 고친다(필드 추가·삭제·이름변경·설명수정, 그룹 변경).
3. **조회** — 데이터 사전에서 특정 데이터·필드를 찾아 답한다.
4. **json 추출(읽기 전용 스냅샷)** — 요청하면 그 시점의 데이터 모델을 json으로 뽑아 별도 파일로 내려준다.

산출물은 **하나의 데이터 모델 문서(md)**이며, 이 스킬은 오직 이 문서를 관리하는 일만 한다.
누가 이 스킬을 어떻게 가져다 쓰는지는 이 스킬의 관심사가 아니다 — 연계는 쓰는 쪽이 정한다.
다만 아래 **"요청 처리 규칙"**과 **"결과 회신"**은 특정 스킬과의 계약이 아니라, 이 스킬이 정확히
동작하기 위한 **입력 요건·출력 형식**이므로 반드시 지킨다.

## 핵심 원칙

1. **비전문가 언어.** 전문 용어(스키마·정규화·외래키 등)를 먼저 꺼내지 않는다. 종류는 "글자/숫자/날짜/참·거짓/목록/이미지"처럼 쉬운 말로.
2. **초안 우선.** 매번 묻고 멈추지 않는다. 정보 덩어리만 받으면 표준 필드를 자동으로 깔아 **전체 초안을 한 번에** 만든 뒤, 사람이 보고 고치게 한다. 자동으로 넣은 건 표시해 빼기 쉽게 한다.
3. **항상 파일로.** 결과는 채팅 텍스트가 아니라 **하나의 데이터 모델 문서(md)**로 낸다.
4. **문서 하나가 원본(SSOT)이다.** 데이터의 원본은 이 md 문서 그 자체다. 문서 안의 그룹별 데이터 사전 표가 곧 원본이다. "어딘가에서 렌더된 파생본"이라는 개념은 없다.
5. **한 개념 = 한 변수 = 한 정의.** 같은 개념을 두 곳에 만들지 않는다. 설명은 데이터 모델 한 곳에만 둔다.
6. **원문 보존.** 받은 정보 덩어리(그룹명·한글명·설명)를 임의로 바꾸지 않는다.
7. **애매하면 지어내지 말고 확인.** 소속 덩어리가 모호하거나 충돌이 의심되면 되묻는다.

## 산출물 형식 — 데이터 모델 문서 하나 (반드시 이 구조)

데이터 모델은 **하나의 문서**로 낸다. 런타임 산출물 파일명은 서비스 이름을 넣는다.

- **`데이터모델-[서비스].md`** — 데이터 모델 문서. 이 문서 안의 **그룹별 데이터 사전 표 자체가
  원본(SSOT)**이다. 스킬이 생성·수정·판정하는 실제 데이터가 여기에 있다.
  구조는 `assets/data-model-template.md`를 따른다.

**문서 맨 앞에 아래 원칙을 반드시 못박는다** (템플릿에 자리가 있다). 문서는 자신이 어떤 상황에서
열리는지 알 수 없으므로, **제작 정황이나 조건부 안내 없이, 상황 불문하고 적용되는 원칙만** 적는다:

> 이 데이터 모델은 데이터(정보 항목·이름·구조)에 관해 기획서를 포함한 다른 어떤 문서보다
> 우선하는 기준(SSOT)이다. 데이터를 바꿀 때는 이 문서에서 바꾼다.
>
> 이 표를 고칠 때는 아래를 지킨다:
> - 이름은 `그룹.필드` 형식을 유지한다(그룹=영문 소문자 단수, 필드=camelCase).
> - 새 필드를 추가하기 전에 같은 뜻의 필드가 이미 있는지 표를 먼저 훑는다(중복 방지).
> - 필드를 삭제하거나 이름을 바꾸면, 그 변수를 쓰는 화면·코드 전체를 함께 갱신한다.
> - 새 그룹(덩어리) 추가는 정보 구조를 바꾸는 일이니 신중히 판단한다. 되도록 기존 그룹 안에서 해결한다.
> - 애매하면 임의로 만들지 말고 기존 규칙(공통 관리 필드·표준 프리셋)을 참고해 일관되게 정한다.

## 시작 입력 — 세 갈래 진입

시작할 때 **받은 것을 보고** 갈래를 정한다. 아래에서 "기획서"라는 말은
"기획서 또는 사용자가 준 정보 덩어리"로 읽는다(기획서가 없어도 단독으로 동작한다).

- **(가) 데이터 모델 파일(md)을 받으면** → 그것을 **기준선(SSOT)**으로 삼아 조회·수정·추가를
  이어간다. **새로 생성하지 않는다.** 기획서(또는 정보 덩어리)도 함께 받았으면 데이터 모델이
  우선이고 기획서는 참고다 — **데이터 모델을 기획서 기준으로 덮어쓰지 않는다.** 데이터 모델에 없는
  덩어리가 기획서에 있으면 "추가할까요?"로 확인한다. (재진입/이어작업의 기본 진입)
- **(나) 정보 덩어리만 받으면**(기획서 파일, 또는 말로 "회원·요청·거래가 있어" 정도) → **[① 생성]**으로
  새 데이터 모델을 만든다.
- **(다) 아무것도 없으면** → 성격을 물어 정보 덩어리부터 받고 **[① 생성]**으로 만든다.

## 명명·표기 규칙

- **변수명은 이 스킬이 짓는다.** 사용자가 영어를 직접 짓게 하지 않는다.
  꼴은 **`그룹.필드`** — 그룹 = 영문 소문자 단수, 필드 = camelCase. (예: `user.nickname`)
- **한 개념 = 한 변수명(전역 유일).** 같은 뜻이 두 변수명으로 갈라지지 않게 한다.
- **한글명은 자연어로.** `[그룹 한글]의 [필드 뜻]` 꼴. 예: `user.nickname` → "사용자의 닉네임".
- **종류는 비전문가 말로**: 글자 / 숫자 / 날짜 / 참·거짓 / 목록 / 이미지.
- **한 변수 = 한 정의.** 설명(정의)은 데이터 모델 한 곳에만 둔다.

---

## ① 생성

1. 받은 정보 덩어리에서 **그룹명·한글명·설명**을 읽는다. **원문은 보존**한다.
2. 각 덩어리에 필드를 **자동으로 깐다** (`references/standard-fields-preset.md` 참고):
   - **공통 관리 필드**(`id`·`createdAt`·`updatedAt`, 상태가 있으면 `status`, 소프트삭제면 `deletedAt`)를 **모든 덩어리에**. 사람에겐 개별 나열 대신 "관리 필드 자동 포함"으로 접어 보여도 된다.
   - **종류별 표준 필드 프리셋**: 범위는 회원·거래계 핵심 5~7종(`user`, `payment`, `order`/`deal`, `review`, `report`, `notification` 정도). 프리셋에 없는 덩어리는 공통 관리 필드만 깔고 나머지는 비워 둔다.
3. 각 필드는 **변수명·한글명(자연어)·종류·필수·설명·쓰이는 화면(선택)**을 갖는다. 자동으로 깐 건
   **"자동(관리)" / "자동(프리셋)"**으로 표시해 빼기 쉽게 한다.
4. **데이터 모델 문서(md)를 만든다.** 초안을 한 번에 다 만든 뒤 사람이 문서로 보고,
   고칠 것은 스킬에게 말해 반영하게 한다. (매 항목 컨펌받지 않는다.)

---

## ② 수정

- 대상은 **시작 입력 (가)로 받았거나 이번에 생성한** 데이터 모델이다. 데이터 모델 문서를 기준선으로,
  **지정된 곳만** 고친다. 지정 안 된 필드·설명은 그대로 둔다.
- **필드 추가/삭제/이름변경/설명수정** 가능. 변수명을 바꾸면 그 변수를 참조하는 곳(있다면 "쓰이는 화면"
  목록)에도 반영해야 함을 알린다.
- **새 그룹(덩어리)의 생성·삭제·병합**은 정보 구조를 바꾸는 일이므로 **사용자 컨펌**을 거친다.
- **무엇을 어디서 고쳤는지 끝에 한두 줄로 통보**한다.

---

## ③ 조회

- 데이터 모델에서 특정 데이터를 찾아 답한다.
  예: "이 서비스에 어떤 데이터가 있어?", "`request`에 어떤 필드가 있어?", "금액 관련 변수 뭐야?", "닉네임은 어느 그룹의 뭐야?".
- 이 조회 기능은 주로 **"사람이 직접 물어볼 때"**를 위한 것이다(사람은 파일을 잘 안 읽으므로).
- 조회는 값을 읽어 답할 뿐, 데이터 모델 문서를 바꾸지 않는다.

## ④ json 추출 (읽기 전용 스냅샷)

사람(또는 다른 도구)이 요청하면, **그 시점의 데이터 모델 문서(md)를 읽어 그 스냅샷을 json으로 뽑아
별도 파일로 내려준다.** 트리거 예: "json으로 뽑아줘", "이 데이터 모델 json으로 줘", "기계가 읽기 좋게 변환해줘".

- **이 json은 "그때 뽑은 사본"일 뿐이며 원본이 아니다.** 원본은 언제나 데이터 모델 문서(md)다.
  **json을 고쳐도 md(원본)에 반영되지 않는다.** 이 사실을 산출물 안내에 반드시 명시한다
  (다시 이중 관리 문제가 생기지 않도록).
- **필드 스키마는 결과 회신 구조체와 동일**하게 각 필드가
  `variable`·`group`·`groupKo`·`nameKo`·`type`·`required`·`description`을 갖는다.
  **그룹별로 묶은 오브젝트 구조**로 뽑는다:

```json
{
  "service": "[서비스 이름]",
  "extractedFrom": "데이터모델-[서비스].md",
  "note": "이 파일은 위 md 원본에서 뽑은 읽기 전용 사본이다. 이 json을 고쳐도 원본(md)에 반영되지 않는다.",
  "groups": {
    "user": {
      "groupKo": "회원",
      "fields": {
        "email": {
          "variable": "user.email",
          "group": "user",
          "groupKo": "회원",
          "nameKo": "회원의 이메일",
          "type": "글자",
          "required": true,
          "description": "로그인·연락에 쓰는 이메일 주소"
        }
      }
    }
  }
}
```

- 런타임 산출물 파일명 예: `데이터모델-[서비스]-추출.json` (원본과 헷갈리지 않게 "추출/스냅샷"을 드러낸다).
- **이 추출은 요청받을 때만** 한다. 생성·수정 때 json을 상시 파일로 함께 만들지 않는다.

## 파일이 우선, 스킬 호출은 예외 (반드시 지킨다)

데이터 모델은 **문서(md=SSOT)로 존재**한다. 그래서 이미 있는 데이터를 **"읽기만"** 할 때는
데이터 모델 문서(md)를 직접 읽으면 되고 **스킬을 부를 필요가 없다.** 스킬을 부르는 경우는
문서만으로 안 되는 상황으로 한정한다:

- 필요한 데이터가 없어 **새로 등록**해야 할 때
- 소속 덩어리가 애매하거나, 기존 것과 **충돌·중복이 의심**될 때
- **새 덩어리가 필요**해 보일 때(컨펌 필요)
- 정보 덩어리 자체에 **문제가 있을 때**(원문과 안 맞음 등)

즉 **읽기(read)는 데이터 모델 문서(md) 직접, 쓰기·판정(write/resolve)은 스킬.**

## 데이터 등록·판정 요청을 받을 때의 처리 규칙 (중복·충돌 안전장치)

누가(사람이든 다른 도구든) 데이터 등록·판정을 요청할 때 아래 절차를 따른다.

**1. 요청 형식 검증.** 요청은 최소한 **(ㄱ)소속 덩어리 후보, (ㄴ)뜻(한 줄), (ㄷ)종류**를 갖춰야 한다.
   - 예: "`request`(요청)에 '지원자 수' 필요, 뜻='이 요청에 지원한 사람 수', 종류=숫자."
   - "지원자 수 필요해"처럼 덩어리·뜻·종류가 없거나 모호하면 **바로 등록하지 말고 되묻는다**: "어느 묶음의 데이터인가요? 무슨 뜻인가요?" (같은 뜻의 데이터가 여러 덩어리에 있을 수 있어, 덩어리를 특정하지 않으면 엉뚱한 곳에 넣거나 중복을 못 거른다.)

**2. 조회 먼저 → 기존 흡수.** 정보가 갖춰지면 **지정된 덩어리 안에서 먼저 데이터 모델 문서를 조회**한다.
   같은 개념이 이미 있으면 새로 만들지 않고 **기존 변수명으로 흡수**한다. 없을 때만 신규 등록한다.
   (뜻·종류로 "이름은 같은데 다른 개념 / 이름은 다른데 같은 개념"을 판정)

**3. 덩어리 후보는 확정이 아니다.** 요청자가 지정한 덩어리가 부적절해 보이면 더 맞는 덩어리를
   제안한다(예: "그건 `request`보다 `deal`이 맞아 보입니다"). **최종 소속·이름은 이 스킬이 정한다.**

**4. 안 맞으면 새 덩어리 제안·확인.** 등록하려는 데이터가 기존 어느 그룹에도 안 맞으면, 즉석에서
   새 그룹을 만들지 말고 **"새 묶음이 필요해 보인다"고 사용자에게 제안·확인**한다. 승인되면 그룹을
   만들고, 기획서 쪽에도 반영이 필요함을 알린다.

**5. 결과 회신(구조체).** 조회·등록·흡수·정정·확인대기 **어느 경우든**, 처리 결과를 아래 구조체로
   명확히 돌려준다. 요청자는 이 `variable`을 그대로 참조한다. 이 구조체는 **데이터 모델 문서의 표
   한 행이 갖는 항목과 동일한 구성**으로 하여, "문서에서 읽은 것"과 "회신으로 받은 것"이 구분 없이
   똑같이 쓰이게 한다.

```json
{
  "variable":    "request.applicantCount",
  "group":       "request",
  "groupKo":     "요청",
  "nameKo":      "요청의 지원자 수",
  "type":        "숫자",
  "required":    false,
  "description": "이 요청에 지원한 사람 수",
  "resolution":  "created"
}
```

`resolution` 뜻:
- `created` — 신규 등록
- `reused` — 이미 있어 그걸 참조
- `corrected` — 비슷한 걸로 통일
- `needs_confirm` — 새 덩어리 필요 등 **사용자 확인 대기**

등록·정정이 일어나면 **데이터 모델 문서를 갱신**한다.

## 정합성 점검 (생성·수정 시 조용히)

- **중복 의심**: 종류·한글명·설명이 비슷한데 변수명이 다른 쌍 → "같은 것 아닌가요?" 확인.
- **명명 규칙 위반**: `그룹.필드` 꼴 아님, 그룹 접두사 없는 전역 변수.
- **애매하면 지어내지 말고** 사용자에게 확인.

---

## 비전문가 응대 규칙 (대화 내내)

- 한 답변에 질문은 **하나, 많아도 둘**. 폭탄 금지.
- "잘 모르겠어요"라고 하면 **구체적 예시 2~3개**를 제시해 고르게 한다.
- 전문 용어가 나오면 **즉시 한 줄로 풀어** 쓴다. (예: "필수 = 이 칸을 비워두면 안 되는 값")
- 영어 변수명은 스킬이 지어 주고, 사용자에게는 **한글명으로** 확인받는다.
- "이건 안 됩니다"보다 "이렇게 잡아뒀는데 필요 없으면 빼세요"로.

## 하지 말 것

- 받은 정보 덩어리 **원문(그룹명·한글명·설명)을 임의로 바꾸지 말 것**(생성 시 초안 보존).
- 시작 입력 (가)에서 받은 데이터 모델을 **기획서 기준으로 덮어쓰지 말 것**(데이터 모델이 SSOT).
- 데이터 모델 문서(md) 안에 **"이 스킬로 만들었다" 같은 제작 정황이나, "스킬이 있으면 이렇게" 식
  조건부 안내를 넣지 말 것.** 문서는 상황을 모르므로 원칙만 상황 불문하고 적는다.
- 사용자가 **영어 변수명을 직접 짓게 하지 말 것**(스킬이 제안).
- 실제로 안 쓸 게 뻔한 필드를 **억지로 깔지 말 것**(지어내기 금지, 애매하면 확인).
- **새 그룹을 컨펌 없이 만들지 말 것.**
- **json 추출을 상시 파일로 착각하지 말 것.** [④ json 추출]은 요청받을 때만 뽑는 읽기 전용 사본이다. 생성·수정 때 json을 함께 만드는 이중 관리로 되돌리지 말 것.
- **결과 회신 구조체의 필드 구성 자체는 바꾸지 말 것.**
- **특정 스킬을 전제한 연계 로직을 넣지 말 것**(요청 처리 규칙·결과 회신은 입력 요건·출력 형식일 뿐, 스킬 결합이 아니다).
