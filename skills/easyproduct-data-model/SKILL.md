# 데이터 모델 만들기

## 목적

기술 지식이 전혀 없는 사람도, AI와의 대화만으로 서비스가 다룰 **데이터 모델(데이터 사전)**을
갖추게 돕는다. 이 스킬이 하는 일은 **넷이다.**

1. **생성** — 정보 덩어리(회원·요청·거래 등)에서 기본 데이터 모델을 추출해 **문서로** 만든다. 이때 각 그룹 표 아래에 **JSON 미러**도 함께 넣는다.
2. **수정** — 그 문서를 계속 고친다(필드 추가·삭제·이름변경·설명수정, 그룹 변경). 표를 고치면 그 그룹의 **JSON 미러도 다시 생성**한다.
3. **조회** — 데이터 사전에서 특정 데이터·필드를 찾아 답한다.
4. **json 묶음 추출** — 요청하면 문서 안의 그룹별 JSON 미러들을 **서비스 단위 한 파일로 묶어** 내려준다.

산출물은 **하나의 데이터 모델 문서(md)**이며, 이 스킬은 오직 이 문서를 관리하는 일만 한다.
누가 이 스킬을 어떻게 가져다 쓰는지는 이 스킬의 관심사가 아니다 — 연계는 쓰는 쪽이 정한다.
다만 아래 **"요청 처리 규칙"**과 **"결과 회신"**은 특정 스킬과의 계약이 아니라, 이 스킬이 정확히
동작하기 위한 **입력 요건·출력 형식**이므로 반드시 지킨다.

## 핵심 원칙

1. **비전문가 언어.** 전문 용어(스키마·정규화·외래키 등)를 먼저 꺼내지 않는다. 종류는 "글자/숫자/날짜/참·거짓/목록/이미지"처럼 쉬운 말로.
2. **초안 우선.** 매번 묻고 멈추지 않는다. 정보 덩어리만 받으면 표준 필드를 자동으로 깔아 **전체 초안을 한 번에** 만든 뒤, 사람이 보고 고치게 한다. 자동으로 넣은 건 표시해 빼기 쉽게 한다.
3. **항상 파일로.** 결과는 채팅 텍스트가 아니라 **하나의 데이터 모델 문서(md)**로 낸다.
4. **표가 원본(SSOT), JSON은 미러.** 데이터의 원본은 md 문서 안의 **그룹별 데이터 사전 표**다. 각 그룹 표 바로 아래의 **JSON 블록은 그 표를 기계가 읽도록 옮긴 미러**로, 항상 표에서 생성한다. **JSON을 손으로 고치지 않고**, 표가 바뀌면 JSON을 다시 만들어 맞춘다(한 파일 안에 진실이 둘이 되지 않게).
5. **한 개념 = 한 변수 = 한 정의.** 같은 개념을 두 곳에 만들지 않는다. 설명은 데이터 모델 한 곳에만 둔다.
6. **원문 보존.** 받은 정보 덩어리(그룹명·한글명·설명)를 임의로 바꾸지 않는다.
7. **애매하면 지어내지 말고 확인.** 소속 덩어리가 모호하거나 충돌이 의심되면 되묻는다.

## 산출물 형식 — 데이터 모델 문서 하나 (반드시 이 구조)

데이터 모델은 **하나의 문서**로 낸다. 런타임 산출물 파일명은 서비스 이름을 넣는다.

- **`데이터모델-[서비스].md`** — 데이터 모델 문서. 이 문서 안의 **그룹별 데이터 사전 표 자체가
  원본(SSOT)**이다. 스킬이 생성·수정·판정하는 실제 데이터가 여기에 있다.
  구조는 `assets/data-model-template.md`를 따른다.
- **`schemas/data-model.v1.schema.json`** — 그룹 JSON 미러의 검증 계약(JSON Schema). 스킬이 들고 있는
  **고정 자산을 산출물 옆 `schemas/`에 그대로 복사**한다(매번 새로 짓지 않는다). 버전을 파일명에 박으므로
  v1으로 쓴 문서는 언제나 v1으로 검증된다. 문서 frontmatter의 `machine.schema`가 이 파일을 가리킨다.

**문서 맨 앞에 frontmatter를 둔다.** 점검 소프트웨어가 본문을 파싱하지 않고도 "이게 어떤 문서이고,
기계 블록을 어디서·어떤 계약으로 찾는지"를 값싸게 알게 하는 규약이다(`assets/data-model-template.md`의
frontmatter를 그대로 채운다): `doc_type: data-model`, `version: 1`, `ssot: table`,
`machine.tag: datamodel.group`, `machine.schema`, `machine.namespace: DATA`.

**이어서 아래 원칙을 반드시 못박는다** (템플릿에 자리가 있다). 문서는 자신이 어떤 상황에서
열리는지 알 수 없으므로, **제작 정황이나 조건부 안내 없이, 상황 불문하고 적용되는 원칙만** 적는다:

> 이 데이터 모델은 데이터(정보 항목·이름·구조)에 관해 기획서를 포함한 다른 어떤 문서보다
> 우선하는 기준(SSOT)이다. 데이터를 바꿀 때는 이 문서에서 바꾼다.
>
> 이 표를 고칠 때는 아래를 지킨다:
> - 이름은 `그룹.필드` 형식을 유지한다(그룹=영문 소문자 단수, 필드=camelCase).
> - 새 필드를 추가하기 전에 같은 뜻의 필드가 이미 있는지 표를 먼저 훑는다(중복 방지).
> - 필드를 삭제하거나 이름을 바꾸면, 그 변수를 쓰는 화면·코드 전체를 함께 갱신한다.
> - 새 그룹(덩어리) 추가는 정보 구조를 바꾸는 일이니 신중히 판단한다. 되도록 기존 그룹 안에서 해결한다.
> - 애매하면 임의로 만들지 말고 기존 규칙(공통 관리 필드·표준 프리셋)을 참고해 일관되게 정한다.

## JSON 미러 — 각 그룹 표 바로 아래 (기계가 읽는 사본)

각 그룹 표 **바로 아래**에 그 그룹 구조를 담은 코드블록을 넣는다. 코드블록은 반드시 ` ```json datamodel.group `으로
**태깅**한다 — 점검 소프트웨어가 "이게 공식 기계 표현"임을 추측이 아니라 규약으로 찾게 하기 위함이다.
**표가 원본, JSON은 미러**다 — 표에서 생성하고, 표가 바뀌면 다시 생성한다. **JSON을 손으로 고치지 않는다.**
소프트웨어가 md를 파싱하지 않고도 이 블록만 읽어 구조를 얻게 하기 위함이다.

**그룹 오브젝트 스키마** (표 한 그룹 = JSON 하나). 검증 계약은 `schemas/data-model.v1.schema.json`:

```json datamodel.group
{
  "id": "DATA.<group>",
  "group": "<영문 소문자 단수>",
  "label": "<그룹 한글명>",
  "description": "<그룹 설명>",
  "fields": [
    {
      "name": "<필드명 camelCase>",
      "label": "<한글명>",
      "type": "string | number | datetime | date | boolean | list | image",
      "required": true,
      "filledBy": "system | user",
      "description": "<설명>"
    }
  ],
  "relations": [
    { "type": "hasMany | belongsTo | hasOne | manyToMany", "target": "<그룹>", "description": "<관계 문장>" }
  ]
}
```

**표 열 → JSON 키 매핑** (변환은 기계적이다):
- `변수명` `그룹.필드` → 그룹은 상단 `group`, 필드는 짧은 `name`(전체 변수 = `group.name`). 필드 안에 전체 변수를 다시 넣지 않는다.
- `종류`(한글) → `type`(영문): 글자→`string`, 숫자→`number`, 날짜→`datetime`(시각 포함)·`date`(날짜만), 참·거짓→`boolean`, 목록→`list`, 이미지→`image`.
- `필수` → `required`: 필수→`true`, 선택→`false`.
- `누가 넣나` → `filledBy`: 시스템→`system`, 사용자→`user`.
- 그룹 끝의 **관계** 문장 → `relations` 배열.

> 사람에게 보이는 표의 `종류`는 여전히 한글(글자·숫자·날짜…)이고, JSON `type`만 영문이다 — 표는 사람용, JSON은 기계용.

**데이터 모델이 담지 않는 것 — "어느 화면에서 쓰이나".** 그 관계의 원본(SSOT)은 화면 설계서의
`screendesign.screens`의 `data`(`display`·`io.sends/receives`·`bindings[].vars`)다. 데이터 모델에
같은 관계를 역방향으로 또 적으면 **이중 기입**이 되어, 화면이 바뀔 때마다 데이터가 안 바뀌었는데도
데이터 모델이 수정되고(가짜 변경 신호) 양쪽이 조용히 어긋난다. **필요하면 화면 설계서에서 유도해
답하고, 데이터 모델에 저장하지 않는다.**

## 시작 입력 — 세 갈래 진입

시작할 때 **받은 것을 보고** 갈래를 정한다. 아래에서 "기획서"라는 말은
"기획서 또는 사용자가 준 정보 덩어리"로 읽는다(기획서가 없어도 단독으로 동작한다).

- **(가) 데이터 모델 파일(md)을 받으면** → 그것을 **기준선(SSOT)**으로 삼아 조회·수정·추가를
  이어간다. **새로 생성하지 않는다.** 기획서(또는 정보 덩어리)도 함께 받았으면 데이터 모델이
  우선이고 기획서는 참고다 — **데이터 모델을 기획서 기준으로 덮어쓰지 않는다.** 데이터 모델에 없는
  덩어리가 기획서에 있으면 "추가할까요?"로 확인한다. (재진입/이어작업의 기본 진입)
- **(나) 정보 덩어리만 받으면**(기획서 파일, 또는 말로 "회원·요청·거래가 있어" 정도) → **[① 생성]**으로
  새 데이터 모델을 만든다.
- **(다) 아무것도 없으면** → 성격을 물어 정보 덩어리부터 받고 **[① 생성]**으로 만든다.

## 명명·표기 규칙

- **변수명은 이 스킬이 짓는다.** 사용자가 영어를 직접 짓게 하지 않는다.
  꼴은 **`그룹.필드`** — 그룹 = 영문 소문자 단수, 필드 = camelCase. (예: `user.nickname`)
- **그룹 anchor(크로스도큐먼트 ID)는 `DATA.<group>`.** 사람용 표의 변수명은 `그룹.필드`(접두사 없음) 그대로 두고,
  **JSON 미러의 `id`에만** `DATA.<group>`을 둔다(표=사람용, JSON=기계용). 다른 스킬 문서는 이 그룹을 `DATA.user`처럼,
  필드는 `user.email`처럼 참조한다 — 접두사 `DATA.`가 곧 "데이터 등기부로 조회하라"는 타입 표시다(cf. IA는 `FEAT.`).
- **한 개념 = 한 변수명(전역 유일).** 같은 뜻이 두 변수명으로 갈라지지 않게 한다.
- **한글명은 자연어로.** `[그룹 한글]의 [필드 뜻]` 꼴. 예: `user.nickname` → "사용자의 닉네임".
- **종류는 비전문가 말로**: 글자 / 숫자 / 날짜 / 참·거짓 / 목록 / 이미지.
- **한 변수 = 한 정의.** 설명(정의)은 데이터 모델 한 곳에만 둔다.

## ① 생성

1. 받은 정보 덩어리에서 **그룹명·한글명·설명**을 읽는다. **원문은 보존**한다.
2. 각 덩어리에 필드를 **자동으로 깐다** (`references/standard-fields-preset.md` 참고):
   - **공통 관리 필드**(`id`·`createdAt`·`updatedAt`, 상태가 있으면 `status`, 소프트삭제면 `deletedAt`)를 **모든 덩어리에**. 사람에겐 개별 나열 대신 "관리 필드 자동 포함"으로 접어 보여도 된다.
   - **종류별 표준 필드 프리셋**: 범위는 회원·거래계 핵심 5~7종(`user`, `payment`, `order`/`deal`, `review`, `report`, `notification` 정도). 프리셋에 없는 덩어리는 공통 관리 필드만 깔고 나머지는 비워 둔다.
3. 각 필드는 **변수명·한글명(자연어)·종류·필수·설명·누가 넣나**를 갖는다. `누가 넣나`는 **값을 채우는
   주체**만 뜻한다 — 시스템이 자동으로 만들어 관리하면 **시스템**, 사람이 입력·선택하면 **사용자**.
   (스킬이 자동으로 깔았는지 사람이 요청했는지는 이 열과 무관하다. 자동으로 깐 항목은 문서 본문이 아니라
   **회신 메시지에서** "이런 표준 항목을 자동으로 깔았습니다"로 알려 빼기 쉽게 한다.)
4. **데이터 모델 문서(md)를 만든다.** 맨 앞에 **frontmatter**를 채우고("산출물 형식" 참고), 각 그룹 표를 채운 뒤
   **그 표 바로 아래에 태그된 JSON 미러 블록**(` ```json datamodel.group `, 맨 위 `"id": "DATA.<group>"`)을 표에서 생성해 넣는다("JSON 미러" 스키마·매핑 참고).
   **`schemas/data-model.v1.schema.json`을 산출물 옆 `schemas/`에 복사**한다(스킬 자산 그대로).
   템플릿의 **"자동 점검할 수 있는 것" 설명 절도 그대로 채운다**(신규 생성 시에도 빠뜨리지 않는다 — 재진입 gap-fill만의 항목이 아니다).
   초안을 한 번에 다 만든 뒤 사람이 문서로 보고, 고칠 것은 스킬에게 말해 반영하게 한다. (매 항목 컨펌받지 않는다.)

---

## ② 수정

- 대상은 **시작 입력 (가)로 받았거나 이번에 생성한** 데이터 모델이다. 데이터 모델 문서를 기준선으로,
  **지정된 곳만** 고친다. 지정 안 된 필드·설명은 그대로 둔다.
- **필드 추가/삭제/이름변경/설명수정** 가능. 변수명을 바꾸면 **다른 문서가 그 변수를 참조하고 있을 수
  있음**(화면 설계서의 `data`, 시나리오 등)을 알리고, 세트 점검기가 죽은 링크를 잡아 준다고 안내한다.
- **새 그룹(덩어리)의 생성·삭제·병합**은 정보 구조를 바꾸는 일이므로 **사용자 컨펌**을 거친다.
- **표를 고치면 그 그룹의 JSON 미러를 다시 생성**해 표와 일치시킨다(JSON을 손으로 부분수정하지 않는다). 지정되지 않은 그룹의 표·JSON은 그대로 둔다.
- **무엇을 어디서 고쳤는지 끝에 한두 줄로 통보**한다.

### 버전업 — frontmatter·태그·JSON 미러가 없는 옛 문서에 채워 넣기

frontmatter나 인라인 JSON이 **없는** 예전 구조의 데이터 모델 md를 받으면(또는 세트가 "구조 동기화"로 부르면),
**표는 그대로 두고** 기계용 요소만 채운다(gap-fill). 표 내용을 바꾸거나 재작성하지 않는다:

- **frontmatter가 없으면** 맨 앞에 frontmatter(`doc_type`/`version`/`ssot`/`machine.*`)를 얹고, `schemas/data-model.v1.schema.json`을 산출물 옆에 복사한다.
- **JSON 미러가 없거나 태그가 없으면** 각 그룹 표 아래에 ` ```json datamodel.group ` 태그 블록을 표에서 생성해 넣고(맨 위 `"id": "DATA.<group>"`), 기존 블록은 태그·`id`를 보강한다.
- **"이 JSON 블록으로 소프트웨어가 자동 점검할 수 있는 것" 설명 절이 없으면** 템플릿과 같은 위치(문서 하단, "누가 넣나 읽는 법" 아래)에 채워 넣는다.
- **끝나면 기계 점검 통과를 "검증 완료"로 보고하지 말 것** — 이 gap-fill은 **구조(기계층)만** 채운 것이라, 산문·미러 충실도(의미) 풀 리뷰를 이어붙이거나 "아직"임을 명시한다(checker-guide "구조 업그레이드").
- 표에서 기계적으로 변환해 넣을 뿐이다("JSON 미러" 스키마·매핑대로). 일부 그룹만 JSON이 있으면 빠진 그룹만 채운다.

## ③ 조회

- 데이터 모델에서 특정 데이터를 찾아 답한다.
  예: "이 서비스에 어떤 데이터가 있어?", "`request`에 어떤 필드가 있어?", "금액 관련 변수 뭐야?", "닉네임은 어느 그룹의 뭐야?".
- 이 조회 기능은 주로 **"사람이 직접 물어볼 때"**를 위한 것이다(사람은 파일을 잘 안 읽으므로).
- 조회는 값을 읽어 답할 뿐, 데이터 모델 문서를 바꾸지 않는다.

## ④ json 묶음 추출 (인라인 미러를 한 파일로)

사람(또는 다른 도구)이 요청하면, **문서 안의 그룹별 JSON 미러들을 서비스 단위 한 파일로 묶어** 내려준다.
트리거 예: "json으로 뽑아줘", "이 데이터 모델 json으로 줘", "기계가 읽기 좋게 하나로 줘".

- **스키마는 인라인 JSON 미러와 동일**하다("JSON 미러" 절 참고). 새 형식을 만들지 않고, 각 그룹의 인라인 블록을 그대로 모아 `groups` 배열에 담는다.
- **원본은 언제나 표(md)다.** 이 묶음 파일은 표에서 나온 미러의 사본일 뿐이라, **파일을 고쳐도 원본(md)에 반영되지 않는다.** 이 사실을 안내에 명시한다.

```json
{
  "service": "[서비스 이름]",
  "source": "데이터모델-[서비스].md",
  "note": "표(md)가 원본이며 이 파일은 그 미러의 사본이다. 이 json을 고쳐도 원본에 반영되지 않는다.",
  "groups": [
    {
      "id": "DATA.user",
      "group": "user",
      "label": "회원",
      "description": "이 서비스에 가입한 사용자.",
      "fields": [
        { "name": "email", "label": "회원의 이메일", "type": "string", "required": true, "filledBy": "user", "description": "로그인·연락에 쓰는 이메일" }
      ],
      "relations": [ { "type": "hasMany", "target": "request", "description": "회원 한 명은 요청을 여러 개 만들 수 있다." } ]
    }
  ]
}
```

- 런타임 산출물 파일명 예: `데이터모델-[서비스]-추출.json` (원본과 헷갈리지 않게 "추출"을 드러낸다).
- **이 묶음 파일은 요청받을 때만** 만든다(인라인 미러는 md 안에 상시 존재하지만, 별도 파일로 묶는 것은 요청 시에만).

## 파일이 우선, 스킬 호출은 예외 (반드시 지킨다)

데이터 모델은 **문서(md=SSOT)로 존재**한다. 그래서 이미 있는 데이터를 **"읽기만"** 할 때는
데이터 모델 문서(md)를 직접 읽으면 되고 **스킬을 부를 필요가 없다.** 스킬을 부르는 경우는
문서만으로 안 되는 상황으로 한정한다:

- 필요한 데이터가 없어 **새로 등록**해야 할 때
- 소속 덩어리가 애매하거나, 기존 것과 **충돌·중복이 의심**될 때
- **새 덩어리가 필요**해 보일 때(컨펌 필요)
- 정보 덩어리 자체에 **문제가 있을 때**(원문과 안 맞음 등)

즉 **읽기(read)는 데이터 모델 문서(md) 직접, 쓰기·판정(write/resolve)은 스킬.**

## 데이터 등록·판정 요청을 받을 때의 처리 규칙 (중복·충돌 안전장치)

누가(사람이든 다른 도구든) 데이터 등록·판정을 요청할 때 아래 절차를 따른다.

**1. 요청 형식 검증.** 요청은 최소한 **(ㄱ)소속 덩어리 후보, (ㄴ)뜻(한 줄), (ㄷ)종류**를 갖춰야 한다.
   - 예: "`request`(요청)에 '지원자 수' 필요, 뜻='이 요청에 지원한 사람 수', 종류=숫자."
   - "지원자 수 필요해"처럼 덩어리·뜻·종류가 없거나 모호하면 **바로 등록하지 말고 되묻는다**: "어느 묶음의 데이터인가요? 무슨 뜻인가요?" (같은 뜻의 데이터가 여러 덩어리에 있을 수 있어, 덩어리를 특정하지 않으면 엉뚱한 곳에 넣거나 중복을 못 거른다.)

**2. 조회 먼저 → 기존 흡수.** 정보가 갖춰지면 **지정된 덩어리 안에서 먼저 데이터 모델 문서를 조회**한다.
   같은 개념이 이미 있으면 새로 만들지 않고 **기존 변수명으로 흡수**한다. 없을 때만 신규 등록한다.
   (뜻·종류로 "이름은 같은데 다른 개념 / 이름은 다른데 같은 개념"을 판정)

**3. 덩어리 후보는 확정이 아니다.** 요청자가 지정한 덩어리가 부적절해 보이면 더 맞는 덩어리를
   제안한다(예: "그건 `request`보다 `deal`이 맞아 보입니다"). **최종 소속·이름은 이 스킬이 정한다.**

**4. 안 맞으면 새 덩어리 제안·확인.** 등록하려는 데이터가 기존 어느 그룹에도 안 맞으면, 즉석에서
   새 그룹을 만들지 말고 **"새 묶음이 필요해 보인다"고 사용자에게 제안·확인**한다. 승인되면 그룹을
   만들고, 기획서 쪽에도 반영이 필요함을 알린다.

**5. 결과 회신(구조체).** 조회·등록·흡수·정정·확인대기 **어느 경우든**, 처리 결과를 아래 구조체로
   명확히 돌려준다. 요청자는 이 `variable`을 그대로 참조한다. 이 구조체는 **JSON 미러의 필드 오브젝트와
   같은 스키마**(`name`·`label`·`type`·`required`·`filledBy`·`description`)에 **`group`·`variable`·`resolution`을 더한** 것이다 —
   "문서(미러)에서 읽은 것"과 "회신으로 받은 것"이 구분 없이 똑같이 쓰이게 한다. `type`은 미러와 같은 **영문 enum**을 쓴다.

```json
{
  "variable":    "request.applicantCount",
  "group":       "request",
  "name":        "applicantCount",
  "label":       "요청의 지원자 수",
  "type":        "number",
  "required":    false,
  "filledBy":    "user",
  "description": "이 요청에 지원한 사람 수",
  "resolution":  "created"
}
```

`resolution` 뜻:
- `created` — 신규 등록
- `reused` — 이미 있어 그걸 참조
- `corrected` — 비슷한 걸로 통일
- `needs_confirm` — 새 덩어리 필요 등 **사용자 확인 대기**

등록·정정이 일어나면 **데이터 모델 문서를 갱신**한다.

## 정합성 점검 — 3층 계약을 따른다 (누가·무엇을·언제)

이 문서의 점검은 세트의 **점검 3층 원칙**을 따른다. 정본은 `easyproduct-suite`의
`references/checker-guide.md`("점검 3층 원칙")이며, 여기서 재서술하지 않고 그 계약을 따른다.
갈림 축은 「의미판단 vs 결정론」이다. 이 문서의 `ssot`는 **`table`**(그룹 표가 원본, JSON 미러는 파생)이다.

- **소프트웨어(점검기, 아무때나·세트 전체)** — ID 형식 · 표↔블록 매칭(coverage) · 블록 유효성(스키마+크로스참조 죽은 링크).
  세트 점검기 `check-docs.mjs`가 담당하고 **항상 세트 전체**로 돈다(크로스참조는 전역 속성이라 로컬 수정도 전체를 본다).
  이 스킬이 단독으로 재구현하지 않는다.
- **LLM(에이전트, 풀 리뷰 때만)** — 산문·표 충실도 · 미러 충실도(표의 **뜻**이 JSON에 맞게 담겼나: type·설명·값).
  풀 리뷰 시기: 최초 완성 / 문서 수정 후(부분 풀 리뷰=수정 항목+파장) / 사용자 요구 / 에이전트 판단.
- **이 스킬이 생성·수정 시 로컬로 조용히 보는 것** (위 소프트웨어 검사의 국소 선반영 — 세트 점검기를 대신하지 않는다):
  - **중복 의심**: 종류·한글명·설명이 비슷한데 변수명이 다른 쌍 → "같은 것 아닌가요?" 확인.
  - **명명 규칙 위반**: `그룹.필드` 꼴 아님, 그룹 접두사 없는 전역 변수.
  - **anchor 형식**: JSON 미러의 `id`가 `DATA.<group>` 꼴인지, `group` 값과 일치하는지(`DATA.` + `group`).
  - **frontmatter·태그 누락**: frontmatter의 `machine.*`, 각 블록의 ` ```json datamodel.group ` 태그가 있는지.
  - **애매하면 지어내지 말고** 사용자에게 확인.
- **수정 방향**: 원본은 **표(`ssot: table`)**다. 표↔JSON 미러가 어긋나면 **표 기준으로 JSON 미러를 다시 생성**한다
  (JSON을 손으로 고치지 않는다). 점검기는 어긋남을 **보고**까지, 재생성은 이 스킬(또는 LLM)이 한다.

## 비전문가 응대 규칙 (대화 내내)

- 한 답변에 질문은 **하나, 많아도 둘**. 폭탄 금지.
- "잘 모르겠어요"라고 하면 **구체적 예시 2~3개**를 제시해 고르게 한다.
- 전문 용어가 나오면 **즉시 한 줄로 풀어** 쓴다. (예: "필수 = 이 칸을 비워두면 안 되는 값")
- 영어 변수명은 스킬이 지어 주고, 사용자에게는 **한글명으로** 확인받는다.
- "이건 안 됩니다"보다 "이렇게 잡아뒀는데 필요 없으면 빼세요"로.

## 하지 말 것

- 받은 정보 덩어리 **원문(그룹명·한글명·설명)을 임의로 바꾸지 말 것**(생성 시 초안 보존).
- 시작 입력 (가)에서 받은 데이터 모델을 **기획서 기준으로 덮어쓰지 말 것**(데이터 모델이 SSOT).
- 데이터 모델 문서(md) 안에 **"이 스킬로 만들었다" 같은 제작 정황이나, "스킬이 있으면 이렇게" 식
  조건부 안내를 넣지 말 것.** 문서는 상황을 모르므로 원칙만 상황 불문하고 적는다.
- 사용자가 **영어 변수명을 직접 짓게 하지 말 것**(스킬이 제안).
- 실제로 안 쓸 게 뻔한 필드를 **억지로 깔지 말 것**(지어내기 금지, 애매하면 확인).
- **새 그룹을 컨펌 없이 만들지 말 것.**
- **인라인 JSON 미러를 손으로 고치지 말 것.** JSON은 표에서 생성하는 미러다 — 표를 고치고 JSON을 재생성한다(표와 JSON을 따로 편집해 어긋나게 두지 말 것).
- **별도 `.json` 파일을 상시 만들지 말 것.** 인라인 미러는 md 안에 상시 있지만, [④ json 묶음 추출]의 **별도 파일**은 요청받을 때만 묶는다.
- **결과 회신 구조체의 필드 구성 자체는 바꾸지 말 것.**
- **`schemas/data-model.v1.schema.json`을 손으로 고치지 말 것.** 스킬 소유 고정 자산을 그대로 복사한다(프로젝트마다 다르게 만들지 않는다). 계약을 바꿀 일이면 새 버전(`v2`)으로 낸다.
- **그룹 anchor `id`를 임의 형식으로 짓지 말 것.** 반드시 `DATA.<group>`(사람용 표 변수명에는 `DATA.` 접두사를 넣지 않는다).
- **특정 스킬을 전제한 연계 로직을 넣지 말 것**(요청 처리 규칙·결과 회신은 입력 요건·출력 형식일 뿐, 스킬 결합이 아니다).
