  사용자가 "세트", "오케스트레이터"라는 말을 쓰지 않아도, **여러 개발 문서를 한꺼번에** 완성하려는 신호가 보이면 이 스킬을 사용하세요.

# 개발 문서 한 번에 완성하기 (스킬 지휘)

## 목적

기술 지식이 없는 사람의 아이디어 하나를, **여러 스킬을 순서대로 조율**해
**개발 문서 세트 전체를 한 흐름에** 완성한다. 사용자는 한 번의 대화로
기획서·설계서·화면 구조·디자인 컨셉·정책·약관을 모두 손에 쥔다.

이 스킬은 문서를 **직접 새로 쓰지 않는다.** 대신 이미 있는 하위 스킬들을
**언제·어떤 순서로 부르고, 그 결과를 어떻게 다음 단계로 넘길지**를 관리하는 지휘자다.

## 지휘하는 스킬과 순서

아이디어가 구체에서 겉모습으로 흐르도록, 아래 순서로 조율한다.

```
easyproduct-doc-builder   → 기획서 + 설계서   (무엇을·왜, 어떻게 만들지)
        ↓  (정체성·유형·핵심기능·대상·규칙·정보 덩어리를 넘김)
easyproduct-ia-designer       → 화면·기능 구조(IA) (어떤 화면·기능으로)
        ↓  (정보 덩어리(그룹)를 넘김)
easyproduct-data-model        → 데이터 모델(md)    (어떤 정보를 다루나 · 최초 생성, 기본 포함)
        ↓  (정체성·대상·유형·IA 라벨을 넘김)
easyproduct-design-concept    → 디자인 컨셉        (어떤 색·글꼴·느낌으로 · 무중단 1안)
        ↓
easyproduct-policy-legal      → 정책서 + 약관·개인정보처리방침 (어떤 규칙·약속으로)
        · 기획서·IA에서 재료를 뽑는다(디자인과 무관한 평행 갈래)
        ↓
◆ 디자인 확정 · 리뷰 게이트 → 1안 확정 + 문서 훑어 컨펌 + 개별 수정 라우팅
        ↓  (여기서 옵트인으로 물음)
easyproduct-screen-design     → 화면 설계서(스토리보드) (옵트인 · 사용자 앱 기본(3.8-A) + 백오피스는 별도 호출(3.8-B) · 데이터 모델이 여기서 성장)
        ↓  (옵트인, 3.8 실행했을 때만)
easyproduct-ia-designer(관계도 모듈) → 기능 관계도 (3.8-C, 프론트↔백오피스 연동 — IA에서 읽기 전용 도출)
        ↓
연결·마무리            → 설계서 back-fill · 상호 링크 · 문서 묶음 안내(색인)
        ↓  (옵트인으로 물음)
spec-kit 인계 안내문    → 이 문서 세트를 spec-kit(SDD)에 넘길 인계장 출력
```

> **기본 흐름(자동 포함)**: 기획·설계 → IA → **데이터 모델(최초)** → 디자인 → 정책·약관 → 게이트 → 연결·색인.
> **옵트인(게이트에서 물어 진행)**: **화면 설계**(디자인 확정이 선행돼야 하고 무거움 — 이때 데이터 모델이 필드까지 성장), **spec-kit 인계 안내문**.

**단계를 그냥 이어 돌리지 않는다.** 각 스킬은 저마다 도입부에서 인사하고
자료를 달라고 한다. 그대로 세 번 반복하면 사용자가 지친다. 이 스킬이
**도입·자료 수집을 한 번으로 합치고**, 앞 단계 결과를 뒤 단계에 먹여
**이미 답한 질문은 건너뛰게** 한다. 이어붙이는 구체적 항목·규칙은
`references/skill-handoff-guide.md`에 있다 — 각 단계로 넘어가기 전에 그 파일의 해당 절을 읽는다.

## 두 가지 실행 모드

사용자를 얼마나 자주 멈춰 세울지가 다르다. Step 0에서 정하되, 자료 양을 보고 기본값을 제안한다.

- **가이드 모드(단계별)** — 문서마다 확인받고 다음으로 넘어간다. 아이디어가 막연하거나
  참고 자료가 거의 없을 때 기본. 사용자가 함께 빚어 간다.
- **자동 모드(한 번에)** — 멈춤을 최소화하고 끝까지 초안으로 밀어붙인 뒤, 마지막에 한 번 크게 보여준다.
  가져온 자료가 넉넉하거나 "알아서 빠르게"라고 할 때 기본.

**자료가 얇다고 자동 모드를 가이드로 되돌리지 않는다.** 자동 모드에서 Stage 1은
`easyproduct-doc-builder`를 그 스킬의 **초안 모드**로 실행한다(이 스킬이 동작을 덮어쓰는 게 아니라,
doc-builder가 정식으로 가진 모드를 고르는 것이다). 초안 모드는 기획서를 인터뷰 없이 자료 + 합리적
가정으로 **초안부터 먼저** 만들고, 가정한 것은 "가정"에 표시한다. 가이드 모드에서는 doc-builder를
기본(대화) 모드로 실행한다.

두 모드 모두 **두 곳의 게이트는 지킨다**: (1) **기획서 확정 게이트**(아래 Stage 1) — 기획서는 모든
문서의 토대라 여기가 틀리면 나머지가 전부 어긋난다. 자동 모드에서도 기획서 요약은 반드시 한 번
보여주고 바로잡을 틈을 준다(침묵·"맞아요"면 통과). (2) **디자인 확정·리뷰 게이트**(아래 Stage 3.7) —
무중단으로 걸어둔 디자인 1안을 여기서 사용자가 확정하고, 그동안 만든 문서를 훑어 컨펌한다.
자동 모드라도 이 두 게이트에서는 사용자에게 확인을 구한다(화면 설계·spec-kit 인계는 이 게이트에서
옵트인으로 물어 진행하므로, 자동 모드가 이 지점을 그냥 통과하지 않는다).

## Step 0: 오리엔테이션 + 자료 확인 (딱 한 번, 짧게)

하위 스킬들의 Step 0를 여기서 한 번으로 합친다. **이 뒤로 각 스킬의 도입 인사·자료 요청은 실행하지 않는다.**

- "아이디어 하나를 **개발에 바로 쓸 수 있는 문서 세트**로 만들어 드릴게요. 순서대로 ① 기획서·설계서 → ② 화면·기능 구조 → ③ 디자인 컨셉을 만들고, 앞에서 정한 건 뒤에서 다시 묻지 않습니다. 기술 용어는 몰라도 됩니다."
- **"미리 적어두신 메모·기획안·자료나, 이미 만든 문서가 있으세요? 있으면 먼저 보여주세요. 거기서 채울 수 있는 건 채우고 모자란 것만 여쭤봅니다."**
- 자료 양을 보고 모드를 제안한다: 자료가 넉넉하면 "이 자료면 제가 **한 번에** 쭉 초안을 잡고 마지막에 보여드릴까요, 아니면 **문서마다 확인**하며 갈까요?" / 자료가 얇으면 가이드 모드로 시작하겠다고 안내한다.

수집한 자료는 **세 단계가 공유**한다. 뒤 단계에서 자료를 다시 달라고 하지 않는다.

## Step 1: 어디서 시작할지 정하기 (재진입 · 부분 요청)

바로 Stage 1로 가기 전에, 아래를 확인해 **불필요한 재작업을 막는다.**

- **이미 만든 문서가 있으면** 그것을 기준선으로 삼고, **없는 문서만** 새로 만든다.
  단계가 아니라 **문서 단위**로 따진다 — 기획서 · 설계서 · 화면 구조 · 데이터 모델 · 디자인 컨셉을
  각각 확인해, 있는 건 재사용하고 **없는 건 반드시 채운다.** 기본 세트에 어느 하나가 빠진 채 끝나면 안 된다.
  (예: 기획서만 있으면 → 기획서는 재사용, **설계서는 아직 없으니 만들고**(Stage 1의 Phase B) 그다음 Stage 2·2.5·3로.)
  (데이터 모델·화면 설계는 각각 시작점 판단이 다르다 — 데이터 모델 문서가 이미 있으면 재사용하고,
  화면 설계는 옵트인이라 기존 게 없으면 게이트에서 물어 진행한다.)
- **"있다/없다"만으로 끝내지 않는다 — 오래된 구조인지도 확인한다.** 기존 문서가 있으면, 단순
  존재 여부만이 아니라 **해당 하위 스킬의 현재 산출물 구조(그 스킬의 assets 템플릿)와 대략
  맞는지**도 함께 살핀다. 눈에 띄게 다르면(예: 화면설계서가 도메인별로 안 갈리고 한 파일뿐이다,
  UI 컴포넌트 인벤토리 파일 자체가 없다, 로그인 필요 필드가 없다, 기능 ID(FEAT)가 없다,
  **정책서에 규칙 ID(`POL.*`)가 없다**, **데이터 모델에 그룹별 인라인 JSON 미러가 없다** 등)
  **"오래된 구조로 보이는데 최신 구조로 업그레이드할까요?"**를 한 번 확인한다. 예라고 하면
  **재사용이 아니라 그 하위 스킬의 재진입/버전업 절차**로 넘긴다(처음부터 다시 만들지 않고
  gap만 채우는 절차 — 스킬마다 이름이 다르다):
  - `easyproduct-doc-builder` → 풍부화 모드
  - `easyproduct-ia-designer` / `easyproduct-screen-design` → "재진입 — 스킬 버전업 업그레이드"
  - `easyproduct-data-model` → "② 수정"(시작 입력 (가), 기존 파일 기준선 — **그룹별 인라인 JSON 미러 gap-fill 부여 포함**)
  - `easyproduct-policy-legal` → "Step 5: 재진입 — 부분 수정"(구조 동기화 목적으로도 활용 — **`POL.*` ID 부여 포함**)
  - 또한 기존 세트에 **시나리오가 없으면**, **IA 이상이 갖춰져 있으면**(기획서·IA가 있고, 정책·데이터가 있으면 더 좋음)
    "사용자 여정을 시나리오로 정리할까요?"를 **한 번 제안**한다(옵트인 신규 도입 — `easyproduct-scenario`의 파생으로 기존 확정 문서에서 처음 만든다).
    화면 설계서가 있으면 스텝을 화면까지 정밀하게 잇고, 없으면 IA·정책 수준의 여정으로 도출한다(그 한계를 한 줄 밝힌다).
  단순 재사용 여부가 애매하면(구조 차이가 크지 않아 보이면) 굳이 묻지 않고 그대로 재사용한다 —
  사소한 차이까지 매번 확인받아 사용자를 피곤하게 하지 않는다.
  - **업그레이드(gap-fill)는 "구조만 맞춘 것"이지 "검증 완료"가 아니다(중요).** 버전업으로 frontmatter·블록을
    채운 뒤에는 반드시 **Stage 4의 기계 점검(3-2)에 더해 LLM 풀 리뷰(3·3-1: 산문·미러 충실도)를 이어붙인다**
    — 업그레이드는 세트에 걸친 대규모 수정이라 "수정 후 풀 리뷰" 대상이다(checker-guide "구조 업그레이드" 참고).
    소프트웨어(기계층)만 통과했으면 사용자에게 **"통과/완료"로 보고하지 말고 "구조는 맞췄으나 의미 점검(풀 리뷰)은
    아직"**임을 명시한다. 기계 통과를 검증 완료로 흘려 보고하지 않는다.
- **일부만 원하면**(예: "기획서랑 화면 구조까지만") 그 범위까지만 조율하고 나머지 단계는 접는다.
  이 스킬은 세트가 기본이지만, 사용자가 그은 범위를 존중한다.
- 어디까지 필요한지 애매하면 한 번만 확인한다: "① 기획·설계 → ② 화면 구조 → ③ 데이터 모델 → ④ 디자인 →
  ⑤ 정책·약관까지 기본으로 만들고, 그 뒤 **화면 설계·시나리오·spec-kit 인계**는 물어서 진행할까요?"

## Stage 1: 기획서 · 설계서 (easyproduct-doc-builder 조율)

`easyproduct-doc-builder`의 SKILL.md를 읽고 그 흐름을 따르되, **아래만 조정한다.**

- 산출물 경로: 기획서는 **`ssot/plan.md`**, 설계서는 **`ssot/design-spec.md`**로 저장한다.
- **모드를 골라 실행한다.** 자동 모드면 doc-builder를 그 스킬의 **초안 모드**로, 가이드 모드면 **기본(대화) 모드**로 실행한다.
  (doc-builder가 두 모드를 정식으로 갖고 있으므로, 동작을 새로 덮어쓰지 않고 모드만 지정한다.)
- **Step 0(오리엔테이션·자료 요청)은 실행하지 않는다.** 이미 Step 0에서 했다. 공유 자료를 그대로 쓴다.
- 자료가 있으면 그 스킬의 Step 1(자료 흡수)로 추출부터 하고, **빈 곳·얇은 곳만** 묻는다(초안 모드면 가정으로 채운다).
- **기획서 게이트는 반드시 지킨다.** 요약 카드(정체성·핵심 행동 3개·성공 기준·가정)를 보여주고 확정받는다.
  자동 모드에서도 이 요약은 보여준다. 확정 전에는 설계서로 넘어가지 않는다.
- 설계서(Phase B)를 만들 때, **IA·디자인이 채울 두 자리는 지금 비워 두고 표시**한다:
  - "대략의 구성" → `[이후 화면·기능 구조(Stage 2)에서 채움]`
  - "겉모습·느낌" → `[이후 디자인 컨셉(Stage 3)에서 채움]`
  마지막 연결 단계(Stage 4)에서 이 자리를 실제 내용으로 채운다.
- "제품 원칙" 선택 모듈이 나오면(분위기·색·접근성 신호), 여기서 정한 원칙을 **Stage 3의 출발점으로 넘긴다**고 기억한다.
- **유저 스토리 (세트 디폴트 ON).** 세트에서는 doc-builder의 "유저 스토리" 선택 모듈을 **기본으로 켜서** 호출한다
  (개별 스킬은 기본 꺼짐이지만, 통합은 개발 전 문서를 완비하는 게 목적이라 자동 생성이 기본이다).
  산출 경로는 **`supporting/user-stories.md`**. 분량은 아래 우선순위로 정한다 — 자동 모드에서도 이것 때문에 멈추지 않는다:
  1. 사용자가 그 문서를 콕 집어 분량을 지정하면(예: "유저 스토리는 간단히") 그걸 따른다.
  2. 없으면, 사용자가 문서 전반을 "자세히"로 원했으면 유저 스토리도 **자세히**로.
  3. 둘 다 없으면 **적당히**(디폴트).
  "유저 스토리는 빼줘"처럼 끄라고 하면 이 옵션만 끈다. 생성되면 Stage 4 색인에 항목으로 추가한다.

Stage 1이 끝나면 다음 단계로 넘길 것을 정리해 둔다(자세한 목록은 연결 가이드):
**한 줄 정체성 · 서비스 유형 · 형태(웹/앱)·규모·유료 여부 · 핵심 기능과 규칙·금지 · 대상 사용자(특별 대상 포함) · 핵심 정보 덩어리.**

### Stage 1.5: 시나리오 스케치 — 앞 게이트 (옵트인, 흐름 복잡 시 권장)

기획 확정 직후, **화면을 짓기 전에** 흐름 결함을 값싸게 잡는 조기 게이트다. **핵심 행동이 여럿이거나 분기·상태 전이가 많은
복잡한 흐름**이면 한 번 제안한다: "화면을 만들기 전에 사용자 여정을 먼저 훑어 흐름이 말이 되는지 볼까요?" (단순한 서비스면 건너뛴다.)

- 예면 `easyproduct-scenario`의 **Step S1(스케치)**를 호출한다. 여정은 기획서 핵심 행동에서 도출한다(자연어, GWT 강요 안 함).
- 스케치에서 **흐름 결함**(안 이어짐·빠진 단계·논리 모순)이 드러나면, 시나리오 스킬의 **"설계 결함 처리 원칙"**대로 후보와 회귀 위치(기획/IA)를 **보고**하되,
  **되돌릴지는 사용자가 정한다**(자동 재설계 금지). 사용자가 택하면 해당 단계로 돌아가 고친 뒤 이어간다.
- **결정 이관 — 타이밍 주의**: 이 시점(Stage 1.5)엔 IA·데이터·정책이 **아직 생성 전**이다. 그래서 스케치에서 나온 새 결정은
  ⓐ **이미 있는 원본(기획서)** 에 해당하면 지금 옮기고, ⓑ 아직 안 만든 문서(화면·기능→IA, 데이터, 정책) 몫이면
  **"스케치 산출물"에 후보로 모아 두었다가 뒤 Stage(2·2.5·정책)의 생성 입력으로 먹인다.** 결정을 스케치에만 남기지 않는다.
- 스케치 산출물은 `[상태: 탐색]`, 폐기 가능(뒤에서 파생물로 대체). **아래 Stage 2·2.5·정책 단계가 이 산출물을 입력으로 받는다.**

## Stage 2: 화면 · 기능 구조 IA (easyproduct-ia-designer 조율)

`easyproduct-ia-designer`의 SKILL.md를 읽고 따르되, **아래만 조정한다**(도입·자료요청만 생략, 핵심 동작은 그대로).

- 산출물 경로: **`ssot/ia.md`**.
- **Step 0과 자료 요청은 실행하지 않는다.** Stage 1의 기획서·설계서를 입력으로 먼저 읽는다.
- **Stage 1.5 스케치가 있으면 입력으로 흡수한다(드라이버).** 스케치가 뽑아둔 "이 여정에 필요한 화면·기능" 후보를 IA 초안의
  화면·기능 목록에 반영한다(스케치는 흐름, IA는 그 흐름을 화면·기능 구조로 확정). 스케치가 없으면 이 항목은 건너뛴다.
  스케치에 담겨 있던 **정책·데이터 성격 결정**은 각각 정책 단계·Stage 2.5(데이터)로 넘겨 그 단계 입력이 되게 한다.
- Step 1의 3~4개 질문 중 **기획서에서 답이 나온 것은 묻지 않는다.** 보통 유형·정체성·핵심기능·형태·유료여부는
  이미 넘어와 있어, 남는 질문이 0~1개다. 남은 것만 짧게 묻거나, 자동 모드면 추론해 채우고 "가정"에 남긴다.
- 초안 우선 방식을 지킨다: 유형 기본기능 + 백오피스까지 자동으로 채운 **전체 초안**을 만든다.
- 가이드 모드면 초안을 한 번 보여주고 "빼거나 더할 것"만 받아 고친다. 자동 모드면 초안대로 확정하고
  가정·자동추가 항목을 문서에 표시한다.
- **화면·기능 구조 문서를 파일로** 만든다(`ssot/ia.md`, 그 스킬의 기본 산출 방식). 이 IA는 Stage 4에서 설계서의 "구성" 자리로 들어간다.
- **유스케이스 (세트 디폴트 ON).** 세트에서는 ia-designer의 "유스케이스" 선택 모듈을 **기본으로 켜서** 호출한다
  (개별 스킬은 기본 꺼짐이지만, 통합은 문서 완비가 목적이라 자동 생성이 기본이다). 산출 경로는 **`supporting/use-cases.md`**.
  분량 우선순위는 유저 스토리와 동일 — 자동 모드에서도 멈추지 않는다:
  ① 그 문서를 콕 집은 분량 지정 → ② 없으면 문서 전반 "자세히" 요청 시 자세히 → ③ 둘 다 없으면 적당히(디폴트).
  "유스케이스는 빼줘"라고 하면 이 옵션만 끈다. 생성되면 Stage 4 색인에 항목으로 추가한다.

## Stage 2.5: 데이터 모델 최초 생성 (easyproduct-data-model 조율 · 세트 기본 ON)

`easyproduct-data-model`의 SKILL.md를 읽고 따르되, **아래만 조정한다**(도입·자료요청만 생략).

- **세트 기본 포함이다.** 기획서의 정보 덩어리(그룹)만 있어도 프리셋을 얹어 **최초 데이터 모델**을
  만들 수 있으므로, 여기까지는 묻지 않고 기본으로 생성한다. ("데이터 모델은 빼줘"라고 하면 끈다.)
- **Step 0과 자료 요청은 실행하지 않는다.** Stage 1 기획서의 "기억해 둬야 할 정보(정보 덩어리 — 그룹명·
  한글명·설명)"를 입력으로 넘긴다(doc-builder가 그룹 수준으로 확정해 둔 것). IA도 참고로 읽는다.
- 그 스킬대로 각 덩어리에 **공통 관리 필드 + 종류별 표준 필드 프리셋**을 자동으로 깔아
  **`ssot/data-model.md` 한 파일**을 만든다. 이 문서(표)가 데이터 SSOT다(기획서보다 우선).
  (요청 시 그 시점의 읽기 전용 json 스냅샷을 추출할 수 있으나, 상시 산출물은 아니다.)
- **여기서는 "최초 생성"까지만.** 개별 필드가 실제로 자라나는 것은 화면 설계(Stage 3.8)에서
  화면을 그리며 data-model에 추가되는 몫이다. 이 단계에서 필드를 억지로 다 채우려 하지 않는다.
- 자동 모드면 초안대로 두고 자동/가정 항목을 표시한다. 가이드 모드면 md(표)를 한 번 보여주고
  "빼거나 더할 그룹·필드"만 받는다(수정은 스킬이 이 문서에 직접 반영한다).
- 생성되면 Stage 4 색인에 데이터 모델 항목으로 추가한다.

## Stage 3: 디자인 컨셉 (easyproduct-design-concept 조율)

`easyproduct-design-concept`의 SKILL.md를 읽고 따르되, **아래만 조정한다**(도입·자료요청만 생략, 핵심 동작은 그대로).

- **Step 0과 자료 요청은 실행하지 않는다.** Stage 1의 기획서·(있으면 제품 원칙)와 Stage 2의 IA를 먼저 읽는다.
- Step 1의 질문 중 **대상·유형·이미 정해진 색/로고는 앞 문서에서 뽑아** 건너뛴다. 남는 것은 보통
  분위기·좋아하는 사이트·밝게/어둡게 정도다. **가이드 모드면 이 3개만 짧게 묻고**, 자동 모드(배치 흐름)면
  자료·유형에서 추론해 채우고 "가정"에 남긴 뒤 곧장 3안 생성으로 간다.
- 그 스킬대로 **서로 다른 3안**을 만들고(1안=추천), **무중단으로 1안을 확정**한다. 어느 안이 좋은지 물어 멈추지 않는다.
- **디자인 컨셉 문서 + 3안 미리보기(UI 키트·샘플·비교표)를 파일로** 만든다. 확정된 1안 요약은 Stage 4에서 설계서의 "겉모습" 자리로 들어간다.
  경로: 컨셉 문서는 **`ssot/design-concept.md`**, 3안 미리보기(HTML·PNG)는 **`reference/design-preview/`**
  (보존 대상 — `temp/`가 아니다. 안 선택의 근거로 계속 참조될 수 있어 목업과 달리 버리지 않는다).
- **미리보기 이미지(PNG)를 "이 환경에선 안 된다"고 넘겨짚지 않는다.** 먼저 변환을 **시도**한다
  (`scripts/render_png.py`, Playwright/headless Chromium — 이 스킬의 HTML은 CSS 변수를 쓰므로
  **wkhtmltoimage는 지원하지 않는다**, 폴백으로 시도하지 않는다). 실제로 Playwright를 못 쓸 때만
  HTML 미리보기로 대체하고, 그 사실을 한 줄로 알린다. 도구가 있는데도 PNG를 생략하지 않는다.

## Stage 3.5: 정책 · 약관 (easyproduct-policy-legal 조율)

`easyproduct-policy-legal`의 SKILL.md를 읽고 따르되, **아래만 조정한다**(도입·자료요청·성격 파악 질문만 생략, 핵심 동작·안전 원칙은 그대로).

- **디자인과 무관한 평행 갈래다.** 재료는 **기획서 + IA**에서 뽑는다(회원·결제·신고·위치·마케팅 신호). 디자인 컨셉은 쓰지 않는다.
  순서상 여기 두지만 Stage 2 뒤 어디에 와도 무방하다.
- **독립 스킬의 "성격 파악 질문"은 실행하지 않는다.** 기획서·IA에 이미 답이 있으므로 거기서 뽑고, 운영주체 이름·연락처처럼
  문서에 없는 값만 `[확인 필요]`로 남긴다(자동 모드면 묻지 않고 그대로 비워 둔다).
- **세트 디폴트: 정책서 · 약관 둘 다 자동 생성.** (개별 스킬은 요청 시 갈라 만들지만, 세트는 완비가 목적이라 둘 다 기본이다.)
  경로: 정책서는 **`ssot/policy.md`**, 약관·개인정보처리방침은 **`ssot/terms-privacy.md`**.
  - "정책서만" / "약관만"으로 콕 집으면 그것만, "정책/약관 빼줘"면 이 단계를 건너뛴다.
  - 분량 우선순위는 유저 스토리·유스케이스와 동일 — ① 콕 집은 분량 지정 → ② 전체 "자세히" → ③ 적당히(디폴트). 자동 모드에서 멈추지 않는다.
- **안전 원칙은 절대 조정하지 않는다.** 약관·개인정보처리방침 문서 앞뒤의 **전문가 검토 경고**, 지어내지 않기(사람만 아는 값은 `[확인 필요]`)는
  그 스킬 그대로 유지한다. 세트가 무중단으로 돈다고 해서 이 경고를 빼거나 완성본처럼 포장하지 않는다.
- 정책서·약관을 각각 **파일로** 만든다. 생성되면 Stage 4 색인에 항목으로 추가한다.

## Stage 3.7: 디자인 확정 · 리뷰·컨펌 게이트 (이 스킬의 고유 단계)

기본 세트(기획·설계·IA·데이터 모델·디자인·정책·약관)가 다 나온 뒤, 화면 설계로 넘어가기 **직전에**
한 번 멈춰 확정·검토한다. 자동 모드라도 이 게이트는 지킨다.

1. **디자인 1안 확정.** 디자인 컨셉은 무중단으로 1안을 걸어 두었다(Stage 3). 화면 설계는 확정된
   시각 방향이 있어야 하므로, 여기서 사용자에게 **1·2·3안 중 확정**을 구한다: "디자인은 1안(추천)으로
   잡아뒀는데 이대로 갈까요, 2·3안으로 바꿀까요?" 사용자가 안 번호를 고르면 디자인 컨셉 문서의
   "선택된 안"만 그 번호로 바꾼다(easyproduct-design-concept Step 3.5-(가)). 새 방향을 원하면 그 스킬의
   3.5-(나)로 3안을 새로 만든다.
2. **문서 훑어 컨펌.** 지금까지 만든 핵심 문서(기획서·설계서·IA·데이터 모델·디자인·정책서·약관)를
   **요약해 보여주고** 이상이 없는지 확인한다: "이대로 화면 설계로 넘어가도 될까요? 고칠 문서 있으세요?"
3. **개별 수정 → 해당 스킬로 라우팅.** 사용자가 특정 문서의 수정을 요구하면, **그 문서의 스킬을 수정
   모드로 재진입**시켜 고치고 게이트로 돌아온다. 전체를 다시 만들지 않는다. 매핑:
   - 기획서·설계서 → `easyproduct-doc-builder`(풍부화/재진입 모드로 지정 부분만)
   - 화면 구조(IA) → `easyproduct-ia-designer`(검토+수정, 바뀐 부분만)
   - 데이터 모델 → `easyproduct-data-model`(수정 — 데이터 모델 문서를 직접 고침)
   - 디자인 컨셉 → `easyproduct-design-concept`(안 번호면 3.5-(가), 새 방향이면 3.5-(나))
   - 정책서·약관 → `easyproduct-policy-legal`(**Step 5 재진입** — 지정 조항만, `#n`·검토 경고·1·2·3부 구조 보존)
   수정이 앞 문서의 근거를 바꾸면(예: 기능 추가) 영향 받는 뒤 문서도 해당 스킬로 다시 손본다.
   문서를 고쳤으면 back-fill·상호 링크가 어긋나지 않게 Stage 4에서 다시 맞춘다.
4. **옵트인 두 가지를 여기서 묻는다.** 컨펌이 끝나면:
   - **"화면 설계서까지 만들까요?"**(기본은 사용자 앱 기준) → 예면 Stage 3.8-A로. 아니오면 Stage 4(연결)로 바로 간다.
     (관리자 백오피스 화면 설계서는 3.8-A가 끝난 뒤 별도로 묻는다 — Stage 3.8-B 참고.)
   - (화면 설계 여부와 별개로) 맨 마지막에 **"spec-kit으로 넘길 인계 안내문을 만들까요?"**(Stage 5)를 묻는다.

## Stage 3.8: 화면 설계 (easyproduct-screen-design 조율 · 옵트인)

Stage 3.7에서 사용자가 화면 설계를 원할 때만 실행한다. 무겁고(전 화면) 디자인 확정이 선행돼야 하므로
기본 자동 흐름에 넣지 않고 옵트인으로 둔다.

**이 스킬은 한 번에 한 범위(사용자 앱 또는 백오피스)만 다룬다.** 사용자 앱과 관리자 백오피스는
인증·배포·개발 리포지토리가 갈리는 별개 프로젝트인 경우가 대부분이므로, 문서 단계부터 완전히
분리한다. 두 번 호출될 수 있다.

### 3.8-A: 사용자 앱 화면 설계 (기본)
- `easyproduct-screen-design`을 **기본 범위(사용자 앱)**로 호출한다. 도입·자료 요청은 생략한다.
  재료: **IA의 사용자(프론트) 화면(FEAT ID·사이트맵 순서) · 확정 디자인 컨셉(Stage 3.7에서 확정,
  여백 척도·타이포 위계 포함) · 데이터 모델 문서**.
- 디자인 "확정"은 이 스킬이 다시 하지 않는다. Stage 3.7에서 이미 확정한 안을 그대로 입힌다.
- **이 스킬은 화면설계서부터는 "초안 우선"이 아니다.** 처음부터 컴포넌트·배치·문구가 확정된
  명세로 만든다(다른 Stage와 다른 예외 — 화면설계서는 곧 구현 명세이기 때문). 다만 진행 리듬은
  유지한다: **메인 화면을 풀세트(글 정의+데이터+HTML 목업)로 먼저 만들어 컨펌**받고(공통 UI 정의·
  화면 전역 정책·**UI 컴포넌트 인벤토리 최초 등록**도 이때 함께 확정), 나머지 화면도 똑같이
  정확한 명세로 만들되 목업(HTML 렌더)만 이후 선택적으로 진행한다.
- **데이터 모델이 여기서 성장한다.** 화면을 그리다 새 데이터가 필요하면, 직접 만들지 않고
  `easyproduct-data-model`에 의도(덩어리·뜻·종류·쓰는 화면)를 넘겨 확정 변수를 회신받아 참조한다.
  이미 있는 변수는 데이터 모델 문서를 직접 읽는다. 배치(3~4개) 단위로 모아 등록한다.
- **UI 컴포넌트 인벤토리도 여기서 함께 자란다.** 새 컴포넌트가 필요하면 먼저 인벤토리를 조회해
  재사용 가능한지 확인하고, 없으면 등록한다(재사용 단위는 중앙 인벤토리, 특정 화면 전용은
  로컬 `UI.FEAT.*`로 그 도메인 파일 안에).
- 산출물은 **여러 파일**이다(더 이상 한 파일이 아님):
  - `screens/user/screen-design-user-index.md` — 공통 UI 정의 + 도메인 파일 링크(사이트맵 순서)
  - `screens/user/screen-design-user-{domain}.md` — 도메인별 화면 전부(사이트맵 순서) + 로컬 컴포넌트
  - `ssot/ui-components-user.md` — 중앙 UI 컴포넌트 인벤토리
  - (선택) `temp/mockups/user/화면-[이름]-목업.html` — 참고물, `.gitignore` 대상, 재생성 가능
  전부 생성되면 Stage 4 색인에 추가한다(인덱스·도메인 파일들·인벤토리 각각 링크).

### 3.8-B: 백오피스 화면 설계 (옵트인 — 별도 호출)
3.8-A가 끝나면 한 번 묻는다: **"관리자(백오피스) 화면 설계서도 만들까요?"** 예일 때만 실행한다.

- `easyproduct-screen-design`을 **백오피스 범위**로 **별도로 다시** 호출한다. 재료: **IA의 관리자(백오피스)
  화면(FEAT ID·사이트맵 순서) · 확정 디자인 컨셉(같은 안) · 데이터 모델 문서(3.8-A와 동일한 문서, 공유한다)**.
- 메인/기준 화면은 **관리자 대시보드(또는 로그인 후 첫 화면)**로 따로 컨펌받는다. 공통 UI 정의도
  **관리자용으로 별도**(LNB 좌측 메뉴 중심)로 확정한다 — 사용자 앱의 공통 UI와 공유하지 않는다.
- **디자인은 확정 컨셉의 "절제된 변형"을 적용한다**(easyproduct-screen-design이 백오피스 범위일 때 자동
  적용하는 규칙 — 채도 낮춤·그림자/라운드 축소·여백 축소로 밀도 향상·강조색은 상태표시로만 절제).
  이 변형 규칙을 백오피스 공통 UI 정의 상단에 명시한다.
- **UI 컴포넌트 인벤토리는 백오피스 전용으로 별도 등록한다** — 절제된 변형이 적용되므로 사용자 앱
  인벤토리(`ssot/ui-components-user.md`)와 값이 다르다. 새 파일 `ssot/ui-components-backoffice.md`
  를 만들되, 같은 이름의 컴포넌트라도 값(색·radius·간격 등)은 절제된 변형 기준으로 따로 정한다.
- **데이터 모델은 3.8-A와 완전히 공유한다.** 새 데이터 필요 시 절차는 동일(의도 넘기고 확정 변수
  회신). 백오피스 전용 데이터 모델을 따로 만들지 않는다 — 두 프로젝트가 같은 회원·거래 데이터를
  다른 화면으로 보여줄 뿐이다.
- 산출물은 3.8-A와 같은 구조를 백오피스 범위로 만든다:
  - `screens/backoffice/screen-design-backoffice-index.md`
  - `screens/backoffice/screen-design-backoffice-{domain}.md`
  - `ssot/ui-components-backoffice.md`
  - (선택) `temp/mockups/backoffice/화면-[이름]-관리자-목업.html`
  전부 생성되면 Stage 4 색인에 **별도 항목**으로 추가한다(사용자 앱 화면설계서·인벤토리와 합치지 않는다).

### 3.8-C: 기능 관계도 (옵트인 — easyproduct-ia-designer의 선택 모듈)

Stage 3.8이 실행됐을 때만(3.8-A만 했든, 3.8-B까지 했든) 그 끝에 한 번 묻는다:
**"프론트↔백오피스 연동 관계도도 만들까요? (한 기능이 다른 기능을 트리거하는 관계 — 가입→회원관리,
신고→신고처리, 결제→정산 등 — 를 그림으로 보여줍니다)"** 예일 때만 실행한다. 화면 설계 자체를
건너뛴 경우(Stage 3.7에서 화면 설계서를 원치 않았던 경우)는 이 옵트인도 묻지 않는다.

- `easyproduct-ia-designer`의 "선택 모듈: 기능 관계도"를 호출한다. **도입·자료 요청은 생략**한다 —
  이미 만들어진 Stage 2의 IA 문서를 입력으로 바로 넘긴다(그 스킬은 IA 문서를 **읽기만** 하고
  바꾸지 않는다).
- 그 스킬대로 **프론트↔백오피스 연동 관계도**(트리거 관계)를 도출한다. UI 흐름 관계도(화면 전이)는
  화면 설계서의 "이어지는 화면" 항목과 역할이 겹치므로 세트에서는 기본으로 만들지 않는다 — 사용자가
  따로 요청하면 그때 함께 만든다.
- 노드는 **FEAT ID**(Stage 2에서 확정된 것)로 그린다. IA 문서가 원본(SSOT)이고 이 관계도는 **읽기
  전용 파생물**이다 — IA가 바뀌면 다시 도출하면 된다는 것을 산출물에 명시한다.
- 산출물: **`reference/feature-relation-map.md`**(읽기 쉬운 관계 목록 + mermaid 그림). 생성되면 Stage 4 색인에
  "참조용(파생물)"으로 추가한다.

### 3.8-D: 파생 시나리오 (옵트인 — easyproduct-scenario의 파생 단계)

Stage 3.8(화면 설계)이 실행됐을 때만, 그 끝에 한 번 묻는다:
**"사용자 여정을 시나리오로 정리할까요? (여러 화면·기능·정책을 관통하는 흐름을 Given/When/Then + 추적 링크로
정리해, 문서끼리 어긋나지 않는지 점검하고 나중에 개발 테스트로 넘길 수 있게 합니다)"** 예일 때만 실행한다.
화면 설계를 건너뛴 경우는 이 옵트인도 묻지 않는다.

- `easyproduct-scenario`의 **Step S3(파생)**를 호출한다. 도입·자료 요청은 생략하고, 확정된 IA·데이터·정책·화면
  설계서를 입력으로 넘긴다(그 스킬은 원본을 **읽기만** 한다).
- 시나리오는 각 스텝이 닿는 `FEAT.*`·`그룹.필드`·`POL.*`를 **링크만** 하는 **읽기 전용 파생물**이다
  (기능 관계도와 같은 원칙 — 원본이 바뀌면 다시 도출). GWT 정형화는 이 파생 단계에서만 한다.
- 산출물: **`supporting/scenarios/scenario-{도메인}-{이름}.md`**(mermaid flowchart 파생 뷰 포함). 생성되면 Stage 4 색인에 "보강자료"로 추가한다.
- 파생 중 **설계 결함**(흐름이 안 이어짐 등)이 드러나면 Stage 4의 정합·추적 점검과 같은 방식으로 **회귀 후보와 함께 보고**하고 사용자가 진행을 정한다.
- 이 뒤 게이트와 별개로, 흐름이 복잡한 서비스는 **앞 게이트(Stage 1.5 시나리오 스케치)**로 화면을 짓기 전에 흐름을 먼저 검증하는 편이 비용이 싸다.

## Stage 4: 연결 · 마무리 (이 스킬의 고유 단계)

세 하위 스킬 이상의 결과를 **하나의 세트로 엮는다.** 여기서만 하는 일이다.

> **점검 3층 계약 (정본: `references/checker-guide.md` "점검 3층 원칙").** 아래 점검 하위단계는 3층에 대응한다 — 갈림 축은 「의미판단 vs 결정론」:
> - **소프트웨어(결정론·아무때나·항상 세트 전체)** = **3-2 기계 점검**(`check-docs.mjs`: ID 형식 · 산문↔블록 매칭(coverage) · 블록 유효성(스키마+크로스참조 죽은 링크)). 크로스참조는 전역 속성이라 로컬 수정이어도 **세트 전체**로 돈다.
> - **LLM(의미판단·풀 리뷰 때만)** = 산문 충실도 + 미러 충실도. **3(완성도)·3-1(시나리오 정합)**이 이 층이며, 각 문서 미러가 산문의 **뜻**을 맞게 담았는지도 이때 본다.
> - **풀 리뷰 시기**: 세트 **최초 완성 시(이 Stage 4가 그 1회)** / 이후 문서 수정 시 **부분 풀 리뷰(수정 항목 + 파장)** / 사용자 요구 / 에이전트 판단. 부분 풀 리뷰라도 **소프트웨어(3-2)는 항상 세트 전체**로 돌려 파장을 식별한 뒤, LLM 풀 리뷰는 수정 항목 + 소프트웨어가 새로 깬 곳만 본다.
> - **수정 방향**: 어긋나면 각 문서 `ssot`(table/prose)의 **원본 쪽을 진실로** 두고 블록/미러를 **원본 기준으로 재생성**한다. 점검은 **보고**까지, 재생성은 해당 스킬/LLM이 한다.

1. **설계서 확인 · back-fill.** 세트에는 설계서가 **반드시** 포함된다 — 이번 실행에서 새로 만들었든 원래 있었든.
   (재진입으로 설계서가 없으면 Stage 1의 Phase B로 먼저 만든다.) 그다음 Stage 1에서 비워 둔 두 자리를 채운다:
   "대략의 구성" ← Stage 2 IA의 핵심(화면 지도·주요 화면), "겉모습·느낌" ← Stage 3의 확정 컨셉 요약(주색·글꼴·톤).
2. **상호 링크.** 각 문서 머리에 "이 세트의 다른 문서" 한 줄을 넣어 서로를 가리키게 한다.
   (화면설계서가 사용자·백오피스 둘 다 있으면, 각 문서가 서로를 "별개 프로젝트 문서"로 가리키고
   공유하는 데이터 모델도 함께 가리킨다.)
3. **완성도 점검.** `easyproduct-doc-builder`의 `references/speckit-fields.md` 체크리스트로,
   개발(spec-kit)이 뒤에서 추출할 내용이 세트 전체에 빠짐없이 있는지 **조용히** 확인한다(사용자에게 보여주는 단계 아님).
   빠진 게 있으면 해당 문서에 `[확인 필요: …]`로 남기거나 짧게 되묻는다.
   — 이것은 **누락(completeness)** 점검이다.
3-1. **시나리오 정합·추적 점검 (시나리오가 있을 때만).** 3.8-D에서 시나리오를 만들었으면,
   `easyproduct-scenario`의 **Step S4**(절차: 그 스킬 `references/traceability-check.md`)로 문서 간 **어긋남(consistency)**을 점검한다.
   완성도 점검이 "빠졌나"를 보는 것과 달리, 이것은 "**시나리오가 가리키는 ID가 실제로 있고, 원본과 모순되지 않는가**"를 본다:
   - **죽은 링크**(시나리오가 참조하는 `FEAT.*`·`그룹.필드`·`POL.*`가 원본에 없음)는 확실히 잡아 보고한다.
   - **커버리지 누락**(기획서 핵심 행동·핵심 유스케이스 중 **어떤 시나리오에도 안 닿는 것**)은 테스트 구멍이 되므로 **반드시** 보고한다.
   - **의미적 모순**(시나리오 흐름 ↔ IA·정책 결정)은 확신할 때만 불일치로 보고하고, 애매하면 "미확인"으로 남긴다(거짓 안심 금지).
   - 원본이 바뀐 흔적이 있으면 해당 시나리오를 **재도출**하라고 표시한다.
   - **설계 결함 후보**(흐름이 안 이어짐·단계 누락 등, 원본 쪽 문제로 보이는 것)는 **회귀 후보 단계(기획/IA/화면)와 함께 사용자에게 제시**하고,
     **되돌릴지는 사용자가 정한다**(자동 재설계 금지). 사용자가 회귀를 택하면 해당 스킬의 재진입 절차로 넘겨 원본을 고친 뒤 시나리오를 재도출한다.
   결과는 **누락·불일치가 있을 때만** 조용히 보고한다(완성도 점검과 같은 톤, 단 커버리지 누락은 반드시). 시나리오가 없으면 이 단계는 건너뛴다.
3-2. **기계 점검 (frontmatter·스키마·크로스도큐먼트 참조 — 시나리오 없어도 항상).** 각 문서는 이제 frontmatter와
   기계 블록(데이터 모델 `datamodel.group`, IA `ia.features`, 정책 `policy.rules`, 화면 설계 `screendesign.screens`,
   인벤토리 `uicomponents.list`, 디자인 `design.tokens`, 시나리오 `scenario.trace`)을 갖는다. 소프트웨어로 다음을 점검한다:
   - **스키마 검증**: 각 기계 블록이 문서 frontmatter가 가리키는 payload 스키마(`schemas/*.json`)에 맞는지(형식·enum·anchor 형식). 위반은 해당 문서의 문제로 보고.
   - **크로스도큐먼트 참조 무결성**: 접두사로 어느 등기부를 볼지 정해 자동 조회한다 — 화면 설계의 `feat`→`ia.features`, `components`(+`data.io.ui`·`data.bindings.ui`)→`uicomponents.list`, `data.display`·`data.io.sends`·`data.io.receives`·`data.bindings.vars`→`datamodel.group`; 시나리오 `refs`→각 종류의 등기부; UI/화면의 토큰 참조→`design.tokens`; 시나리오·화면의 `POL.*`→`policy.rules`. **끊긴 참조(죽은 링크)**를 보고한다.
   - 이 점검은 3-1(시나리오 정합)과 겹치는 부분이 있으나 **시나리오가 없어도** 돌고, 사람이 못 보는 문서 수십 개의 ID 참조를 **기계가 일괄** 확인한다는 점에서 완성도·정합 점검을 보완한다. 위반이 있을 때만 조용히 보고한다.
   - **실행 방법**: 이 스킬에 딸린 점검기 **`scripts/check-docs.mjs`**(무의존 Node)를 문서 세트 폴더에 겨눠 돌린다 — `node <이 스킬 경로>/scripts/check-docs.mjs <프로젝트 폴더>`(node를 쓸 수 있는 환경). 규약·판정 기준은 `references/checker-guide.md`. node를 못 쓰는 환경(claude.ai 등)이면 그 가이드의 "점검자 알고리즘"대로 직접 읽어 점검하거나, 자동 점검이 어려우면 그 사실을 알린다. **점검기는 프로젝트 폴더로 복사하지 않는다**(문서 세트엔 문서 + `schemas/*.json`만 나간다 — 점검기는 이 스킬이 들고 실행하는 자산).
   - **점검 결과별 인계.** 점검기는 **보고만** 한다 — 실제 수정은 해당 스킬로 넘긴다:
     - `옛 스키마 형식 — 스키마 이행 필요`(data-model) → **`easyproduct-data-model` 스킬의 [⑤ 스키마 이행]**으로 그 파일을 옮긴다(내용 보존, `auto:preset`은 뜻으로 판단하고 회신에서 알림). ※ 0.5.0 이행용 한시 안내.
     - `스키마 사본이 스킬 자산과 다름` → **어느 쪽이 최신인지 먼저 확인**한 뒤, 문서 옆 사본이 옛것이면 스킬 자산으로 맞추고, 스킬이 옛것이면 스킬을 업그레이드한다(점검기는 방향을 단정하지 않는다).
     - `죽은 링크`·`중복 그룹 정의`·`스키마 위반` → 해당 원본 문서를 소유한 스킬로 넘겨 고친다(점검기가 직접 고치지 않는다).
4. **문서 묶음 안내 생성.** `assets/doc-bundle-index-template.md`를 채워, 어떤 문서가 있고 무엇을 위한 것이며
   어떤 순서로 읽는지, 지금 선택된 디자인 안이 무엇인지 한 장에 정리한 **색인 문서**(`00-index.md`, 프로젝트
   폴더 루트)를 만든다. **이 색인은 SSOT가 아니라 참조용 파생물이다** — 만든 시점의 폴더 상태를 훑어 정리한
   스냅샷일 뿐, 실제 최신 근거는 항상 각 원본 문서다. 템플릿 지시대로 **생성 시각**을 채우고,
   **"이후 문서가 개별 수정되면 이 목록이 실제와 달라질 수 있다"는 경고**와 **"이 파일은 지워도 개발에
   지장 없다"는 안내**를 반드시 남긴다(템플릿에 이미 문구가 있으니 날짜만 채우면 된다).
   **유저 스토리·유스케이스·데이터 모델·정책서·약관은 세트에서 기본 생성되므로 색인에 함께 넣는다**(사용자가 "빼줘"로 끈 것만 제외).
   **화면 설계서·시나리오(옵트인)를 만들었으면 그것도 색인에 넣는다**(시나리오는 "보강자료"로 표시). 약관·개인정보처리방침 항목에는 "초안·점검용, 전문가 검토 필요"라는 한 줄을,
   데이터 모델 항목에는 "이 문서(표)가 데이터의 SSOT"라는 한 줄을 색인에도 덧붙인다.
   - **frontmatter + 문서 매니페스트.** 색인 맨 앞에 frontmatter(`doc_type: doc-bundle-index`·`machine.{tag: docbundle.docs, schema}`)를 두고, 사람용 문서 목록과 함께 하단에 ` ```json docbundle.docs ` 매니페스트 블록을 만든다 — 세트에 **실제로 만들어진 모든 문서**를 `{docType, path, role}`로 담는다(사람용 표가 원본, 이 블록은 미러). `schemas/docbundle.v1.schema.json`을 색인 옆 `schemas/`에 복사한다. 하네스는 이 한 블록에서 세트의 모든 문서를 발견하고 각 경로의 frontmatter를 대조한다.
4-1. **`.gitignore` 생성.** 프로젝트 폴더 루트에 `.gitignore`를 만들고 아래 세 줄을 넣는다:
   - `temp/` (목업은 `ssot/`·`screens/`만 있으면 재생성 가능하므로 버전관리 대상에서 뺀다)
   - `speckit-handoff-*.md` (spec-kit 인계 안내문은 세트 폴더 구조 밖의 산출물이라 정해진 자리가
     없다. 혹시 프로젝트 폴더 안에 남더라도 이 패턴이 커밋을 막는 안전장치다. Stage 5를 아직
     안 만들었어도 나중에 만들 가능성을 감안해 처음부터 넣어 둔다.)
   - `00-index.md` (색인은 SSOT가 아니라 그 시점 스냅샷인 파생물이다. **삭제하라고 지시하지
     않는다** — 사람이 나중에 다시 열어볼 수 있게 디스크에는 남겨두되, 저장소에 커밋되어 여러
     사람 사이에서 오래된 스냅샷이 진실처럼 남는 걸 막기 위해 git 추적에서만 뺀다.)
4-2. **색인 재생성 (유틸리티, 상시 가능).** Stage 4 이후 언제든, 사용자가 "색인 다시 만들어줘",
   "색인 최신화해줘"처럼 요청하면 세트를 처음부터 다시 만들지 않고 **이 단계만 다시 실행**한다:
   현재 프로젝트 폴더를 실제로 훑어(`ssot/`·`supporting/`·`reference/`·`screens/` 등에 무엇이 있는지
   확인) 그 시점 기준으로 `00-index.md`를 새로 쓴다(생성 시각도 그 시점으로 갱신). 이 재생성은
   가벼운 유틸이라 별도 게이트 없이 바로 실행한다. Step 1(재진입 판단)에서 기존 색인이 실제 폴더와
   눈에 띄게 달라 보이면(예: 색인엔 없는 화면 설계서 폴더가 실제로 존재) "색인이 오래된 것 같은데
   다시 만들까요?"를 한 번 제안할 수 있다.
5. **마무리 안내(짧게).** 만든 문서를 요약해 알린다(만든 것만 언급 — 화면 설계서는 옵트인이라 만들었을 때만).
   "이대로 개발에 쓰시면 되고, 바꾸고 싶은 부분은 해당 문서만 고치면 됩니다. **약관·개인정보처리방침은 초안이라, 실제 사용 전 법률 전문가 검토가 필요합니다.**"
   spec-kit 명령(`/speckit.specify` 등)을 사용자에게 안내하지 않는다. 대신 원하면 Stage 5의 **인계 안내문**을 만들어 준다.

## Stage 5: spec-kit 인계 안내문 (옵트인 · 이 스킬의 고유 단계)

Stage 3.7에서 사용자가 "spec-kit으로 넘길 안내문을 만들까요?"에 예라고 했을 때만, 세트가 다 나온 뒤 실행한다.
목적: 완성된 문서 세트를 **spec-kit(SDD)으로 개발할 AI 에이전트에게 건네는 인계장**을 출력한다.
사용자는 이 안내문을 spec-kit 세션의 첫 입력으로 붙여넣기만 하면 된다.

- **spec-kit 명령을 직접 지정하지 않는다.** spec-kit도 AI 에이전트 위에서 도는 도구라, 명령 이름·순서·단계는
  그쪽이 알아서 판단하게 둔다. 우리는 **"이런 문서들이 있고, 각각이 무엇이며, 이걸 근거로 spec-kit 방식에
  맞춰 진행하라"**만 넘긴다. (명령명은 버전에 따라 바뀌므로 하드코딩하지 않는다.)
- **인계 안내문은 정해진 폴더에 넣지 않는다.** `ssot/`·`supporting/`·`reference/`·`screens/`·`temp/`
  어디에도 편입시키지 않고, **만들어서 바로 사용자에게 전달**한다(claude.ai에서는 present_files로,
  Claude Code에서는 만든 파일 경로를 알려준다). 이 문서는 spec-kit 세션 첫 입력으로 한 번 쓰고 나면
  다시 참조할 일이 없는 산출물이라, 세트 폴더 구조 안에 자리를 마련해 두지 않는다.
  (혹시 Claude Code 환경에서 이 파일이 프로젝트 폴더 안에 그대로 남더라도, `.gitignore`에 이미
  `speckit-handoff-*.md` 패턴이 등록돼 있어 실수로 커밋될 위험은 없다 — Stage 4-1 참고.)
- `assets/speckit-handoff-template.md`를 채워 인계 안내문 파일을 만든다. 담을 것:
  1. **한 줄 상황** — 개발 착수 전 문서 세트가 준비돼 있으니, 이 문서들을 근거로 spec-kit(SDD) 방식으로 진행해 달라.
  2. **문서 목록 + 한 줄 소개 + 파일명** — 이번에 실제로 만든 문서만(기획서·설계서·IA·데이터 모델·
     디자인 컨셉·유저 스토리·유스케이스·정책서·약관·(있으면)화면 설계서), **실제 저장 경로**(`ssot/plan.md` 등)를 그대로 적는다.
  3. **어디를 보면 되는지 안내(강제 아님)** — 무엇을·왜·수용 기준은 기획서·유저 스토리·유스케이스에,
     기술 방향은 설계서에, 데이터 구조는 데이터 모델 문서에, 화면은 화면 설계서(글 정의)에, 불변 원칙은
     정책서·제품 원칙에 있다는 식으로 **위치만** 알려주고, 어느 spec-kit 단계에 넣을지는 그쪽 AI가 판단하게 한다.
  4. **문서별 SSOT · 보강자료 · 참조용 구분표** — 세트의 각 문서(및 화면 설계서 글정의/목업, 디자인 컨셉
     문서/3안 미리보기처럼 **한 스킬 안에서도 원본과 파생물이 갈리는 경우**)를 "SSOT(결정이 사는 곳)",
     "보강자료(다른 SSOT의 내용을 검증 가능한 형태로 가공 — 유저 스토리·유스케이스. 원본과 다르면 원본이
     우선)", "참조용(파생물·임시 참고물, 표준 아님)" 세 갈래로 표로 정리해 넣는다. 이미 정해진 우선순위를
     정확히 반영한다: 데이터 모델 문서가 기획서보다 우선(데이터 구조에 한해) / 화면 상세는 화면 설계서
     글정의가 IA 화면정의보다 우선(단 범위는 IA 안) / 목업·3안 미리보기·색인은 참조용. **약관·개인정보처리방침은
     이 축과 별개로 "초안 상태(전문가 검토 전 확정 아님)"임을 표 아래 별도로 명시한다.**
  5. **진행 태도** — 문서에 없는 건 지어내지 말고 확인, 사람이 단계별로 검토할 수 있게 점진적으로 진행(spec-kit의 점진 구현 철학과 맞음).
- 안내문을 만들고 나면 **present_files로 즉시 전달**한다(또는 Claude Code 환경이면 파일 경로를 알려준다).
  색인에는 "spec-kit 인계 안내문(전달됨, 세트 폴더 구조 밖)" 항목을 추가한다.

**화면 설계서가 사용자 앱·백오피스 둘 다 있으면, 인계 안내문도 프로젝트별로 둘 만든다.**
사용자 앱과 백오피스는 별개 프로젝트로 개발될 가능성이 높으므로(인증·배포·리포지토리가 다름),
인계장 한 개에 두 화면 설계서를 섞지 않는다. 둘 다 위와 같이 **만들어 바로 전달**하며, 파일명은
`speckit-handoff-user.md` / `speckit-handoff-backoffice.md`(전달용 이름일 뿐, 저장 폴더 의미 아님)로 구분한다.
- `speckit-handoff-user.md` — 문서 목록에 **사용자 앱 화면설계서**만 넣는다(백오피스 화면설계서는
  이 프로젝트와 무관하므로 뺀다).
- `speckit-handoff-backoffice.md` — 문서 목록에 **백오피스 화면설계서**만 넣는다.
- **데이터 모델은 두 인계장 모두에 동일하게 넣는다** — 두 프로젝트가 공유하는 SSOT이기 때문이다.
  각 인계장에 한 줄 명시한다: "이 데이터 모델은 [반대쪽 프로젝트명]과 공유합니다. 이 프로젝트에서
  데이터 구조를 바꾸면 반대쪽에도 영향이 갑니다 — 임의로 바꾸지 말고 데이터 모델 자체를 함께 갱신하세요."
- 기획서·설계서·IA·디자인 컨셉·정책서·약관 등 공통 문서는 두 인계장에 모두 넣되, IA는 "이 문서 중
  [사용자/관리자] 화면·기능 부분이 이 프로젝트에 해당한다"를 한 줄 덧붙인다.
- 화면 설계서가 하나만 있으면(백오피스를 안 만들었으면) 인계장도 하나만 만든다(지금까지의 단일
  인계장 방식 그대로).

## 지휘하는 스킬이 설치되어 있지 않을 때

하위 스킬 중 일부가 없을 수 있다. 그때는:

- **가능하면 그 스킬을 그대로 부른다.** 설치돼 있으면 위 Stage처럼 SKILL.md를 읽고 따른다.
- **없으면 조용히 대체하지 말고 알린다.** "그 문서를 만드는 전용 스킬이 없어, 같은 방식으로 제가 직접 정리해 볼게요"라고
  한 줄 밝히고, 해당 단계의 산출물(기획서/IA/데이터 모델/디자인 컨셉/정책·약관/화면 설계서)을 이 스킬의 원칙(비전문가 언어·초안 우선·항상 파일)으로 만든다.
  단, **정책·약관을 직접 만들 때도 전문가 검토 경고와 지어내지 않기 원칙은 반드시 지킨다.** 데이터 모델을 직접 만들 때도 이 문서(표)가 SSOT라는 원칙을, 화면 설계를 직접 만들 때도 글 정의=SSOT·목업=참고물 원칙을 지킨다.
- **순서·이어붙이기·마무리는 스킬 유무와 무관하게 유지한다.** 어느 단계를 직접 채웠든, Stage 4의 연결·안내는 똑같이 한다.

## 비전문가 응대 규칙 (대화 내내)

- 도입·자료 요청은 **전 과정 통틀어 한 번**. 단계마다 다시 인사하지 않는다.
- 한 답변에 질문은 **하나, 많아도 둘**. 하위 스킬들의 질문을 한꺼번에 쏟지 않는다.
- 앞 단계에서 답한 것을 **뒤 단계에서 다시 묻지 않는다.** 그게 이 스킬의 존재 이유다.
- 전문 용어는 즉시 한 줄로 풀어 쓴다. "이건 안 됩니다"보다 "이렇게 해뒀는데 필요 없으면 빼세요"로.
- 가끔 지금 어디쯤인지 한 줄로 알린다: "기획서 끝났고, 이제 화면 구조로 갑니다." 사용자가 통제감을 느끼게.

- `scripts/skill-lint.mjs` — **스킬 부합 린트(무의존 Node).** 베타 머지 게이트의 **기계 가능** 항목(빈 껍데기 금지·뜻 필수·정합성 점검 절 존재·점검기 자산)을 자동 점검하고 실패 시 종료코드 1. 문서 세트가 아니라 **스킬 저장소 자체**에 겨눈다(`node skills/easyproduct-suite/scripts/skill-lint.mjs`). 근거·의미 판단 항목은 checker-guide "스킬 부합 점검".
- `assets/doc-bundle-index-template.md` — 세트를 한 장으로 묶는 색인 문서 구조 (이대로 채운다. frontmatter + `docbundle.docs` 매니페스트 포함)
- `assets/speckit-handoff-template.md` — spec-kit(SDD)으로 넘길 인계 안내문 구조 (Stage 5에서 이대로 채운다)
- `schemas/docbundle.v1.schema.json` — 색인 `docbundle.docs` 매니페스트 블록의 JSON Schema 계약(스킬 소유 고정 자산, 색인 옆 `schemas/`로 복사).

## frontmatter 규약 질문 대응

사용자(대개 기술 사용자·개발자·에이전트)가 문서의 **frontmatter나 기계 블록의 의미**를 물으면 — 예: "이 `ssot: table`이 뭐야?", "`doc_type`은 어디에 쓰여?", "`machine.tag`가 뭐하는 거야?", "점검 소프트웨어가 이 필드를 어떻게 쓰지?" — `references/checker-guide.md`를 근거로 답한다.

- **핵심 구분을 분명히 한다**: 스키마는 값의 *구조*만 강제하고(예: `ssot`는 `table|prose`), 값으로 *무엇을 할지*(행동)는 그 계약서가 규정한다. 필드 자체가 사용법을 알려주지는 않는다.
- 필드별 뜻·점검자 행동·doc_type/anchor 레지스트리·참조 라우팅은 계약서 표에 있다 — 그걸 요약해 답한다.
- 비전문가가 물으면 쉬운 말로("문서 맨 앞의 이 메모는 소프트웨어가 이 문서를 읽고 점검하기 위한 표시입니다"), 개발자·에이전트가 물으면 계약서 세부로 답한다.

**점검·유지보수 키트 제공 (요청 시 · 선제 제안).** 이 문서 세트를 **이어받는 사람·에이전트가 easyproduct 스킬을 안 가질 수 있다.** 그때 세트로 나가는 건 문서 + `schemas/*.json`뿐이라 계약 설명과 점검기가 빠진 반쪽 다리다. 그래서 요청 시(또는 선제 제안 시) 아래를 **자족 키트**로 제공한다:

1. **`scripts/check-docs.mjs`** — 생성된 **문서 세트**를 점검하는 무의존 Node 점검기(우리 스킬 없이 CI·에이전트가 실행).
2. **`references/checker-guide.md`** — 계약·점검 알고리즘·**점검 3층 원칙**·**"스킬 없이 유지보수하는 법"** 절.
3. **스키마** — 이미 세트에 동봉되어 나감(추가 조치 불필요).

- **기본 출력은 그대로 깨끗하게**(문서 + 스키마). 위 키트는 **요청받을 때** 보여주거나 파일로 내려준다.
- **선제 제안 트리거**: "이걸 **CI에 물릴 것**", "**개발팀이 이어받음**", "**다른 에이전트가 점검**"처럼 우리 스킬 없이 소비·유지보수될 신호가 보이면 먼저 제안한다("점검기와 유지보수 안내를 함께 드릴까요?").
- **키트에 포함하지 않는 것 — `scripts/skill-lint.mjs`.** 그건 **생성 문서가 아니라 easyproduct 스킬 저장소 자체**를 점검하는 제작자용 게이트다. 다운스트림과 무관하므로 **제외**한다(check-docs는 포함, skill-lint는 제외 — 이 구분을 지킨다).
