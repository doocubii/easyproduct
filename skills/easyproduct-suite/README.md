# easyproduct-suite

> 개발 지식이 없는 사람의 아이디어 하나를, 여러 전문 스킬을 순서대로 조율해 개발 착수 전 문서 세트 전체(기획서·설계서 → 화면·기능 구조 → 데이터 모델 → 디자인 컨셉 → 정책·약관 → 옵트인으로 화면 설계서·인계 안내문)로 한 흐름에 완성해 주는 지휘(오케스트레이터) 스킬입니다.

[easyproduct 스킬 세트](../../README.md)의 일부입니다.

## 무엇을 하나
- 이 스킬은 문서를 직접 새로 쓰지 않고, 이미 있는 하위 스킬들을 **언제·어떤 순서로 부르고 그 결과를 어떻게 다음 단계로 넘길지** 관리하는 지휘자 역할을 합니다.
- 각 스킬의 도입 인사·자료 요청을 **한 번으로 합치고**, 앞 단계에서 정한 것(정체성·유형·핵심기능·대상·정보 덩어리 등)을 뒤 단계에 먹여 **같은 질문을 반복하지 않게** 합니다.
- 실행 방식을 두 모드로 나눕니다 — **가이드 모드**(문서마다 확인받으며 진행)와 **자동 모드**(멈춤을 최소화하고 끝까지 초안으로 밀어붙인 뒤 마지막에 한 번 크게 보여줌).
- 두 곳의 게이트(기획서 확정 게이트, 디자인 확정·리뷰 게이트)는 모드와 무관하게 반드시 지킵니다.
- 여러 스킬의 결과를 하나의 세트로 엮고(설계서 back-fill·상호 링크·색인 문서 생성), 원하면 spec-kit(SDD)으로 넘길 인계 안내문까지 만들어 줍니다.

## 언제 쓰나
- "개발 문서 전부/처음부터 끝까지 만들어줘", "기획부터 디자인까지 한 번에"처럼 **여러 문서를 한 번에** 원할 때
- 아이디어만 있고 무엇부터 어디까지 만들지 막막한데 **여러 문서가 다 필요할** 때
- 기획서나 화면 구조 중 하나를 만든 뒤 "나머지도 이어서 다 해줘"라고 할 때(있는 건 재사용, 없는 것만 채움)
- spec-kit / SDD로 프로젝트를 시작하려는데 그 앞단 문서 세트 전부가 필요할 때
- 반대로 딱 한 종류의 문서(기획서만 / 화면 구조만 / 디자인만)를 원하면, 이 스킬 대신 해당 개별 스킬을 쓰세요.

## 산출물
이 스킬은 오케스트레이터라, 자체 문서를 새로 쓰기보다 세트 전체 산출물을 조율합니다. 산출물은 대개 프로젝트 폴더 아래에 쌓입니다.

- 기본 세트(자동 포함): 기획서(`ssot/plan.md`)·설계서(`ssot/design-spec.md`) · 화면·기능 구조 IA(`ssot/ia.md`) · 데이터 모델(`ssot/data-model.md`) · 디자인 컨셉(`ssot/design-concept.md`) + 3안 미리보기(`reference/design-preview/`) · 정책서(`ssot/policy.md`)·약관·개인정보처리방침(`ssot/terms-privacy.md`) · 유저 스토리(`supporting/user-stories.md`)·유스케이스(`supporting/use-cases.md`)
- 이 스킬 고유 산출물: 세트를 한 장으로 묶는 **색인 문서**(`00-index.md`, 프로젝트 폴더 루트)와 `.gitignore`
- 옵트인(물어서 진행): 화면 설계서(`screens/user/`, `screens/backoffice/`) + UI 컴포넌트 인벤토리 · 기능 관계도(`reference/feature-relation-map.md`) · 파생 시나리오(`supporting/scenarios/`) · **spec-kit 인계 안내문**(정해진 폴더에 넣지 않고 만들어 바로 전달)

## 단독 실행 vs 세트
- 이 스킬 자체가 하나의 **오케스트레이터**입니다 — 단독으로 실행하지만, 그 안에서 하위 스킬 6개(doc-builder · ia-designer · data-model · design-concept · policy-legal · screen-design)와 시나리오 스킬을 순서대로 조율합니다.
- 하위 스킬 중 일부가 설치돼 있지 않으면, 조용히 대체하지 않고 "전용 스킬이 없어 같은 방식으로 직접 정리하겠다"고 한 줄 밝힌 뒤 이 스킬의 원칙(비전문가 언어·초안 우선·항상 파일)으로 해당 산출물을 만듭니다. 순서·이어붙이기·마무리는 스킬 유무와 관계없이 유지합니다.

## 구성 파일
- `SKILL.md` — 조율 순서·모드·게이트·단계별 조정 규칙을 담은 스킬 본문
- `references/skill-handoff-guide.md` — 단계 사이에 무엇을 넘기고, 설계서를 어떻게 back-fill하며, 모드 기본값과 게이트 라우팅을 어떻게 정하는지 (각 Stage 진입 전 해당 절을 읽음)
- `assets/doc-bundle-index-template.md` — 세트를 한 장으로 묶는 색인 문서 구조 (Stage 4에서 이대로 채움)
- `assets/speckit-handoff-template.md` — spec-kit(SDD)으로 넘길 인계 안내문 구조 (Stage 5에서 이대로 채움)

## 세트 안에서의 위치
- 이 스킬은 세트의 **지휘자**입니다. 아이디어가 구체에서 겉모습으로 흐르도록, 기획·설계 → IA → 데이터 모델 → 디자인 → 정책·약관 순으로 하위 스킬을 호출하고 각 단계의 결과를 다음 단계로 이어붙입니다.
- 기본 세트가 다 나오면 디자인 확정·리뷰 게이트에서 한 번 멈춰 확인받고, 그 자리에서 화면 설계서·spec-kit 인계 안내문을 옵트인으로 물어 진행합니다.
- 개별 문서를 고쳐야 하면 전체를 다시 만들지 않고 해당 문서의 스킬만 수정 모드로 재진입시킨 뒤 게이트로 돌아옵니다.
