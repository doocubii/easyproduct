아래는 개발에 착수하기 전에 준비한 문서 세트입니다. **이 문서들을 근거로 spec-kit(SDD) 방식에 맞춰
개발을 진행해 주세요.** 무엇을·왜부터 시작해 계획·태스크·구현으로 spec-kit의 흐름대로 가시면 됩니다.

## 준비된 문서 (이번에 만든 것만, 이 프로젝트에 해당하는 화면 설계서만)
| 문서 | 무엇인가 | 파일 |
|---|---|---|
| 기획서 | 무엇을·왜 만드는가(정체성·핵심 기능·규칙·성공 기준·정보 덩어리) | `ssot/plan.md` |
| 설계서 | 어떻게 만드는가(형태·규모·기술 방향·구성) | `ssot/design-spec.md` |
| 화면·기능 구조(IA) | 어떤 화면·기능으로 이루어지는가(프론트+관리자 — 이 프로젝트는 [사용자/관리자] 부분에 해당). 각 기능에 고유 ID(`FEAT.도메인.기능명`)가 붙어 있어, 화면 설계서·유스케이스가 이름이 아니라 이 ID로 정확히 참조합니다 | `ssot/ia.md` |
| 데이터 모델 | 다루는 정보의 그룹·필드·구조. **⚠ [반대쪽 프로젝트명]과 공유하는 SSOT**입니다 — 이 프로젝트에서 데이터 구조를 바꾸면 반대쪽에도 영향이 갑니다. 임의로 바꾸지 말고 데이터 모델 자체를 함께 갱신하세요 | `ssot/data-model.md` |
| 디자인 컨셉 | 색·글꼴·컴포넌트·톤(확정 안). **백오피스 프로젝트라면** 이 값에 절제된 변형(채도 낮춤·밀도 향상)을 적용한 것이 화면 설계서에 반영돼 있습니다 | `ssot/design-concept.md` |
| 유저 스토리 | 누가·무엇을·왜 원하는가(+수용 기준) | `supporting/user-stories.md` |
| 유스케이스 | 주요 목표별 흐름·예외·사후조건 | `supporting/use-cases.md` |
| 정책서 | 운영 규칙(회원·결제·환불·신고 등) | `ssot/policy.md` |
| 이용약관·개인정보처리방침 | 법적 문서 **초안·점검표** | `ssot/terms-privacy.md` |
| (있으면) 화면 설계서 — **이 프로젝트분만** | 인덱스+도메인별 파일. 글 정의(SSOT)+데이터+목업. IA 사이트맵 순서 | `screens/{scope}/screen-design-{scope}-index.md` (+ 도메인 파일들) |
| (있으면) UI 컴포넌트 인벤토리 — **이 프로젝트분만** | 재사용 컴포넌트 규격(SSOT). 화면 정의의 `UI.*` ID 근거 | `ssot/ui-components-{scope}.md` |
| (있으면) 기능 관계도 | 프론트↔백오피스 연동(트리거) 관계 그림. **참조용**(원본은 IA) | `reference/feature-relation-map.md` |

## 문서별 SSOT · 보강자료 · 참조용 구분 (충돌 시 이 표를 따르세요)

여러 문서를 동시에 읽다 보면 서로 미세하게 안 맞아 보이는 순간이 있을 수 있습니다. 그럴 때
아래 세 갈래로 판단해 주세요.
- **SSOT** — 결정이 실제로 사는 곳. 다른 문서와 어긋나면 이 문서가 맞습니다.
- **보강자료** — 어떤 SSOT의 내용을 검증 가능한 형태(수용 기준·흐름 등)로 풀어쓴 문서. 그 자체로
  새 결정을 담고 있지 않으며, 원본 SSOT와 다르면 원본이 맞습니다. 다만 무시해도 되는 문서는
  아닙니다 — spec-kit이 수용 기준·시나리오를 세울 때 실제로 참조해야 할 자료입니다.
- **참조용** — 다른 문서에서 파생되었거나 선택을 돕기 위한 임시 자료. 표준으로 삼지 않습니다.

### SSOT 문서 — 결정이 사는 곳
| 문서 | 무엇의 SSOT인가 |
|---|---|
| 기획서 | 서비스의 범위·기능·규칙·성공 기준 (단, **데이터 구조는 데이터 모델이 우선**) |
| 설계서 | 기술 스택·구조 결정 |
| 화면·기능 구조(IA) | 화면·기능의 **목록과 범위** (단, 화면 설계서가 있으면 **개별 화면의 상세 동작은 그쪽이 우선** — 범위 자체는 여전히 IA를 벗어나지 않음) |
| 데이터 모델 | 데이터 항목·이름·구조. **기획서보다 우선**(데이터 구조에 한해) |
| 디자인 컨셉 문서의 "선택된 안" 값 | 색상 코드·글꼴·컴포넌트 규칙 |
| 정책서 | 운영 규칙의 구체 값(수수료율·환불·제재 기준 등) |
| 화면 설계서의 "글 정의" | 화면별 상세 동작·구성. 있으면 IA의 화면 정의보다 상세 기준 |
| UI 컴포넌트 인벤토리 | 재사용 컴포넌트(버튼·카드·네비 등)의 규격·파라미터. 화면 정의의 `UI.*` ID가 가리키는 근거 |

### 보강자료 — SSOT를 검증 가능한 형태로 가공 (원본과 다르면 원본이 우선, 그러나 실제로 참조해야 함)
| 문서 | 무엇을 보강하는가 |
|---|---|
| 유저 스토리 | 기획서의 대상·행동·기능을 "누가·무엇을·왜 + 수용 기준"으로 검증 가능하게 풀어씀 |
| 유스케이스 | 기획서·IA의 핵심 흐름을 "행위자·사전조건·기본/대안/예외 흐름·사후조건"으로 상세화 |

### 참조용 문서 — 파생물·임시 참고자료 (표준으로 삼지 않는다)
| 문서 | 성격 |
|---|---|
| 화면 설계서의 HTML 목업 | 디자인·UI를 눈으로 확인하는 참고물. 목업의 레이아웃·배치를 근거로 삼지 않는다 |
| 디자인 컨셉의 3안 비교·키트·샘플(HTML·PNG) | 안을 고르도록 도운 **임시 참고물**. 표준은 컨셉 문서에 적힌 값이다 |
| (있으면) 기능 관계도 | IA에서 도출한 프론트↔백오피스 연동(트리거) 관계 그림. **원본은 IA** — IA와 다르면 IA가 맞다 |
| 이 인계 안내문 자체 · 문서 세트 안내(색인) | 문서를 소개하는 지도일 뿐, 그 자체가 결정의 근거는 아니다 |

### 별도 표시 — 이용약관·개인정보처리방침은 "초안 상태"
위 구분과는 다른 축의 문제입니다. 이 문서는 **법적 효력을 갖는 최종본이 아니라 초안·점검표**입니다.
대한민국 법령 기준으로 작성됐고, **실제 사용 전 반드시 법률 전문가의 검토**가 필요합니다.
확정된 법률 문서로 취급하지 말고, `[확인 필요]`로 남은 고유값은 사업자·전문가가 채워야 함을
그대로 유지해 주세요.

## 어디를 보면 되는지 (spec-kit 단계 매핑은 참고일 뿐, 강제 아님)
- **무엇을·왜 + 수용 기준** → 기획서(SSOT) · 유저 스토리 · 유스케이스(보강자료)
- **기술 방향(스택·구조)** → 설계서(SSOT)
- **데이터 구조** → 데이터 모델(SSOT)
- **화면 동작·구성** → 화면 설계서의 **글 정의**(SSOT, 있으면). HTML 목업은 참조용
- **불변 원칙(지켜야 할 규칙)** → 정책서(SSOT) · (있으면) 제품 원칙
- 어느 것을 spec-kit의 어느 단계·산출물에 넣을지는 **spec-kit 흐름에 맞춰 판단**해 주세요.
