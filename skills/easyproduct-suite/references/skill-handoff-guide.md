# 스킬 연결 가이드 (단계 사이에 무엇을 넘기나)

이 파일은 지휘 스킬이 **각 단계로 넘어가기 직전에** 읽는 참고서다.
핵심은 하나: **앞 단계에서 이미 정한 것을 뒤 단계에서 다시 묻지 않게** 정확히 넘긴다.
잘못 넘기면 사용자가 같은 질문에 두 번 답하게 되고, 이 스킬의 이유가 사라진다.

## 목차
1. 실행 모드 기본값 정하기
2. Stage 1 → Stage 2 로 넘기는 것 (기획·설계 → IA)
3. Stage 1·2 → Stage 3 으로 넘기는 것 (→ 디자인)
4. Stage 4: 설계서 back-fill 짝맞춤
5. 재진입(이미 만든 문서가 있을 때) 판단
6. 놓치기 쉬운 함정
7. Stage 2 → Stage 2.5 로 넘기는 것 (IA·기획 → 데이터 모델)
8. Stage 3.7 디자인 확정·리뷰 게이트 (확정 + 컨펌 + 수정 라우팅)
9. Stage 3.8 화면 설계로 넘기는 것 (+ 데이터 모델 성장, + 범위 분리)
10. Stage 5 spec-kit 인계 안내문
11. Stage 3.8-C 기능 관계도로 넘기는 것

---

## 1. 실행 모드 기본값 정하기

Step 0에서 자료 양·사용자 태도를 보고 기본 모드를 **제안**한다(강요하지 않는다).

| 상황 | 기본 제안 |
|---|---|
| 참고 자료·기존 문서가 넉넉함 | **자동 모드** — 추출로 대부분 채워지니 멈춤이 적어도 정확하다 |
| "알아서 / 빠르게 / 다 해줘" 태도 | **자동 모드** |
| 아이디어가 한두 문장뿐, 자료 없음 | 사용자 선호대로. **자동이면** doc-builder를 초안 모드로(신호+가정으로 기획서 초안→게이트), **가이드면** 기본 모드로(핵심만 인터뷰) |
| 대상·목적이 예민하거나 확신이 없어 보임 | **가이드 모드** |

- 자동 모드라도 **기획서 요약 게이트는 반드시** 보여준다(SKILL.md Stage 1).
- **자료가 얇다고 자동 모드를 가이드로 강제하지 않는다.** 자동이면 기획서도 초안-후-게이트로 간다(가정은 "가정"에 표시).
- 모드는 도중에 바꿀 수 있다. 사용자가 "너무 빠르다/느리다" 하면 즉시 전환한다.

## 2. Stage 1 → Stage 2 (기획·설계 → 화면 구조)

Stage 1이 끝나면 아래를 정리해 Stage 2 시작 시 **그대로 채워 넣고, 해당 질문을 건너뛴다.**

| 넘기는 것 | Stage 2에서 대체하는 질문 | 비고 |
|---|---|---|
| 한 줄 정체성 | "누구를 위한 무엇인가요?" | 기획서 그대로 |
| 서비스 유형 | "어떤 종류의 서비스인가요?" | 유형별 기본기능 선택의 근거 |
| 핵심 기능 + 규칙·금지 | "가장 중요한 기능 한두 개는요?" | 규칙·금지는 화면 정의의 "규칙·조건/안 되는 것"으로 |
| 형태(웹/앱) · 유료 여부 | "웹인가요 앱인가요 / 돈을 받나요?" | 설계서의 종류·규모에서 가져옴 |
| 대상 사용자 | (백오피스 규모 판단에 참고) | 개인용이면 관리자 최소본 |
| 핵심 정보 덩어리(회원·주문 등) | (백오피스 짝맞춤에 참고) | 관리 대상 도출 근거 |

- 남는 질문이 거의 없을 때가 정상이다. **0개면 곧장 초안 생성**, 1개 남으면 그것만 짧게 묻는다.
- 기획서에 규칙·금지가 이미 있으면 IA 문서는 "화면에서 어떻게 동작하는지"에 집중하고 규칙은 기획서에서 인용한다(중복 서술 금지).

**Stage 1.5 스케치가 있었으면(옵트인) 함께 넘긴다:**

| 스케치가 뽑은 것 | 어디로 | 비고 |
|---|---|---|
| 여정에 필요한 **화면·기능 후보** | Stage 2(IA) 입력 | 스케치=흐름 → IA가 화면·기능 구조로 확정(드라이버) |
| **정책 성격 결정** | 정책 단계 입력 | 스케치엔 후보로만, 확정은 정책서(`POL.*`) |
| **데이터 성격 결정** | Stage 2.5(데이터) 입력 | 스케치엔 후보로만, 확정은 데이터 모델(`그룹.필드`) |

- Stage 1.5 시점엔 IA·데이터·정책이 아직 없으므로, 스케치의 결정은 **이미 있는 기획서엔 지금 옮기고, 나머지는 위 표대로 뒤 Stage 생성 입력으로** 넘긴다(스케치에만 남기지 않는다).

## 3. Stage 1·2 → Stage 3 (→ 디자인 컨셉)

Stage 3 시작 시 아래를 채워 넣고 해당 질문을 건너뛴다.

| 넘기는 것 | 출처 | Stage 3에서 대체/활용 |
|---|---|---|
| 한 줄 정체성 · 서비스 유형 | 기획서 | 1안(추천안) 방향 결정 근거 |
| 대상 사용자, 특히 **특별 대상**(노인·어린이·시각약자·외국어 등) | 기획서 / 사용자 정의 모듈 | 접근성 강화 여부. 특별 대상 있으면 대비·글자 크기·터치 영역 강화 |
| 이미 정해진 색 · 로고 · 이름 스타일 | 기획서 / **제품 원칙 모듈** | 있으면 주색으로. "이미 정해진 것" 질문 생략 |
| 제품 원칙(분위기·"쉬워야 한다"·접근성) | 제품 원칙 모듈 | 컨셉의 출발점. 원칙과 안 맞는 방향은 3안에서 뺀다 |
| IA의 실제 메뉴·라벨 | Stage 2 IA | **샘플 페이지에 사실감만** 준다. IA를 그대로 조립하지는 않는다(느낌 샘플) |

- 제품 원칙 모듈이 있었다면 그것이 **가장 강한 단서**다. easyproduct-design-concept은 그 한 줄 원칙을 색상 코드·글꼴·컴포넌트까지 확장하는 역할이므로, 원칙을 무시하고 새로 잡지 않는다.
- 가이드 모드면 분위기·좋아하는 사이트·밝게/어둡게 정도만 묻는다. 자동 모드면 이것도 추론해 "가정"에 남기고 3안으로 직행한다.

## 4. Stage 4: 설계서 back-fill 짝맞춤

Stage 1에서 비워 둔 설계서의 두 자리를 채운다.

| 설계서의 빈 자리 | 채우는 내용(출처) |
|---|---|
| `[이후 화면·기능 구조에서 채움]` (대략의 구성) | Stage 2 IA의 **화면 지도 + 주요 화면 이름** 요약. 전체 IA를 복사하지 말고, "자세한 건 화면·기능 구조 문서 참고"로 가리킨다 |
| `[이후 디자인 컨셉에서 채움]` (겉모습·느낌) | Stage 3 **확정 1안 요약**: 브랜드 성격 한 줄 + 주색(hex) + 제목/본문 글꼴 + 톤 한 줄. "자세한 건 디자인 컨셉 문서 참고" |

- 설계서는 **가리키는 요약**만 갖는다(중복 방지). 진짜 내용은 각 전용 문서에 있다.
- 상호 링크: 네 문서 모두 머리에 "이 세트의 다른 문서: 기획서 · 설계서 · 화면 구조 · 디자인 컨셉" 한 줄을 넣는다.

## 5. 재진입 판단 (이미 만든 문서가 있을 때)

Step 1에서 가져온 자료가 우리 산출물(또는 유사 형식)이면, **어느 단계 산출물인지** 가려 시작점을 정한다.

| 가져온 것 | 시작점 |
|---|---|
| 아무것도 없음 / 메모만 | Stage 1부터 (기획서 + 설계서) |
| 기획서만 (설계서 없음) | 기획서 재사용 + **설계서 생성**(Stage 1의 Phase B) → Stage 2 → Stage 3 |
| 기획서 + 설계서 | 둘 재사용 → Stage 2부터 |
| 기획서 + 화면 구조 | Stage 3부터 (설계서 없으면 Phase B로 먼저 채움) |
| 셋 다 있음 | 연결·마무리(Stage 4)만: 상호 링크·back-fill·안내 문서가 빠졌으면 그것만 보완 |

- **핵심: 재진입은 "단계"가 아니라 "문서" 단위로 없는 것만 채운다.** 설계서는 Stage 1의 산출물이라
  기획서만 있는 경우 아직 없는 상태다 — 반드시 만들어 세트에 포함한다. 세트에 설계서가 빠진 채 끝나면 안 된다.

- 기준선 문서는 **함부로 바꾸지 않는다.** 얇은 곳만 보강하고, 무엇을 보강했는지 끝에 한 줄로 알린다.

## 6. 놓치기 쉬운 함정

- **도입을 세 번 하지 마라.** 각 스킬 SKILL.md의 Step 0/자료 요청은 Stage에서 실행하지 않는다. 이미 Step 0에서 한 번 했다.
- **설계서를 너무 일찍 확정하지 마라.** IA·디자인이 채울 두 자리는 Stage 4까지 `[…에서 채움]`으로 둔다.
- **디자인에서 선택을 기다리며 멈추지 마라.** 3안을 보여주되 1안으로 이미 확정해 둔다(easyproduct-design-concept의 무중단 원칙).
- **백오피스를 빠뜨리지 마라.** IA 초안은 프론트만 있으면 미완성이다. 회원·결제·문의가 생기면 관리 화면이 따라온다.
- **파일로 만들어라.** IA·디자인 컨셉은 사용자가 요청하지 않아도 항상 파일로. 마지막 안내 문서도 파일로.
- **개별 스킬 트리거를 훔치지 마라.** 사용자가 "디자인만" 원하면 이 스킬로 끌고 오지 말고 그 단계만 하거나 개별 스킬에 맡긴다.
- **재진입에서 설계서를 빠뜨리지 마라.** 기획서만 가져왔으면 설계서는 아직 없다 — Phase B로 만들어 세트에 넣는다.
- **PNG를 미리 포기하지 마라.** "이 환경은 안 될 것"이라 넘겨짚지 말고 먼저 렌더를 시도하고, 도구가 정말 없을 때만 HTML로 대체한다.
- **유저 스토리·유스케이스는 세트에서 기본 켜짐.** 자동 생성이 기본이고, 사용자가 "빼줘"로 끈 것만 뺀다. 분량은 개별 지정 > 전체 "자세히" > 적당히(디폴트) 순. 자동 모드에서 분량을 물어보느라 멈추지 않는다.
- **정책서·약관도 세트에서 기본 켜짐**(재료는 기획서+IA, 디자인과 무관). "정책서만/약관만"이면 그것만, "빼줘"면 건너뛴다. **약관·개인정보처리방침의 전문가 검토 경고와 지어내지 않기 원칙은 절대 빼지 마라** — 무중단이라고 완성본처럼 포장하면 안 된다.
- **데이터 모델은 최초 생성까지만 기본, 필드 성장은 화면 설계에서.** Stage 2.5에서 다 채우려 하지 마라.
- **화면 설계·spec-kit 인계는 옵트인.** 자동 모드라도 게이트(3.7)에서 물어 진행한다. 묻지 않고 자동으로 밀지 마라.
- **디자인 확정은 게이트(3.7)에서.** 디자인 스킬(Stage 3)은 무중단으로 1안을 걸어만 두고, 확정은 화면 설계 직전 게이트에서 사용자에게 받는다.

## 7. Stage 2 → Stage 2.5 (IA·기획 → 데이터 모델)

`easyproduct-data-model`은 **기획서에서 정한 정보 덩어리(그룹)**를 받아 최초 데이터 모델을 만든다.

| 넘기는 것 | 출처 | 데이터 모델에서 쓰임 |
|---|---|---|
| 정보 덩어리: 그룹명(영문)·한글명·설명 | 기획서 "기억해 둬야 할 정보" | 각 그룹에 프리셋(공통 관리 필드+종류별 표준 필드)을 깔 근거 |
| 서비스 유형·유료 여부 | 기획서/설계서 | 어떤 표준 그룹(user·payment·order/deal 등)이 켜질지 판단 |

- doc-builder가 그룹을 **그룹 수준으로만** 넘긴다(필드 없음). 데이터 모델이 프리셋으로 필드를 깐다.
- **"어느 화면에서 쓰이나"는 데이터 모델이 담지 않는다.** 그 관계의 원본은 화면 설계서
  (`screendesign.screens`의 `data`)이므로, IA의 화면 라벨을 데이터 모델로 넘기지 않는다.
  필요하면 화면 설계서에서 유도해 답한다(데이터 모델에 역방향으로 또 적으면 이중 기입이 된다).
- 산출물은 **`ssot/data-model.md` 한 파일**(이 문서의 표가 원본·SSOT). 여기서는 최초 생성까지만.
  (요청 시 그 시점의 읽기 전용 json 스냅샷을 추출할 수 있으나, 상시 산출물은 아니다.)
- "데이터 모델 빼줘"면 건너뛰되, **화면 설계를 하려면 데이터 모델이 반드시 필요**하므로 그때 만들어야 함을 알린다.

## 8. Stage 3.7 디자인 확정·리뷰 게이트

화면 설계로 넘어가기 직전의 확정·검토 지점. 세 가지를 한다.
1. **디자인 1안 확정** — 무중단으로 걸어둔 1안을 사용자에게 확인. 안 번호 바꾸면 `easyproduct-design-concept` 3.5-(가)(선택 표시만), 새 방향이면 3.5-(나)(3안 새로).
2. **문서 훑어 컨펌** — 기획서·설계서·IA·데이터 모델·디자인·정책·약관 요약 제시 후 "고칠 것 있나" 확인.
3. **개별 수정 라우팅** — 고칠 문서를 그 스킬의 수정 모드로 재진입시킨다:

| 고칠 문서 | 라우팅 대상(모드) |
|---|---|
| 기획서·설계서 | `easyproduct-doc-builder` (풍부화/재진입 — 지정 부분만) |
| 화면 구조(IA) | `easyproduct-ia-designer` (검토+수정 — 바뀐 부분만) |
| 데이터 모델 | `easyproduct-data-model` (수정 — 데이터 모델 문서를 직접 고침) |
| 디자인 컨셉 | `easyproduct-design-concept` (안 번호=3.5-가 / 새 방향=3.5-나) |
| 정책서·약관 | `easyproduct-policy-legal` (**Step 5 재진입** — 지정 조항만, `#n`·검토 경고·1·2·3부 구조 보존) |

- 수정이 앞 문서의 근거를 바꾸면 **영향 받는 뒤 문서도** 해당 스킬로 다시 손본다. 고쳤으면 Stage 4에서 back-fill·상호 링크를 다시 맞춘다.
- 게이트 끝에 **옵트인 두 개**를 묻는다: "화면 설계서까지 만들까요?"(→3.8) / (마지막에)"spec-kit 인계 안내문 만들까요?"(→Stage 5).

## 9. Stage 3.8 화면 설계로 넘기는 것 (+ 데이터 모델·UI 인벤토리 성장, + 범위 분리)

`easyproduct-screen-design`은 **재료가 이미 다 있다**. 도입·자료요청 없이 아래를 넘긴다.
**이 스킬은 한 번에 한 범위(사용자 앱 또는 백오피스)만 다루므로, 호출할 때마다 범위를 명시한다.**
**이 스킬만은 "초안 우선"이 아니다** — 화면설계서는 곧 구현 명세이므로 처음부터 컴포넌트·배치·
문구가 확정된 상태로 만든다(리듬은 메인 컨펌→나머지 진행으로 유지, 대충 만들고 나중에 다듬지 않음).

### 3.8-A (사용자 앱, 기본)
| 넘기는 것 | 출처 | 화면 설계에서 쓰임 |
|---|---|---|
| 화면 목록·화면 정의(발판, FEAT ID·사이트맵 순서) — **사용자(프론트) 화면만** | Stage 2 IA | 도메인별 파일 분리·순서의 기준. 전 화면을 풍부하고 정확하게 살 붙일 최소 스펙 |
| **네비게이션 구조**(항목 → 목적지 FEAT ID → 노출 조건, GNB/Footer 구성) | Stage 2 IA(`references/네비게이션-설계-원칙.md` 기준) | **공통 UI 정의**가 이를 재료로 실제 배치·스티키·모바일 접힘·반응형으로 구체화(메인 컨펌 때 확정) |
| **확정** 디자인 컨셉(색·글꼴·컴포넌트·여백 척도·타이포 위계) | Stage 3.7에서 확정된 안 | **UI 컴포넌트 인벤토리**(신규)와 공통 UI 정의·목업에 그대로 입힘. 화면 정의의 간격·텍스트 위계는 이 척도·위계를 참조(새 숫자 안 지음) |
| 데이터 모델 문서 | Stage 2.5(+수정본) | 화면 데이터 정의가 변수를 **직접 읽어 참조** |
| 기획서 규칙·금지 | Stage 1 | 화면 정의의 규칙·조건으로 인용(중복 서술 금지) |

### 3.8-B (백오피스, 옵트인 — 3.8-A 완료 후 별도로 물어 별도 호출)
| 넘기는 것 | 출처 | 화면 설계에서 쓰임 |
|---|---|---|
| 화면 목록·화면 정의(발판, FEAT ID·사이트맵 순서) — **관리자(백오피스) 화면만** | Stage 2 IA | 3.8-A와 별개로 처음부터 진행 |
| **네비게이션 구조**(LNB 그룹·역할별 노출, 대시보드→목록→상세→탭·액션 계층) | Stage 2 IA(백오피스 몫) | **공통 UI 정의(백오피스용)**가 배치·스티키·접힘으로 구체화 — 3.8-A와 같은 원리를 LNB 중심으로 적용 |
| **확정** 디자인 컨셉(같은 안) | Stage 3.7에서 확정된 안 | **절제된 변형**(채도 낮춤·그림자/라운드 축소·밀도 향상·강조색 절제)을 적용해 관리자용 공통 UI·**UI 컴포넌트 인벤토리(별도 파일)**에 입힘 |
| 데이터 모델 문서 (3.8-A와 동일 문서, 공유) | Stage 2.5(+수정본) | 같은 SSOT를 참조. 백오피스 전용으로 따로 만들지 않는다 |
| 기획서 규칙·금지 | Stage 1 | 동일 |

메인/기준 화면과 공통 UI 정의(네비게이션 등)·UI 컴포넌트 인벤토리는 두 범위가 **각자 따로**
갖는다(사용자 앱=GNB/탭바, 백오피스=LNB 좌측메뉴). 산출물도 완전히 분리한다:
- 사용자 앱: `screens/user/screen-design-user-index.md` + 도메인별 파일 + `ssot/ui-components-user.md`
- 백오피스: `screens/backoffice/screen-design-backoffice-index.md` + 도메인별 파일 + `ssot/ui-components-backoffice.md`
한 파일·한 인벤토리에 합치지 않는다. Stage 4 색인에도 각각 별도 항목으로 올린다.

- **데이터 성장 루프**(두 범위 공통): 화면을 그리다 새 데이터가 필요하면, 화면 설계 스킬이 직접 만들지 않고 `easyproduct-data-model`에
  **의도(덩어리 후보·뜻·종류·쓰는 화면)**를 넘겨 확정 변수(구조체)를 회신받아 참조한다. 이미 있는 변수는 데이터 모델 문서 직접 읽기.
- **UI 컴포넌트 인벤토리 성장 루프**(두 범위 공통, 각자 자신의 인벤토리에): 새 컴포넌트가 필요하면 먼저 그 범위의
  인벤토리를 조회해 재사용 가능한지 확인하고, 없으면 등록한다. 재사용 단위는 중앙 인벤토리에, 특정 화면 전용은
  그 도메인 파일 안에 `UI.FEAT.도메인.기능명[.화면종류].컴포넌트명` 형식의 로컬 컴포넌트로 둔다.
  **배치(3~4개) 단위로 모아** 등록해 왕복을 줄인다. 새 그룹이 필요하면 데이터 모델 스킬이 사용자 컨펌을 받는다.
- 메인 화면 풀세트 컨펌(+공통 UI 정의·화면 전역 정책 확정) → 나머지 글정의+데이터 일괄 → 목업은 선택적.

## 10. Stage 5 spec-kit 인계 안내문

`assets/speckit-handoff-template.md`를 채워 인계장을 만든다. **spec-kit 명령을 지정하지 않는다** — 문서 소개 +
어디를 보면 되는지 + **문서별 SSOT·보강자료·참조용 구분표** + 진행 시 지킬 것만 넘기고, 명령·단계는 spec-kit 위
AI가 판단하게 둔다. **이번에 실제로 만든 문서만** 표에 남긴다.

이 구분표는 단순 이분류(SSOT/참조용)가 아니라 **세 갈래**다 — SSOT(결정이 사는 곳) / 보강자료(다른 SSOT를
검증 가능한 형태로 가공, 원본과 다르면 원본 우선이지만 spec-kit이 실제로 참조해야 함 — 유저 스토리·유스케이스가
여기 해당) / 참조용(파생물·임시 참고자료, 표준 아님). 그리고 **한 스킬 안에서도 원본/파생물이 갈리는 경우**를
정확히 반영한다:
- 데이터 모델: 문서(표) 자체가 SSOT(기획서보다 우선, 데이터 구조에 한해). 상시 산출물은 이 한 파일뿐이며,
  요청 시 뽑는 json 스냅샷은 읽기 전용 사본일 뿐 SSOT가 아니다(표에 올리지 않는다).
- 화면 설계서: 글 정의=SSOT(있으면 IA 화면정의보다 상세 기준, 범위는 IA 안), HTML 목업=참조용
- 디자인 컨셉: 문서의 "선택된 안" 값=SSOT, 3안 비교·키트·샘플(HTML·PNG)=참조용(임시 참고물)
- 유저 스토리·유스케이스는 기획서의 **보강자료**(참조용이 아님) — 기획서와 다르면 기획서가 우선하되, spec-kit이 수용 기준·시나리오를 세울 때 실제로 참조해야 할 자료
- 약관·개인정보처리방침은 이 축과 별개로 **"초안 상태"**(전문가 검토 전 확정 아님)를 표 아래 별도 명시

**화면 설계서가 사용자 앱·백오피스 둘 다 있으면 인계장도 프로젝트별로 두 개** 만든다(둘이 별개
프로젝트로 개발될 가능성이 높기 때문 — 3.8-A/B 참고). 각 인계장은:
- 자기 프로젝트에 해당하는 화면 설계서만 문서 목록에 넣는다(반대쪽 화면 설계서는 뺀다).
- 기획서·설계서·IA·디자인 컨셉·정책서·약관 등 공통 문서는 양쪽에 모두 넣는다.
- **데이터 모델은 양쪽에 모두 넣되, "반대쪽 프로젝트와 공유하는 SSOT"임을 명시**한다 — 한쪽에서
  구조를 바꾸면 반대쪽에도 영향이 간다는 경고를 포함한다.
- 화면 설계서가 하나뿐이면(백오피스를 안 만들었으면) 인계장도 하나만 만든다.

## 11. Stage 3.8-C 기능 관계도로 넘기는 것

`easyproduct-ia-designer`의 "선택 모듈: 기능 관계도"에 넘긴다. 재료는 **Stage 2에서 이미 만든 IA 문서
하나뿐**이다 — 도입·자료 요청 없이 그 문서를 그대로 넘긴다.

| 넘기는 것 | 출처 | 관계도에서 쓰임 |
|---|---|---|
| IA 문서(화면 지도·기능 목록, FEAT ID 포함) | Stage 2 | 트리거 관계·노드(FEAT ID)를 그대로 도출 |

- 이 모듈은 **읽기 전용 파생**이다. IA 문서를 바꾸지 않는다. IA가 나중에 바뀌면 관계도는 다시
  도출하면 된다는 것을 산출물에 명시한다(easyproduct-data-model의 json 추출과 같은 성격).
- 세트에서는 **프론트↔백오피스 연동(트리거) 관계도만** 기본으로 만든다. UI 흐름(화면 전이) 관계도는
  화면 설계서의 "이어지는 화면" 항목과 역할이 겹치므로 세트 기본에는 안 넣는다(사용자가 따로 요청하면
  만든다).
- 화면 설계(Stage 3.8) 자체를 안 했으면 이 옵트인도 묻지 않는다 — 화면이 사용자·백오피스로 갈라져
  실제로 존재할 때 연결 고리를 보여주는 그림이 의미가 크기 때문이다.
- 산출물은 IA와 별도 파일(`reference/feature-relation-map.md`)이며, 색인·spec-kit 인계장 모두에서
  **참조용(파생물)**으로 분류한다.
