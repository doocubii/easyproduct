# easyproduct-ia-designer

> 개발 지식이 없어도 몇 가지 질문에만 답하면, 서비스의 화면·기능 구조(IA)를 통째로 초안으로 잡아주는 스킬입니다. 서비스 유형만 알면 가입/로그인·FAQ 같은 기본 기능을 자동으로 채우고, 사용자 화면(프론트)과 관리자 화면(백오피스)을 함께 설계합니다.

[easyproduct 스킬 세트](../../README.md)의 일부입니다.

## 무엇을 하나
- 초반에 꼭 필요한 3~4가지만 묻고, 나머지는 AI가 추론해 **전체 초안을 한 번에** 만듭니다.
- 서비스 유형(커뮤니티·쇼핑몰·예약·SaaS 등)에 맞춰 **당연히 있어야 할 기본 기능을 자동으로** 채웁니다.
- 사용자가 쓰는 화면만이 아니라, 그걸 운영·관리하는 **관리자(백오피스) 화면도 함께** 설계합니다.
- 모든 기능에 `FEAT.도메인.기능명` 형식의 **고유 ID를 처음으로 확정**해, 뒤 단계가 이름이 아니라 ID로 정확히 참조하게 합니다.
- 자동으로 넣은 것·가정한 것·아직 안 정한 것을 따로 표시해, 사용자가 **보고 쉽게 빼거나 더할 수 있게** 합니다.

## 언제 쓰나
- "화면 구조", "메뉴 구조", "사이트맵", "정보 구조", "페이지 구성"을 짜달라고 할 때.
- "어떤 화면들이 필요하지?", "관리자 페이지는 뭐가 있어야 하지?"처럼 화면·기능 구성이 막막할 때.
- 아이디어나 기능 목록은 있는데 "이걸 어떤 화면·기능으로 만들지" 감이 안 올 때.
- 기획서를 만든 뒤 화면·기능 구조로 넘어가려 할 때.

## 산출물
- **화면·기능 구조 문서** — 기능 목록(프론트·백오피스)·사이트맵·화면 정의·네비게이션·핵심 흐름을 담은 파일(기본 `.md`, 원하면 `.docx`). 항상 파일로 만듭니다.
- (선택) **유스케이스 문서** — 요청 시에만. "사용자가 이루려는 목표" 단위의 흐름 정리.
- (선택) **기능 관계도 문서** — 요청 시에만. IA 문서에서 도출한 UI 흐름 관계도 + 프론트↔백오피스 연동 관계도(별도 파일).

## 단독 실행 vs 세트
- 기본은 **독립 실행**입니다. 이 스킬만으로도 초안을 만들 수 있습니다.
- easy-dev-doc-builder(기획서)가 함께 있으면 이어 씁니다. 기획서가 있으면 먼저 읽어 유형·핵심 기능을 뽑고 그 질문은 건너뜁니다. 아이디어가 너무 막연하면 기획서를 먼저 만들도록 권할 수 있습니다.
- 여기서 만든 화면·기능 구조는 설계서의 "대략의 구성" 자리에 그대로 들어갑니다.

## 구성 파일
- `SKILL.md` — 스킬 본문(진행 절차·핵심 원칙·기능 ID 규칙·백오피스 설계 규칙 등).
- `assets/화면구조-템플릿.md` — 최종 산출물의 읽기 쉬운 구조(이대로 채웁니다).
- `assets/유스케이스-템플릿.md` — (선택 모듈) 유스케이스 문서 구조. 요청 시에만.
- `assets/기능관계도-템플릿.md` — (선택 모듈) 기능 관계도 문서 구조. 요청 시에만.
- `references/유형별-기본기능.md` — 유형별 자동 기본기능 + 백오피스 짝맞춤 + 화면 도출·네비게이션 힌트.
- `references/네비게이션-설계-원칙.md` — 메뉴·이동 규칙 설계 원칙(상태·역할별 메뉴, GNB/Footer, 백오피스 LNB).

## 세트 안에서의 위치
- 기획서(easy-dev-doc-builder) → **IA(이 스킬)** → 화면 설계서로 이어지는 중간 단계입니다.
- 기능 ID(`FEAT.도메인.기능명`)를 **이 스킬이 최초로 확정**하고, 화면 설계서 등 뒤 단계는 이 ID를 이어받아 인용만 합니다.
- 유스케이스·기능 관계도는 이 IA 문서를 재료로 삼는 파생 산출물이며, 더 큰 사용자 여정은 `easyproduct-scenario`가 맡습니다.
