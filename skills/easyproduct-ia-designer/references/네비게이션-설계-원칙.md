# 네비게이션 설계 원칙 (IA용)

> 이 문서는 IA(화면·기능 구조)가 메뉴·이동 규칙을 정의할 때 지키는 원칙이다. 특정 서비스 유형에
> 종속되지 않되, 유형에 따라 해당 사항만 적용한다.
> 실제 배치·스티키·반응형·로딩/빈/권한 상태는 여기서 다루지 않는다 — 그건 화면 설계서의
> "공통 UI 정의" 몫이다.

## 관심사 분리
- IA가 정의하는 것: 메뉴 항목·목적지(FEAT ID)·노출 조건·계층·이동 규칙.
- IA가 정의하지 않는 것(화면 설계서 몫): 좌/중/우 배치, 고정(스티키), 접힘(반응형), 로딩·빈·권한 상태.
- 판별: "좌/중/우·고정·접힘·로딩/빈 상태"라는 말이 나오면 화면 설계서로 넘긴다.

## 홈과 계층
- **로그인이 필수인 서비스는** 첫 화면이 로그인 상태로 갈린다: 로그아웃용 진입(공개) 화면과
  로그인 후 홈을 구분한다. (게스트 열람이 되는 서비스는 해당 없음 — 특정 행동에서만 로그인 유도)
- 콘텐츠가 계층적인 서비스는 공개 홈 → 서브 홈(카테고리/소개) → 기능·상세로 내려가게 한다.

## GNB vs Footer
- 핵심 기능·자주 쓰는 메뉴는 GNB(상단)에 이름으로 직접 노출한다.
- Footer엔 덜 중요한 보조·정보성만: 고객센터·FAQ·공지·약관·개인정보처리방침·사업자 정보 등.
- 원칙적으로 핵심 기능을 GNB·Footer에 이중 노출하지 않는다. 단, 고객센터처럼 접근성 목적의
  의도적 중복은 예외로 허용한다.
- 판별: "매일·자주" → GNB / "가끔·필요할 때만" → Footer.

## 상태·역할별 메뉴 구성
- 역할이 여럿인 서비스는 메뉴를 로그인 상태·계정/컨텍스트·역할별로 갈라 각각 정의한다.
  (역할이 하나뿐인 개인용 서비스는 해당 없음)
- 형식은 반드시 **"항목 → 목적지(FEAT ID) → 노출 조건"**으로 적는다. 목적지는 이름이 아니라
  그 화면이 속한 기능의 FEAT ID로 가리킨다 (예: "마이페이지 → `FEAT.user.mypage` → 로그인 시").

## 이동 규칙
- 로고 = 홈(상태별). 대표 항목명 = 해당 소개/진입 화면. 브레드크럼으로 상위 단계 복귀.
- 데이터가 많은 목록 화면(관리자 등)은 전역 검색으로 목록을 건너뛰고 상세로 직진입하게 한다.
  데이터가 적은 화면엔 불필요하다.

## 프론트 · 백오피스는 대칭으로 각자 정의
- 사용자 화면과 관리자 화면은 각각 계층·헤더·메뉴 구성을 정의한다(한쪽만 하면 미완성).
- 프론트: GNB 중심 계층 + Footer.
- 백오피스: LNB(좌측 메뉴) 중심, 메뉴는 그룹화 + 역할별 노출, 계층은
  대시보드(홈) → 목록 → 상세 → 상세 내 탭·액션. Footer 대신 필요하면 유틸리티 바(버전·환경·
  도움말)를 둘 수 있다(필수 아님). 대외 표시(사업자 정보 등)는 프론트가 담당하므로 반복하지 않는다.

## 체크리스트
- [ ] 계층(홈→서브홈→기능)을 그렸는가 (해당 시)
- [ ] GNB 메뉴를 상태·역할별 "항목 → 목적지(FEAT ID) → 노출조건"으로 적었는가
- [ ] 핵심 기능을 GNB에 두고 Footer와 불필요하게 중복하지 않았는가
- [ ] Footer엔 보조·정보성만 두었는가
- [ ] 이동 규칙(로고=홈·브레드크럼)을 적었는가
- [ ] 프론트·백오피스를 각각 정의했는가(백오피스 LNB 그룹·역할 노출 포함)
- [ ] 배치·스티키·반응형·상태를 화면 설계서로 넘기고 여기선 안 다뤘는가
