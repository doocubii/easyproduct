# 화면·기능 구조(IA) 짜기

## 목적

기술 지식이 없는 사람이 **핵심 몇 가지만 답하면**, AI가 서비스의
**전체 화면·기능 구조를 초안으로 통째로 잡아준다.** 그 다음 사용자가 고친다.

두 가지가 이 스킬의 핵심이다.
1. **유형 기반 자동 생성** — 서비스 종류만 알면 가입/로그인, FAQ, 이용약관, 마이페이지처럼
   "이런 서비스면 당연히 있어야 하는 기능"을 사용자가 말하지 않아도 자동으로 채운다.
2. **프론트 + 백오피스 동시 설계** — 사용자가 쓰는 화면만이 아니라, 그걸 운영·관리하는
   **관리자(백오피스) 화면도 함께** 설계한다. 회원이 가입하면 회원 관리가, 결제가 있으면 정산이 따라온다.

## 일하는 방식 — 초안 우선 (중요)

**매번 물어보고 컨펌받으며 진행하지 않는다.** 그러면 사용자가 너무 오래 붙잡힌다.
대신 이렇게 한다:

```
초반에 꼭 필요한 3~4가지만 묻는다
        ↓
AI가 나머지를 전부 추론해 전체 초안을 만든다 (유형 기본기능 + 백오피스 포함)
        ↓
완성된 초안을 한 번에 보여주고 "빼거나 더하고 싶은 것"을 받아 고친다
```

- 중간에 하나하나 확인받지 않는다. **모르는 건 합리적으로 가정해 채우고, 무엇을 가정했는지 문서에 표시**한다.
- "이것도 넣을까요?"를 반복하지 않는다. **일단 넣고 "자동으로 넣은 것"으로 표시**해 나중에 빼게 한다.
- 초안이 완벽할 필요는 없다. 사용자가 보고 고칠 수 있는 **출발점**을 빠르게 주는 게 목표다.

## 핵심 원칙

1. **비전문가 언어.** 전문 용어를 먼저 꺼내지 않는다. 불가피하면 한 줄로 풀어 쓴다. ("사이트맵 = 화면 지도")
2. **읽기 쉬운 산출물.** 기능·화면을 평범한 목록과 문장으로. 어려운 표기법을 쓰지 않는다.
3. **최소 질문.** 초반 질문은 3~4개를 넘기지 않는다.
4. **자동으로 채우고 표시한다.** 유형 기본기능·백오피스는 자동 생성하고 "자동으로 넣은 것"에 모아 둔다.
5. **빼기 쉽게.** 사용자가 지우기 쉬운 형태로 준다. 더하는 것보다 빼는 게 편하도록 넉넉히 담는다.
6. **자세히 쓴다.** 초안은 나중에 구현의 근거가 된다. 특히 화면 정의는 만들 사람(또는 AI)이 읽고 그대로 만들 수 있을 만큼 구체적으로. 자세히 쓰는 비용은 사용자가 아니라 AI가 지므로 아끼지 않는다.
7. **기능마다 고유 ID를 붙인다.** 모든 기능에 `FEAT.도메인.기능명` 형식의 ID를 부여해, 뒤 단계(화면 설계서 등)가 이름이 아니라 ID로 정확히 참조하게 한다. 형식과 원칙은 아래 "기능 ID 규칙"을 따른다.

## 기능 ID 규칙 (`FEAT.도메인.기능명`)

기획서 → IA → 화면 설계서로 이어질 때, 기능·화면을 이름으로만 가리키면 서로 다른 도메인의
같은 이름 기능(예: "멤버 찾기"와 "상품 찾기")이 문서 사이에서 헷갈린다. 그래서 이 스킬이
**각 기능에 고유 ID를 최초로 확정**한다.

**형식**
- 기능 ID: `FEAT.도메인.기능명`
  - `도메인`·`기능명`은 **영문 소문자 camelCase**(데이터 모델의 그룹명·필드명과 같은 스타일).
  - 예: `FEAT.member.search`(멤버 찾기), `FEAT.product.search`(상품 찾기) — 이름이 같아도 도메인으로 완전히 구별된다.
  - `FEAT`는 **대문자로 고정**한다. 데이터 모델의 그룹명은 항상 소문자이므로, 대문자 `FEAT`는 "이건 기능/화면 ID다"를 한눈에 구분되게 하는 장치다. 별도 예약어 규칙 없이 대소문자만으로 데이터 ID와 절대 겹치지 않는다.
  - **산문 안에서는 항상 백틱으로 감싼다**: `` `FEAT.member.search` ``.
- 도메인 잡는 법: 그 기능이 다루는 대상을 도메인으로. 프론트·백오피스가 같은 대상을 다루면 같은 도메인을 쓰고 기능명으로 가른다(예: `FEAT.member.search` / `FEAT.member.manage`). 특정 대상이 없는 공통·관리 기능은 `auth`(로그인 등)·`admin`(대시보드·권한 등)·`system` 같은 도메인을 쓴다(예: `FEAT.auth.login`, `FEAT.admin.dashboard`).

**원칙**
- **최초 확정은 여기서.** 기획서는 이름만 가지고 있다. 사용자가 기획서에서 말한 핵심 기능을 IA로 펼칠 때, 그 기능에 **처음으로 ID를 매기는 것이 이 스킬**이다.
- **모든 기능에 부여.** 프론트·백오피스 기능 전부, 자동으로 넣은 기본 기능(가입/로그인 등)에도 부여한다(예: `FEAT.auth.login`).
- **ID는 이 스킬만 바꾼다.** 한 번 정하면 화면 설계서 등 뒤 단계는 이 ID를 **이어받아 인용만** 하고 새로 만들거나 바꾸지 않는다. (데이터 모델의 변수명을 그 스킬만 짓는 것과 같은 원칙.)
- ID는 개발이 참조하는 안정된 식별자다. 사용자에게 보이는 **이름(라벨)은 자유롭게 바꿔도** ID는 유지한다.

**표기 예시 (기능 목록)**
- `FEAT.member.search` — 멤버 찾기
- `FEAT.product.search` — 상품 찾기

**하지 말 것**
- ID 없이 기능을 이름만으로 나열하지 말 것.
- 도메인을 생략하고 기능명만으로 ID를 짓지 말 것(그러면 다시 "찾기" 충돌이 재현된다).
- 화면종류(list/detail/write/edit) 세그먼트를 이 스킬에서 미리 채우려 하지 말 것 — 그건 화면 설계서 스킬이 화면 단위로 파생시키는 몫이다. **IA는 기능 ID(`FEAT.도메인.기능명`, 3마디)까지만 확정한다.**

## easy-dev-doc-builder와의 관계

기본은 **독립 실행**이다. 하지만 같이 설치돼 있으면 서로 이어 쓴다.
- **입력으로**: 사용자가 이미 만든 기획서(무엇을·왜)가 있으면 먼저 읽어, 한 줄 정체성·핵심 기능·유형을 거기서 뽑고 그 질문들은 건너뛴다.
- **아이디어가 너무 막연하면**: "무엇을 만들지"조차 안 잡혔으면, easy-dev-doc-builder로 기획서를 먼저 만드는 걸 권할 수 있다. (다만 이 스킬만으로도 초안은 만들 수 있다 — 그게 초안 우선의 취지다.)
- **출력으로**: 이 문서의 화면·기능 구조는 설계서의 "대략의 구성" 자리에 그대로 들어간다.

## Step 0: 시작 (짧게)

- "몇 가지만 여쭤보고, 화면과 기능 구조를 제가 먼저 통째로 초안으로 잡아드릴게요. 그다음 마음껏 고치시면 됩니다."
- **기획서·메모·자료가 있으면 먼저 달라고 한다.** 있으면 읽고 유형·핵심 기능을 뽑아 아래 질문 중 답이 되는 건 건너뛴다.

## Step 1: 핵심 질문 (3~4개만)

한 번에 하나씩, 짧게. 자료에서 이미 답이 나온 건 묻지 않는다.

1. **유형** — "어떤 종류의 서비스인가요?" 감을 못 잡으면 예시를 든다: "커뮤니티/게시판, 쇼핑몰, 예약, 업무 도구, 콘텐츠·미디어, 중고·매칭 거래, 소셜, 소개 사이트, 온라인 강의, 모임·이벤트, 배달·호출, 개인 기록·트래커 중 가까운 게 있을까요?" → `references/유형별-기본기능.md`의 유형 선택에 쓴다.
2. **한 줄 정체성** — "한 문장으로, 누구를 위한 무엇인가요?" (기획서가 있으면 생략)
3. **핵심 기능 1~2개** — "이 서비스를 이 서비스답게 만드는, 가장 중요한 기능 한두 개는요?" (나머지 기능은 자동으로 채우니 여기서 다 받을 필요 없다)
4. **형태 & 돈** — "웹사이트인가요, 폰 앱인가요?" / "돈을 받는 서비스인가요?" (결제·정산 기능 포함 여부를 가른다)

> 관리자(백오피스)는 **따로 묻지 않는다.** 유형과 프론트 기능에서 자동으로 도출한다.
> 유형이 애매하거나 여러 개에 걸치면, 가장 가까운 것 하나를 고르고 나중에 확인 문장 하나로만 되짚는다.

## Step 2: 전체 초안 자동 생성 (묻지 않고 채운다)

`references/유형별-기본기능.md`를 열어 다음을 조립한다. `assets/화면구조-템플릿.md`를 채운다.

1. **기능 목록(프론트)** = 사용자가 말한 **핵심 기능** + 유형 기능 + 공통 기능(가입/로그인·마이페이지·검색·고객센터·FAQ·약관 등). 자동으로 넣은 것은 "기본(자동)"으로 구분. **각 기능 항목에 `FEAT.도메인.기능명` ID를 부여**한다(기본 기능도 예외 없이).
2. **기능 목록(백오피스)** = 프론트에서 생긴 것마다 짝을 맞춰 관리자 기능을 도출(회원 관리·콘텐츠/주문/예약 관리·문의 관리·안내글 편집·통계·권한, 결제 있으면 정산·환불). 짝맞춤 표는 참고 파일에 있다. **백오피스 기능에도 ID를 부여**한다(예: `FEAT.admin.member`, `FEAT.admin.settlement`).
3. **화면 지도(사이트맵)** — 사용자 화면과 관리자 화면을 나눠, 큰 묶음으로. 목록형 기능은 "목록 → 상세 → 쓰기/편집"으로 갈린다. **각 화면 옆에 그 화면이 대응하는 기능 ID를 함께 적는다**(예: `찾기 화면 → `FEAT.member.search``).
4. **화면 정의** — 이 문서의 알맹이다. 주요 화면(핵심 화면 + 관리자 주요 화면)마다 **자세히** 적는다: **대응 기능(ID)** / **로그인 필요** / 목적 / 진입 경로 / 화면에 보이는 것 / 여기서 할 수 있는 일(동작→결과) / 상태·예외(빈 상태·오류·권한 없음) / 이어지는 화면. 한 화면이 여러 기능을 담으면 ID를 여러 개 적는다. **"로그인 필요"는 `필수` / `무관(콘텐츠 분리)` / `무관(동일)` 중 하나로 자동 판단해 채운다**(개인 소유물이면 필수, 공개+개인화 혼합이면 콘텐츠 분리, 완전 공개면 동일). 이는 네비게이션의 "메뉴 노출 조건"과는 다른 문제이며, 자동으로 채운 값은 "자동으로 넣은 것"에 표시한다. **나중에 이걸 보고 구현할 사람(또는 AI)이 화면을 그릴 수 있을 만큼 구체적으로** 쓴다. 초안 우선 방식이라 설명을 길게 쓰는 비용은 사용자가 아니라 AI가 진다 — 아끼지 말고 채운다.
5. **네비게이션** — `references/네비게이션-설계-원칙.md`를 **반드시 참고**해, 메뉴·이동 규칙을 상태·역할별로 **"항목 → 목적지(`FEAT` ID) → 노출 조건"** 형식으로 정의한다. 프론트(GNB + Footer)와 관리자(LNB 그룹·역할별 노출)를 **각각** 적는다. 배치·고정·반응형·상태는 여기서 다루지 않는다(화면 설계서 몫).
6. **핵심 흐름** — 가장 중요한 작업 2~3개를 화면 경로로. 각 단계에서 무슨 일이 일어나는지 한 줄씩.
7. **자동으로 넣은 것 / 가정 / 아직 안 정한 것** — 자동 추가·임의 결정·미정 항목을 각각 모아 표시.
8. **(기계 표현) 기능 인덱스** — 문서 하단에, 위 기능 목록에서 뽑은 `ia.features` 블록을 만든다(` ```json ia.features ` 태그). 문서에 나온 **모든 `FEAT.*` 기능**을 `{ id, label, audience, status }`로 한 블록에 담는다(`audience`=`FEAT.admin.*`면 `admin` 아니면 `user`, `status`=부가 기능이면 `later`·나머지 `current`). 산문이 원본이고 이 블록은 파생 인덱스다 — 기능을 더하거나 빼면 이 블록도 다시 만든다.

한꺼번에 확인받지 말고, 위를 **한 번에 다 채운 초안**으로 만든다. **문서 맨 앞 frontmatter**도 템플릿대로 채운다(`doc_type: ia`·`version`·`ssot: prose`·`machine.*`).

## Step 3: 한 번에 검토 + 수정

완성된 초안 전체를 보여주고 한 번만 크게 묻는다:
"이렇게 초안을 잡아봤어요. **빼고 싶은 기능, 빠진 것, 이름이 어색한 것** 있으면 알려주세요. 없으면 이대로 확정할게요."

- 사용자가 고칠 부분을 말하면 반영하고, 나머지는 그대로 둔다.
- 다시 처음부터 훑지 말고, 바뀐 부분만 손본다.
- 큰 틀이 괜찮다고 하면 확정한다.

## Step 4: 마무리

1. **읽기 쉬운 화면·기능 구조 문서를 "파일"로 만들어 사용자에게 준다.** 채팅창에만 길게 늘어놓지 말고, 사용자가 열어보고 내려받아 보관할 수 있는 **문서 파일**로 만든다. `assets/화면구조-템플릿.md` 구조를 그대로 채운다(frontmatter + 기능 인덱스 블록 포함).
   - 기본은 **마크다운(.md) 문서**로 만든다. 사용자가 원하면 Word(.docx) 문서로도 만들 수 있다.
   - **`schemas/ia.v1.schema.json`을 산출물 옆 `schemas/`에 복사**한다(스킬 소유 고정 자산 그대로 — 문서 frontmatter의 `machine.schema`가 가리킨다).
   - 사용자가 "파일로 만들어 달라"고 말하지 않아도 **항상 파일로 만든다.** 이게 이 스킬의 기본 산출 방식이다.
   - (Claude Code에서는 파일로 저장하고, claude.ai에서는 문서로 만들어 보여준다 — 환경에 맞게.)
2. 사용자에게 간단히: **"화면과 기능 구조를 문서로 정리했습니다. 여기서 언제든 빼거나 더하시면 됩니다."**
3. 기획서·설계서 흐름을 쓰는 사람이면, 이 구조가 설계서의 구성 부분으로 이어진다고 한 줄만 덧붙인다. 어려운 명령어는 안내하지 않는다.

## 재진입 — 스킬 버전업 업그레이드 (기존 문서를 최신 구조로)

이미 만든 IA 문서를, 그동안 진화한 최신 스킬 구조에 맞춰 **빠진 항목만 채워 올리는** 재진입이다.
방금 만든 문서를 그 자리에서 고치는 Step 3과 다르고, 처음부터 다시 만드는 신규 생성(Step 0~4)과도 다르다.

### 언제 쓰는지
사용자가 예전에 이 스킬(또는 구버전)로 만든 IA 문서를 가져와 "이거 최신 버전으로 맞춰줘",
"업그레이드해줘", "최신 스킬 구조로 다시 만들어줘" 등으로 요청할 때. "처음부터 새로 만들어 줘"는
여기 해당하지 않는다(그건 Step 0~4 신규 생성 흐름).

### 절차 (반드시 이 순서로)
1. 가져온 기존 IA 문서를 **기준선**으로 삼는다. 처음부터 다시 만들지 않는다.
2. 그 문서의 구조를 현재 버전의 `assets/화면구조-템플릿.md`가 요구하는 구조와 **항목 단위로 비교**한다. 지금 템플릿엔 있는데 기존 문서엔 없는 항목(예: 화면 정의의 "로그인 필요" 필드, 기능 `FEAT` ID 형식, 네비게이션 형식, **맨 앞 frontmatter, 하단 `ia.features` 기계 블록, 옆 `schemas/ia.v1.schema.json`, "자동 점검할 수 있는 것" 설명 절** 등)을 **빠진 부분(gap)**으로 표시한다. frontmatter가 없으면 얹고, 기능 인덱스 블록이 없으면 산문의 `FEAT.*`를 모아 만들고, 스키마 파일이 없으면 복사한다.
3. **빠진 부분만 채운다.** 채울 때는 그 항목을 새로 만들 때 이 스킬이 원래 쓰는 **자동 판단 기준을 그대로 적용**한다(예: 로그인 필요는 "개인 소유물→필수 / 공개+개인화 섞임→콘텐츠 분리 / 완전 공개→동일"). 사용자에게 매번 묻지 않고 자동으로 채운 뒤 **"자동으로 넣은 것"에 표시**한다 — 다른 자동 결정과 같은 원칙.
4. **이미 있는 내용은 절대 다시 쓰거나 고치지 않는다.** 이 재진입은 순수하게 "빠진 것만 추가"이며, 기존 내용을 다듬거나 개선하는 절차가 아니다(그건 Step 3의 몫). 이미 있는 기능 ID·화면 정의·네비게이션 등은 글자 하나 건드리지 않는다.
5. 끝나면 **무엇을 새로 채웠는지 목록으로 요약**해 알린다. (예: "이번 업그레이드로 추가된 것: 화면 12개에 '로그인 필요' 필드 추가 — 필수 3개, 콘텐츠 분리 2개, 동일 7개")
6. **같은 파일에 업데이트**한다. 새 파일로 분리하지 않는다.

### 하지 말 것
- 문서를 처음부터 다시 만들지 말 것(재생성이 아니라 gap만 채움).
- 빠진 부분을 채운다는 명목으로 이미 있는 내용을 재작성하지 말 것.
- 특정 필드(로그인 필요 등) 하나만 하드코딩해 처리하지 말 것 — **현재 템플릿과 비교해 빠진 걸 찾는 일반 절차**로 동작해야 다음 스킬 업데이트에도 그대로 재사용된다.

## 선택 모듈: 유스케이스 (요청 시에만, 기본은 만들지 않음)

화면·기능 구조 문서와 **별개로**, 요청이 있을 때만 만드는 추가 문서다.
화면 단위가 아니라 **"사용자가 이루려는 목표" 단위**로 묶어, 그 목표를 이루는 과정을
행위자·사전조건·기본 흐름·대안 흐름·예외 흐름·사후조건으로 정리한다.

### 기본값 — 만들지 않는다
화면·기능 구조 문서(Step 4)는 지금처럼 **항상** 만들지만, 유스케이스는 **디폴트로 만들지 않는다.**
아래 "켜는 방법"에 해당할 때만 만든다.

### 켜는 방법
- 사용자가 "유스케이스도 만들어줘" 같이 요청할 때.
- 또는 상위 오케스트레이터(easy-dev-suite)가 이 옵션(생성 여부 + 분량)을 켜서 호출할 때.

### 재료 — 새로 묻지 않는다 (추출 우선)
이미 만든 것에서 뽑아 재조립한다. **새 인터뷰·새 질문을 벌이지 않는다.**
- IA의 **핵심 흐름**(주요 작업 경로) → 기본 흐름의 뼈대.
- 화면 정의의 **"여기서 할 수 있는 일"(동작→결과)** → 흐름의 각 단계.
- 화면 정의의 **"상태·예외"** → 대안 흐름·예외 흐름.
- (있으면) 기획서의 **규칙·금지** → 사전조건·예외 조건.
- IA·기획서에 없어서 비어 있던 대안·예외 흐름만 **합리적 가정으로** 채우고, 채운 것은 **전부 "가정"에 표시**한다.

### 분량 3단계
켤 때 사용자에게 한 번 묻는다: "간단히 / 적당히 / 자세히 중 어느 정도로 할까요?"
- **간단히**: 핵심 목표 2~3개만, 기본 흐름 위주.
- **적당히(기본)**: 주요 목표별로 기본 흐름 + 주요 예외 흐름.
- **자세히**: 행위자·사전조건·기본/대안/예외 흐름·사후조건 전부.

켜졌는데 사용자가 분량을 말하지 않으면 **"적당히"로 만들고, 그 사실을 한 줄로 알린다.**
(오케스트레이터 호출이면 넘어온 분량 값을 쓰고, 값이 없으면 "적당히".)

### 만드는 방법
1. **목표를 고른다.** IA의 핵심 흐름이 곧 대표 목표다(예: "모임 만들기", "모임 참여하기"). 분량이 "간단히"면 2~3개, 그 이상이면 주요 목표 전부.
2. 목표마다 `assets/유스케이스-템플릿.md`의 블록 하나를 채운다. 위 "재료"에서 뽑고, 비었던 곳만 가정으로. **각 유스케이스에 관련 기능 ID(`FEAT.…`)를 함께 적는다** — ID는 IA에서 확정된 것을 **이어받아 인용만** 하고 새로 만들거나 바꾸지 않는다.
2-1. **각 유스케이스에 안정 ID `UC.도메인.목표`를 붙인다.** 시나리오 등 다른 문서가 유스케이스를 **이름이 아니라 ID로** 참조할 수 있게 하기 위함이다(정책 `POL.*`·기능 `FEAT.*`와 같은 원리). 형식은 `UC.도메인.목표`(영문 소문자 camelCase, 도메인은 관련 `FEAT` 도메인과 맞춘다. 예: `UC.meeting.create` — "산책 모임 만들기"). 이 ID는 IA(유스케이스)가 원본으로 확정하며, **한 번 정하면 목표 이름이 바뀌어도 ID는 유지**한다. 사람에게 보이는 것은 여전히 **이름**이고, ID는 그 옆에 `이름(UC.도메인.목표)` 형태로 함께 적는다.
3. **항상 파일로 만든다.** 화면·기능 구조 문서와는 **별도 파일**. 기본 마크다운(.md), 원하면 Word(.docx). 파일 이름 예: `유스케이스-[프로젝트이름].md`.
4. **비전문가 언어.** 사람에게 보이는 표현은 풀어 쓴다("행위자 = 이 일을 하는 사람"). `UC.*`·`FEAT.*` 같은 ID는 **개발·문서 연결용 기계 식별자**로, 이름 옆에 `이름(ID)`로 조용히 달 뿐 "UC-01, 액터" 같은 딱딱한 표기를 전면에 내세우지 않는다.

> 이 모듈은 기존 흐름(초안 우선 화면 구조 생성, 백오피스 규칙 등)을 **바꾸지 않는다.**
> 재료가 있어야 하므로 화면·기능 구조 문서를 만든 뒤(또는 함께) 만든다.

> **유스케이스와 시나리오의 경계:** 유스케이스는 **한 기능(목표) 단위**의 상호작용 흐름(주/대안/예외)이다.
> 여러 기능·화면·정책을 관통하는 **더 큰 사용자 여정**은 시나리오(`easyproduct-scenario`)가 맡으며,
> 시나리오는 유스케이스를 **베끼지 않고 링크**한다. 같은 흐름을 두 문서에 중복 서술하지 않는다.

## 선택 모듈: 기능 관계도 (기존 IA 문서에서 도출, 기본은 만들지 않음)

**IA 문서에서 관계를 도출해 별도 문서로 그리는 작업**이다. 관계도는 IA 문서에 이미 담긴 정보에서
언제든 다시 뽑아낼 수 있는 **파생물**이므로, **IA 문서 안에 넣지 않는다.** IA 문서는 딱 IA 문서로
두고(SSOT), 관계도는 **별도 파일**로 만든다. (easy-data-model의 JSON 스냅샷, `easyproduct-scenario`의
파생 시나리오와 같은 "읽기 전용 도출"이다 — 원본이 바뀌면 다시 뽑는다.)

### 기본값 — 만들지 않는다
요청이 있을 때만. 화면·기능 구조 문서 생성 흐름은 그대로 두고, 이 작업은 그 위에서 따로 돈다.

### 켜는 방법
- 사용자가 "관계도 만들어줘", "기능 관계도 그려줘" 같이 요청할 때.
- 또는 상위 오케스트레이터(easy-dev-suite)가 이 옵션을 켜서 호출할 때.

### 입력 — 어떤 IA 문서든 된다
- **이 스킬이 방금 만든 IA 문서가 아니어도 된다.** 외부에서 가져온 IA·화면 구조 문서라도, 화면 정의·핵심 흐름·기능 목록 비슷한 내용이 있으면 거기서 도출한다.
- 문서가 맥락에 없으면 먼저 받아서 읽는다. IA 문서를 **읽기만** 하고 **절대 고치지 않는다.**
- `FEAT` ID가 있으면 관계도의 노드로 그대로 쓴다. 없으면 이름으로 쓰되, 이름이 겹칠 수 있음을 한 줄 알리고 가능하면 도메인을 함께 표기한다.

### 그리는 관계도 (둘 다)
1. **UI 흐름 관계도 (화면 전이도)** — 화면 사이의 이동. 노드=화면(+`FEAT` ID), 화살표="무엇을 누르면 어디로".
   - 재료: 화면 정의의 **진입 경로·이어지는 화면·동작→결과**, **핵심 흐름**.
2. **프론트↔백오피스 연동 관계도 (트리거)** — 한 기능이 다른 기능에 일을 만드는 관계(예: 신고→신고 처리, 결제→정산, 가입→회원 관리).
   - 재료: **백오피스 짝맞춤**. 화면 이동으로는 안 보이는 관계다.

> **데이터 흐름 관계도(DFD)는 만들지 않는다.** IA엔 데이터 항목·저장소 정보가 없다(그건 easy-data-model의 몫).
> 요청받아도 "데이터 모델 문서가 함께 있어야 제대로 그린다"고 알리고, IA만으로는 그리지 않는다.

### 만드는 방법
1. **추출 우선.** IA 문서에서 이동선과 트리거를 뽑는다. **새 질문을 벌이지 않는다.** 문서에 없어 추론한 연결은 "가정"에 표시한다.
2. 두 관계도를 각각 **읽기 쉬운 관계 목록(항상 이해 가능) + mermaid 다이어그램(렌더되는 곳에선 그림)**으로 만든다. `assets/기능관계도-템플릿.md`를 채운다.
3. **mermaid 방식**: `flowchart`(방향 `LR` 또는 `TD`). 노드 id는 점 없는 짧은 영문, 라벨에 `FEAT` ID + 이름(`node["이름<br/>FEAT.도메인.기능명"]`). 화살표 라벨(`-->|동작|`)에 동작/트리거. 프론트↔백오피스는 `subgraph 프론트` / `subgraph 백오피스`로 나눠 그린다.
4. **항상 파일로 만든다.** IA 문서와 **별도 파일**. 기본 마크다운(.md), 원하면 Word(.docx). 파일 이름 예: `기능관계도-[프로젝트이름].md`.
5. 비전문가 언어. mermaid가 안 그려지는 환경이면 관계 목록만으로도 이해되게 쓴다.

> 이 모듈은 IA 문서를 **읽기만** 하고 바꾸지 않는다. IA가 바뀌면 관계도는 다시 도출하면 된다.

## 백오피스 설계 규칙

- **프론트만 있고 백오피스가 없는 초안은 미완성이다.** 회원·콘텐츠·주문·문의가 생기면 그걸 볼 사람이 반드시 필요하다.
- 짝맞춤은 `references/유형별-기본기능.md`의 "백오피스 짝맞춤 원칙" 표를 따른다.
- 관리자 기능도 자동으로 넣되, 규모가 아주 작은 개인 프로젝트면 "관리자 최소본(회원·문의·통계만)"으로 가볍게 잡고 그렇게 표시한다.
- **네비게이션은 프론트와 대칭으로 각각 정의한다**(한쪽만 하면 미완성). 백오피스는 **LNB(좌측 메뉴) 중심**으로, 메뉴를 **그룹화 + 역할별 노출**로 적고, 계층은 **대시보드(홈) → 목록 → 상세 → 상세 내 탭·액션**으로 잡는다. Footer 대신 필요하면 **유틸리티 바(버전·환경·도움말)**를 둘 수 있다(필수 아님). 사업자 정보 등 대외 표시는 프론트가 담당하므로 백오피스에서 반복하지 않는다. (상세 원칙: `references/네비게이션-설계-원칙.md`)

## 비전문가 응대 규칙 (대화 내내)

- 초반 질문은 **3~4개**. 그 뒤로는 묻지 말고 초안으로 답한다.
- "잘 모르겠어요"라고 하면 **예시 2~3개**를 제시해 고르게 한다.
- 전문 용어가 나오면 **즉시 한 줄로 풀어** 쓴다.
- "이건 안 됩니다"보다 "이렇게 넣어뒀는데 필요 없으면 빼세요"로.
