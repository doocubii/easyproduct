# 시나리오 · 유스케이스 · 유저스토리 · 정책서 — 역할 경계

이 넷은 **같은 종류가 아니다.** 헷갈려서 같은 내용을 여러 문서에 중복하는 것을 막으려고 경계를 정리한다.
두 개의 축으로 갈린다: **위상**(원본 SSOT냐 파생·보강이냐)과 **다루는 것**(규칙이냐 행동이냐).

## 한눈에

| 문서 | 위상 | 다루는 것 | 단위 | 한 줄 정의 |
|---|---|---|---|---|
| **정책서** | **SSOT(원본)** | 규칙 | 규칙 | "이렇게 운영한다"는 운영 규칙 (`POL.*`) |
| **유저 스토리** | 보강자료 | 행동(요구) | 단일 요구 | "누가·무엇을·왜" — 요구의 가치 |
| **유스케이스** | 보강자료 | 행동(흐름) | 한 기능 | 한 기능의 상호작용 흐름(주/대안/예외) |
| **시나리오** | 파생물(축)→승격 시 테스트 | 행동(여정) | 여러 기능·화면 | 여러 기능·화면·정책을 ID로 관통하는 여정 |

## 구분선 두 개

### (A) 정책서는 나머지 셋과 근본적으로 다르다

정책서는 **규칙의 원본(SSOT)**이다 — "환불은 픽업 24시간 전까지", "미성년자 가입 제한" 같은 **판단·제약**을 정의한다.
나머지 셋(유저스토리·유스케이스·시나리오)은 **사용자의 행동/흐름**을 그리는 서술물이다.

- 흐름 문서(유스케이스·시나리오)는 "사용자가 픽업하러 온다"까지는 그리지만, **"노쇼 시 환불 여부·수수료"** 같은 판단 기준은 담지 않는다.
  그 판단의 원본이 정책서다.
- 그래서 시나리오가 결제/취소 흐름에 닿으면 정책서의 `POL.*`를 **참조**한다. 정책 내용을 시나리오에 옮겨 적지 않는다.

### (B) 유저스토리 → 유스케이스 → 시나리오는 "행동을 다루되 단위가 다르다"

- **유저 스토리** = 가장 작은 단위, 단일 요구의 **가치**. "동네 주민으로서 주변 할인을 둘러보고 싶다, 싸게 사려고." (why/who)
- **유스케이스** = 중간 단위, **한 기능**의 상호작용. "예약하기" 기능의 주 흐름·대안·예외. (한 기능이 어떻게 쓰이나)
- **시나리오** = 가장 큰 단위, **여러 기능·화면·정책을 잇는 여정**. "둘러보기→예약→결제(한도초과 시 분기)→픽업". (여정 + 테스트 씨앗)

## 중복을 피하는 규칙

1. **시나리오는 유스케이스를 베끼지 않고 링크한다.** 같은 흐름을 두 문서에 쓰지 않는다. 시나리오 스텝이 유스케이스를 가리키게 한다.
2. **시나리오는 정책 내용을 옮기지 않고 `POL.*`로 참조한다.**
3. **시나리오는 데이터 정의를 옮기지 않고 `그룹.필드`로 참조한다.**
4. 셋의 경계가 실사용에서 흐려질 수 있다 — 그럴 때 판단 기준: **"단일 요구의 가치면 유저스토리, 한 기능의 상호작용이면 유스케이스,
   여러 기능을 관통하는 여정이면 시나리오."**

## 한 그림

```text
정책서(SSOT·규칙)  ── "무엇이 허용/금지/분기되나" 결정 ──┐
                                                        │ 참조(POL.* 링크)
유저스토리(왜)  →  유스케이스(한 기능 흐름)  →  시나리오(여정 축) ──┘
   가치            상호작용 단위               여러 기능·화면·정책을 ID로 꿰는 실
                                             └→ 승격 시 GWT 인수 테스트
```

세로축은 행동을 점점 큰 단위로 확대하고, 정책서는 옆에서 규칙을 공급한다. 시나리오만이 정책·데이터·화면·유스케이스를
가로질러 링크하는 **유일한 "축"**이다.
