# 정합·추적 점검 (검증) 절차

시나리오가 다른 문서와 **어긋나지 않는지** 확인하는 절차. 사람이 읽고 따라 해도 되고, 에이전트가 자동화해도 된다.
`easyproduct-suite`의 Stage 4(정합·추적 점검)와, 이 스킬 단독 실행의 Step S4에서 쓴다.

> **이 점검이 할 수 있는 것과 없는 것 (먼저 못 박기)**
> - **할 수 있음(신뢰 가능)**: 죽은 링크 탐지 — 시나리오가 가리키는 ID가 실제 문서에 있는지 없는지는 기계적으로 확실히 판정된다.
> - **후보만 제시(사람 확인 필요)**: 의미적 모순 — "흐름이 규칙과 어긋나는가"는 판단이 필요하다. 확신할 때만 불일치로 보고하고,
>   애매하면 "미확인"으로 남긴다. **자동 통과를 "정합 확인됨"으로 착각하지 않는다**(없는 것보다 나쁜 거짓 안심 금지).

## 입력

- 점검 대상 시나리오들(`supporting/scenarios/*.md`)의 **추적표**(닿는 FEAT/데이터/정책/유스케이스 ID).
- 원본 문서들: IA(`FEAT.*`), 데이터 모델(`그룹.필드`), 정책서(`POL.*`), 유스케이스.

## 1. 링크 존재 검사 (기계적)

각 시나리오 스텝·추적표가 참조하는 ID가 원본에 **실재**하는지 확인한다. 참조는 `이름(ID)` 형태이므로 **괄호 안 ID로 대조**한다(이름이 아니라 ID로).

- `FEAT.*` → IA/화면 설계서에 그 기능 ID가 있는가?
- `그룹.필드` → 데이터 모델에 그 변수가 있는가?
- `POL.*` → 정책서에 그 규칙 ID가 있는가? (정책 ID 체계: policy-legal `references/정책-ID체계.md`)
- `UC.*` → 유스케이스(IA)에 그 목표 ID가 있는가? (유스케이스는 `UC.도메인.목표` 안정 ID를 갖는다.)

**없으면 "죽은 링크"**로 보고: "시나리오 `SCN.x.y`가 `FEAT.a.b`를 참조하는데 IA에 없음 → 오탈자이거나, 원본에서 삭제됐거나, 아직 안 만들어짐."

> **옛 문서 배려**: 참조 대상 원본이 아직 ID가 없는 옛 구조(예: `UC.*`가 안 붙은 유스케이스)라면, 그건 **결함이 아니다.**
> ID 부재를 "죽은 링크"로 보고하지 말고, 필요하면 해당 스킬의 ID 부여(버전업)를 권하는 선에서 다룬다.

## 2. 정합 검사 (의미 — 신중)

시나리오가 서술한 흐름/규칙이 원본 결정과 **모순되지 않는지** 본다.

- 예: 시나리오는 "가입 시 관리자 승인 게이트를 거친다"인데 IA/정책은 "셀프서비스 즉시 가입"이면 **불일치**.
- 예: 시나리오는 "24시간 전까지 취소 가능"인데 정책서 `POL.noshow.cancel`은 "48시간 전"이면 **불일치**(시나리오가 낡음).

판정 원칙:
- **확신할 때만** "불일치"로 보고하고, 어느 쪽이 원본(SSOT)인지 밝힌다 → 원본이 옳으니 **시나리오를 재도출**하라고 안내.
- 판단이 애매하면 "**미확인 — 사람 확인 필요**"로 남긴다. 억지로 통과시키지 않는다.

## 3. 커버리지 검사 (필수·기계적)

**기준 목록**(무엇이 "핵심"인지 흔들리지 않게 고정):
- **기획서의 핵심 행동** — doc-builder 기획서 게이트의 "핵심 행동 3개"(고정·식별 가능). **1순위 기준**.
- **IA 핵심 흐름에서 파생된 유스케이스** — ia-designer가 "IA의 핵심 흐름이 곧 대표 목표"라며 만든 유스케이스(`UC.*`). 보조 기준.

이 두 목록을 놓고, 각 항목이 **최소 하나의 시나리오에 닿는지** 대조한다. 어느 핵심 행동/핵심 유스케이스도
어떤 시나리오에도 안 닿으면 **누락으로 반드시 보고**한다 — 시나리오는 테스트의 씨앗이라, 여정 누락은 곧
**테스트·점검의 구멍**이 되기 때문이다(그래서 이 검사는 선택이 아니다).

- 기준은 **핵심 행동·핵심 유스케이스**다. 모든 부수 기능까지 시나리오에 나와야 하는 건 아니다.
- 예: 기획서 핵심 행동에 "결제"가 있는데 어느 시나리오도 `FEAT.billing.*`에 안 닿음 → **결제 여정 누락**으로 보고.
- 이 검사는 목록 대조라 기계적으로 판정된다(의미 판단이 필요한 2번과 달리 확실히 잡는다).

## 4. 재도출 트리거

원본이 바뀐 흔적이 있으면(참조 대상의 정의·규칙이 달라졌으면) 해당 시나리오를 `파생` 상태에서 **재생성**한다.
링크-온리 원칙 덕에 재생성 비용이 낮다 — 원본을 다시 읽어 스텝의 흐름·링크를 갱신하면 된다.

## 5. 설계 결함 후보 (게이트키퍼 — 보고 후 사용자 선택)

작성·추적 중 흐름 결함(안 이어지는 화면·빠진 단계·논리 모순, 또는 원본과의 의미 불일치)이 드러나면 **설계 결함 후보**로 짚는다.
단 **어느 쪽이 틀렸는지 단정하지 않는다** — 시나리오가 낡았을 수도, 원본(화면 흐름)이 결함일 수도 있다. 양쪽 가능성과 **회귀 후보**를 제시한다:

- 핵심 행동·가치 자체가 어긋남 → **기획**(doc-builder) 회귀 후보
- 흐름 논리·화면 누락·연결 문제 → **IA/화면 설계**(ia-designer/screen-design) 회귀 후보
- 시나리오만 낡음(원본이 옳음) → 시나리오 **재도출**로 충분

**진행은 사용자가 정한다.** 자동으로 재설계를 밀어붙이지 않는다. 사용자가 회귀를 택하면 해당 스킬의 재진입 절차로 넘기고, 원본이 고쳐지면 시나리오를 재도출한다.

## 보고 방식

- **누락·불일치가 있을 때만** 조용히 보고한다(정상이면 굳이 길게 알리지 않는다). 기존 완성도 점검과 같은 톤. 단 **커버리지 누락은 반드시** 보고한다.
- 보고 항목: (a) 죽은 링크 목록, (b) 불일치/설계 결함 후보 + **회귀 후보(어느 단계로)**, (c) 미확인 항목, (d) 커버리지 누락.
- 각 항목에 **권장 조치**를 붙이되, 회귀 여부는 **사용자가 선택**하게 제시한다: 링크 수정 / 시나리오 재도출 / ○○ 단계로 회귀(사용자 결정).
