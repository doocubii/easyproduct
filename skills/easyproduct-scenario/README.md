# easyproduct-scenario

> 개발·기획 지식이 없어도 서비스의 "사용자 여정"(시나리오)을 만들고, 그 시나리오가 화면 구조·데이터·정책·유스케이스와 어긋나지 않는지 점검하도록 돕는 스킬입니다. 시나리오는 요구·설계·테스트를 잇는 흐름의 축이되, 결정의 원본(SSOT)은 아닙니다.

[easyproduct 스킬 세트](../../README.md)의 일부입니다.

## 무엇을 하나
- 시나리오의 **세 얼굴**을 다룹니다: **설계 드라이버**(흐름을 먼저 그려 화면·기능을 끌어냄), **검증 파생물**(확정 문서에서 다시 뽑아 문서와 맞는지 확인), **실행 테스트 씨앗**(GWT로 정형화해 인수 시나리오로 승격).
- 기획 직후 핵심 여정을 **자연어로 스케치**해 IA·화면 도출을 이끕니다(옵트인).
- 화면 설계 후 확정본을 참조해 시나리오를 **읽기 전용으로 재도출**하고, `SCN.*` ID로 여러 화면·기능·정책을 꿰어 추적 링크를 답니다.
- 시나리오가 참조하는 ID가 실제 문서에 실재하는지(**죽은 링크**), 원본과 모순되지 않는지(**의미 불일치 후보**) 정합을 점검합니다.
- 두 **거버넌스 규칙**을 지켜 시나리오가 "또 하나의 SSOT"가 되지 않게 막습니다.

## 언제 쓰나
- "시나리오", "사용자 여정", "유저 저니", "플로우", "이 흐름대로 화면이 맞나"를 만들거나 점검하려 할 때.
- "문서끼리 안 맞는 것 같다", "정책이랑 화면이 따로 논다", "빠진 흐름 없나 확인해줘"라고 할 때.
- 기획서·IA를 만든 뒤 "흐름 관점에서 화면을 도출"하거나, spec-kit 인계 전 수용 시나리오를 정리하려 할 때.

## 산출물
- `supporting/scenarios/scenario-{도메인}-{이름}.md` — 시나리오(여정) 문서. 유스케이스·유저스토리 옆에 둡니다.
- 시나리오 ID는 `SCN.{도메인}.{이름}`, 스텝은 그 아래 순번(`SCN.org.onboarding.3`)을 씁니다.
- 각 스텝이 닿는 `FEAT.*`·데이터 변수·`POL.*`(·유스케이스)를 링크로 달고, 문서 끝에 추적표와 원본 링크, 상태 배지(`탐색`/`파생`/`확정`)를 붙입니다.

## 단독 실행 vs 세트
- 기본은 **독립 실행**입니다. "이 서비스 흐름 시나리오 만들어줘"처럼 단독으로도 부릅니다. 기획서·IA가 있으면 읽어 재료로 쓰고, 없으면 스케치 질문으로 흐름부터 그립니다.
- 다른 스킬의 내부 파일을 전제하지 않고 **입력으로 받은 문서**만 참조합니다.
- 세트로 물릴 때는 `easyproduct-suite`가 스케치(기획 직후)·파생(화면 설계 후)·검증(Stage 4) 시점에 호출합니다.

## 구성 파일
- `SKILL.md` — 스킬 본문(시나리오의 위상·두 거버넌스 규칙·생애주기·GWT 적용 시점·스케치/파생/검증/승격 절차).
- `assets/scenario-template.md` — 산출물 템플릿(`SCN.*` ID + GWT + 스텝별 추적 링크 + 추적표 + 원본 링크 + 상태 배지).
- `references/traceability-check.md` — 정합·추적 점검(검증) 절차: 링크 존재/정합/커버리지/재도출/설계 결함.
- `references/scenario-vs-usecase-userstory.md` — 시나리오·유스케이스·유저스토리·정책서의 역할 경계(중복 방지).
- `references/example-walkthrough.md` — 작성 예시(참고 패턴): 채워진 시나리오 + mermaid + 게이트키퍼 보고의 실제 모습.

## 세트 안에서의 위치
- 결정의 원본은 여전히 기획서·IA·정책서·데이터 모델입니다. 시나리오는 정책서의 `POL.*`, IA의 `FEAT.*`, 데이터 모델의 `그룹.필드`를 **베끼지 않고 ID로 참조(링크)만** 합니다(링크-온리 규칙).
- 스케치에서 나온 새 결정은 반드시 원본 문서로 옮겨 적습니다(결정 이관 규칙) — 정책은 policy-legal, 화면·기능은 ia-designer, 데이터는 data-model로 넘깁니다.
- 같은 여정이 앞의 **스케치(드라이버)**와 뒤의 **파생물(검증)**로 두 번 등장하며, `easyproduct-suite`가 스케치·파생·검증 시점에 이 스킬을 호출합니다. 승격 시에는 spec-kit의 인수 시나리오가 됩니다.
