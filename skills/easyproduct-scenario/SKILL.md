  - "문서끼리 안 맞는 것 같다", "정책이랑 화면이랑 따로 논다", "빠진 흐름 없나 확인해줘"라고 할 때
  - 기획서·화면 구조(IA)를 만든 뒤 "흐름 관점에서 화면을 도출"하거나, spec-kit 인계 전 수용 시나리오를 정리하려 할 때
  기획서·IA 등 다른 문서가 있으면 그것을 재료로 삼고, 없으면 스케치 단계부터 시작한다.

# 시나리오 만들기·점검하기

## 목적

서비스의 **사용자 여정**을 시나리오로 정리하고, 그 시나리오가 다른 문서와 정합한지 점검한다.
비전문가도 다룰 수 있게, 앞단(탐색)은 **평범한 문장**으로 그리고, 개발 인계에 가까운 뒤단(파생·승격)에서만
**Given/When/Then(GWT)** 구조로 정형화한다.

시나리오의 **세 얼굴**:
1. **설계 드라이버** — 흐름을 먼저 그려 화면·기능을 끌어낸다.
2. **검증 파생물** — 확정된 문서에서 다시 뽑아, 흐름이 문서와 맞는지 확인한다.
3. **실행 테스트 씨앗** — GWT로 정형화해 spec-kit의 인수 시나리오로 승격한다.

## 시나리오의 위상 (반드시 지킬 원칙)

1. **시나리오는 "흐름의 축"이되 "결정의 SSOT"가 아니다.** 결정(정책·화면·데이터 구조)의 원본은
   여전히 기획서·IA·정책서·데이터 모델이다. 시나리오는 그것들을 **ID로 꿰는 실 + 테스트의 씨앗**이다.
2. **두 거버넌스 규칙** (이 둘이 SSOT를 지킨다 — 어길 것 같으면 멈추고 다시 생각한다):
   - **결정 이관 규칙**: 시나리오(특히 스케치)에서 새로 나온 결정은 **반드시 원본 문서**(IA·기획서·정책·데이터)로
     옮겨 적는다. **시나리오에만 남기지 않는다.** (예: 스케치 중 "무료 한도 초과 시 결제 유도"라는 정책이 정해지면,
     그 규칙은 정책서에 `POL.payment.*`로 들어가야 하고 시나리오는 그것을 참조만 한다.)
   - **링크-온리 규칙**: 확정 시나리오는 원본 내용을 **베끼지 않고 ID로 참조(링크)만** 한다.
     원본이 바뀌면 시나리오를 **재도출**한다. (기능 관계도·JSON 스냅샷과 같은 읽기 전용 파생 원칙.)
3. **spec-kit 인계 시 시나리오는 "보강자료"에서 "인수 시나리오"로 승격**한다. (SSOT 구분표에 명시.)

> 시나리오가 이 두 규칙을 어기면(결정을 시나리오에만 두거나, 원본을 베껴 두면) 곧 원본과 어긋나
> "또 하나의 SSOT"가 되어 버린다. 그것이 이 스킬이 가장 경계하는 실패다.

## 다른 문서와의 관계 (겹치지 않게)

| 문서 | 위상 | 단위 | 한 줄 |
|---|---|---|---|
| 유저 스토리 | 보강자료 | 단일 요구 | "누가·무엇을·왜" — 요구의 가치 |
| 유스케이스 | 보강자료 | 한 기능 | 한 기능의 상호작용 흐름(주/대안/예외) |
| **시나리오** | 파생물(축) → 승격 시 테스트 | **여러 기능·화면의 여정** | 여러 기능·화면·정책을 ID로 관통하는 사용자 여정 |
| 정책서 | **SSOT(규칙)** | 규칙 | 시나리오가 `POL.*`로 **참조**하는 운영 규칙의 원본 |

- 시나리오는 유스케이스를 **베끼지 않고 링크**한다. (유스케이스=한 기능 흐름, 시나리오=여러 기능을 잇는 더 큰 여정.)
- 시나리오가 어떤 규칙을 따르면 정책서의 `POL.*` ID를 **참조**한다(내용을 옮겨 적지 않는다).
- 자세한 경계·중복 방지는 `references/scenario-vs-usecase-userstory.md`.

## 생애주기 (언제·무엇을)

```text
① 스케치(탐색)   → 기획 직후. 흐름으로 IA·화면을 끌어냄. 자유·반복. 자연어 허용. [상태: 탐색] (임시, 폐기 가능)
② 이관(확정)     → 스케치에서 나온 결정을 IA·기획서·정책·데이터로 옮김. (기존 스킬들이 SSOT 확정)
③ 파생(정리)     → 화면 설계 후. 확정본을 참조해 재도출. GWT + 추적 링크. [상태: 파생] (읽기 전용 파생물)
④ 승격(테스트)   → spec-kit 인계 시 Given/When/Then 인수 시나리오로. [상태: 확정]
      +
⑤ 검증(정합)     → 시나리오↔파생 링크의 존재·정합 점검. (죽은 링크·모순 탐지)
```

**중요**: 같은 시나리오가 세트에 **두 번** 등장한다 — 앞의 **스케치(드라이버)**와 뒤의 **파생물(검증)**.
역할이 다르므로 문서 머리에 **상태 배지**(`탐색`/`파생`/`확정`)를 달아 구분한다.

## 표준 형식·산출물 규격

- **위치**: `supporting/scenarios/scenario-{도메인}-{이름}.md` (유스케이스·유저스토리 옆).
- **ID 체계**: 시나리오 `SCN.{도메인}.{이름}`, 스텝은 그 아래 순번(`SCN.org.onboarding.3`).
  FEAT·데이터 변수·정책 ID와 **겹치지 않는 접두어**(`SCN.`)를 쓴다.
- **형식**: Given/When/Then + **스텝별 추적 링크**. 각 스텝이 닿는 `FEAT.*`·데이터 변수·`POL.*`(·유스케이스)를 `[ ]`로 가리킨다.
- **추적표**: 이 시나리오가 닿는 FEAT / 데이터 / 정책 / 유스케이스 ID 목록을 문서 끝에 모은다(검증이 이걸 본다).
- **원본 링크**: 각 결정의 SSOT 문서 위치를 가리키기만 한다(베끼지 않는다).
- **한눈에 보는 그림(mermaid)**: GWT 스텝을 사람이 한눈에 보도록 **mermaid flowchart**를 함께 넣는다. When=행동 노드,
  Then=결과 노드, 대안·예외=분기로 그린다. 이 그림은 **GWT 스텝의 파생 뷰**다(원본은 GWT 텍스트) — 스텝이 바뀌면 다시 그린다.
  (기능 관계도가 mermaid를 쓰는 것과 같은 방식.)
- 템플릿: `assets/scenario-template.md`.

### 비전문가 조화 — GWT는 "언제 쓰나"로 가른다 (중요)

doc-builder의 "Given/When/Then을 쓰지 않는다"(비전문가 원칙)와 **모순되지 않게 단계로 분리**한다:
- **스케치 단계(①)**: **자연어 흐름** 허용. GWT를 강요하지 않는다. ("사용자가 조직을 만들면 관리자가 된다" 식.)
- **파생·승격 단계(③④)**: **GWT 구조 + 추적 링크**로 정형화한다. (개발·테스트가 쓰므로.)
- 즉 GWT는 뒤단에서만. 사용자에게 앞단에서 "Given/When/Then" 용어를 들이대지 않는다.

## 실행 방식 (독립 실행 가능, 세트로도 물림)

- **(세트 연계)** `easyproduct-suite`가 스케치(기획 직후)·파생(화면 설계 후)·검증(Stage 4) 시점에 호출한다.
- **(독립)** "이 서비스 흐름 시나리오 만들어줘"처럼 단독으로도 부른다. 기획서·IA가 있으면 읽어 재료로 쓰고,
  없으면 스케치 질문으로 흐름부터 그린다. 다른 스킬의 내부 파일을 전제하지 않고 **입력으로 받은 문서**만 참조한다.

## Step S1: 스케치 (흐름으로 화면을 끌어냄) — 옵트인

기획서가 확정된 직후, 서비스의 **핵심 여정**을 거칠게 그린다. 목적은 완성이 아니라 **IA·화면 도출을 이끄는 것**.

1. **핵심 여정을 원본에서 도출한다(판단으로 고르지 않는다).** 어떤 시나리오가 있어야 하는지는 임의로 정하지 말고,
   **기획서의 핵심 행동(보통 3개)과 IA의 핵심 흐름·유스케이스**에서 뽑는다. 즉 **핵심 행동마다 여정 하나, 핵심 유스케이스마다 여정 하나**를
   기본으로 잡아 목록을 만든다(예: 핵심 행동 "둘러보기·예약·픽업" → 여정 "둘러보기 → 예약·결제 → 픽업"). 이렇게 해야 뒤(S4)에서
   커버리지 누락을 원본 기준으로 검사할 수 있다. 원본에 없는 부수 흐름까지 억지로 만들지는 않는다.
2. **자연어로 스텝을 나열한다.** GWT를 강요하지 않는다. 분기(대안·예외)가 보이면 함께 적는다.
   - 이 단계에서 나온 **새 결정**(정책·데이터·화면 규칙)은 곧바로 **결정 이관 규칙**에 따라 원본 문서로 넘길 후보로 표시한다.
3. **흐름에서 필요한 화면·기능을 뽑아** IA/화면 설계로 넘긴다. ("이 여정을 위해 상품목록·예약·결제·픽업확인 화면이 필요")
4. 스케치는 `[상태: 탐색]`으로 표시하고, **폐기 가능**함을 분명히 한다(뒤에서 파생물로 대체된다).
5. **앞 게이트로 쓴다(조기 결함 발견).** 흐름을 끝까지 그리다 보면 "이 화면들이 안 이어진다 / 단계가 빠졌다 / 순서가 말이 안 된다"가
   드러난다. 화면을 짓기 **전**이라 고치는 비용이 가장 싸다. 결함이 보이면 **보고하고, 어디를 고쳐야 할지(기획/IA) 후보를 제시**하되
   **진행 여부는 사용자가 정한다**(→ 아래 "설계 결함 처리 원칙"). 시나리오가 스스로 기획/IA를 고치지 않는다.

> 무거운 단계라 **기본은 옵트인**이다. 다만 "핵심 행동이 여럿이거나 분기·상태 전이가 많은 복잡한 흐름"이면 **앞 게이트로 권장**한다
> (화면을 짓기 전에 흐름 결함을 값싸게 잡는 이점이 크다).

## Step S2: 결정 이관 (스케치 → 원본 문서)

스케치에서 정해진 결정을 **원본으로 옮긴다**. 이 스킬이 직접 원본을 고치기보다, **무엇을 어디로 옮겨야 하는지 목록**을 만들어
해당 스킬(doc-builder·ia-designer·policy-legal·data-model)로 넘긴다.

- 정책 성격 결정 → 정책서(`POL.*`). 화면·기능 구조 → IA(`FEAT.*`). 데이터 구조 → 데이터 모델(`그룹.필드`).
- 이관이 끝나기 전에는 시나리오를 "확정"으로 부르지 않는다. (결정이 시나리오에만 남아 있으면 안 된다.)

## Step S3: 파생 (확정본에서 재도출) — 읽기 전용

화면 설계까지 끝난 **확정 문서**를 재료로, 시나리오를 **GWT + 추적 링크**로 다시 뽑는다.

1. `assets/scenario-template.md`로 각 여정을 `SCN.{도메인}.{이름}`으로 작성한다.
2. 각 스텝을 **Given/When/Then**으로 쓰고, 닿는 ID를 `[FEAT.*]` `[그룹.필드]` `[POL.*]`로 **링크만** 단다(내용 복사 금지).
3. 문서 끝에 **추적표**(닿는 FEAT/데이터/정책/유스케이스 목록)와 **원본 링크**를 정리한다.
4. **mermaid flowchart**로 여정을 한눈에 보이게 그린다(GWT 스텝의 파생 뷰 — When=행동, Then=결과, 대안·예외=분기).
5. `[상태: 파생]`으로 표시한다. 원본이 바뀌면 이 문서는 **다시 뽑는다**(그림도 함께 다시 그린다).

## Step S4: 검증 (정합·추적 점검) — 이 스킬의 핵심 역할

시나리오가 다른 문서와 **어긋나지 않는지** 점검한다. 절차·판단 기준은 `references/traceability-check.md`.

핵심 4단계:
1. **링크 존재 검사(기계적·신뢰 가능)**: 각 스텝이 참조하는 `FEAT.*`·데이터 변수·`POL.*`(·유스케이스) ID가
   실제 문서에 **실재**하는가? 없으면 **죽은 링크**로 보고.
2. **정합 검사(의미·주의)**: 시나리오가 서술한 흐름/규칙이 원본과 **모순되지 않는가?**
   (예: 시나리오는 "승인 게이트 있음"인데 IA는 "셀프서비스"면 불일치.)
   — 이 검사는 확신할 수 있을 때만 "불일치"로 보고하고, **판단하기 어려운 부분은 솔직히 "미확인"으로 남긴다.**
   자동 통과를 "정합 확인됨"이라는 신호로 쓰지 않는다(거짓 안심 금지).
   — **불일치의 방향을 단정하지 않는다.** 시나리오가 낡았을 수도, **원본(화면 흐름) 자체가 결함**일 수도 있다.
   어느 쪽이 옳은지 시나리오가 판정하지 말고, 아래 "설계 결함 처리 원칙"에 따라 **양쪽 가능성과 회귀 후보를 제시**한다.
3. **커버리지 검사(필수·기계적)**: **기획서의 핵심 행동**과 **핵심 유스케이스**를 목록으로 놓고, 각 항목이
   **최소 하나의 시나리오에 닿는지** 대조한다. 어느 핵심 행동/유스케이스도 어떤 시나리오에도 안 닿으면 **누락으로 보고**한다
   (테스트·점검의 구멍이 되므로 선택이 아니라 반드시 확인). 부수 기능까지 전부 닿아야 하는 것은 아니다 — 기준은 **핵심 행동·핵심 유스케이스**다.
4. **재도출 트리거**: 원본이 바뀌었으면 해당 시나리오를 `파생` 상태에서 재생성한다.

- 결과는 **누락·불일치가 있을 때만** 조용히 보고한다(기존 완성도 점검과 같은 톤). 단 **커버리지 누락은 반드시** 보고한다.
- 검증이 할 수 있는 것과 없는 것을 분명히 한다: 죽은 링크·커버리지 누락은 확실히 잡지만, 의미적 모순은 **후보를 제시**할 뿐 최종 판정은 사람 몫이다.

### 설계 결함 처리 원칙 (게이트키퍼 역할 — 보고 후 사용자 선택)

시나리오를 작성·검증하다 흐름 결함(안 이어지는 화면·빠진 단계·논리 모순, 또는 원본과의 의미 불일치)이 드러나면,
시나리오는 **게이트키퍼**로서 이를 짚는다. 단 **시나리오는 SSOT가 아니므로 스스로 고치거나 어디가 틀렸다고 단정하지 않는다.**

1. **결함을 "설계 결함 후보"로 보고**한다 — 무엇이 어긋나는지, 그리고 **어디에서 비롯됐을 가능성이 큰지(회귀 후보)**를 함께 제시한다:
   - 핵심 행동·가치 자체가 어긋남 → **기획**(`easyproduct-doc-builder`) 회귀 후보
   - 흐름 논리·화면 누락·연결 문제 → **IA/화면 설계**(`easyproduct-ia-designer`/`easyproduct-screen-design`) 회귀 후보
   - 시나리오만 낡음(원본이 옳음) → 시나리오 **재도출**로 충분
2. **진행은 사용자가 정한다.** "이 결함을 고치러 ○○ 단계로 돌아갈까요, 아니면 시나리오만 맞출까요?"를 제시하고,
   사용자가 고른 쪽으로만 움직인다. **자동으로 재설계를 밀어붙이지 않는다.**
3. 사용자가 회귀를 택하면, 해당 스킬의 **재진입/부분 수정 절차**로 넘겨 원본을 고치게 하고(그 스킬이 자기 문서의 SSOT),
   원본이 고쳐지면 시나리오를 **재도출**한다. (세트 실행 중이면 `easyproduct-suite`의 리뷰 게이트 라우팅을 그대로 탄다.)

> 요약: 시나리오는 **결함을 조기·정확히 드러내는 게이트**이되, **결정하지 않고 제기·라우팅 제안만** 한다. 되돌릴지는 사람이 정한다.

## Step S5: 승격 (spec-kit 인계)

`파생`/`확정` 시나리오를 spec-kit의 spec.md **"Acceptance Scenarios"**에 그대로 매핑할 수 있게 정리한다.
GWT 형식이라 별도 변환이 거의 필요 없다. SSOT 구분표에 "시나리오 = 보강자료 → **인수 시나리오로 승격**"을 한 줄로 남긴다.

## 재진입 — 부분 수정·재도출

이미 만든 시나리오를 고칠 때:
- **원본이 바뀐 경우**: 해당 시나리오를 처음부터 다시 쓰지 말고, **바뀐 원본을 참조해 재도출**한다(링크-온리라 비용이 낮다).
- **부분 수정**: 지정되지 않은 스텝·추적 링크는 글자 그대로 둔다. `SCN.*` ID는 유지한다.
- **상태 배지**를 항상 갱신한다(`탐색`↔`파생`↔`확정`).
