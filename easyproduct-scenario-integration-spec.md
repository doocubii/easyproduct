# easyproduct 스킬 세트 개정 명세 — "시나리오"를 1급 산출물로 추가하기

> **이 문서의 독자**: easyproduct 스킬 세트(오케스트레이터 `easyproduct-suite` + 하위 스킬들)를 **제작·수정하는 에이전트/개발자**.
> **이 문서의 목적**: 현재 세트에 **없는 "시나리오"를 1급 산출물로 추가**하고, 시나리오와 그 파생 문서(IA·데이터·화면·유스케이스·정책) 사이의 **정합·추적을 검증하는 역할을 신설**하기 위한 요구사항·설계·스킬별 변경 지시.
> **이 문서 자체는 SSOT가 아니다** — 스킬 개정을 위한 인계장(제안서)이다. 스킬에 반영되면 이 문서의 역할은 끝난다.

---

## 0. 한 줄 요약
easyproduct 세트는 지금 "시나리오"를 만들지 않는다(의도적으로 spec-kit에 미룸). 그 결과 **요구사항↔설계↔테스트를 잇는 흐름의 축이 없고, 문서 간 어긋남(consistency)을 검증하는 주체도 없다.** 이 명세는 시나리오를 **"추적성의 축(traceability spine)"**으로 세트에 추가하고, 그 정합을 검증하는 스텝을 신설하는 방법을 규정한다.

---

## 1. 현재 상태 (조사 결과 — 무엇이 없는가)

### 1.1 시나리오는 아무도 만들지 않는다 (의도된 공백)
- `easyproduct-doc-builder` SKILL.md: **"수용 시나리오를 따로 묻지 않는다 … Given/When/Then 표기를 쓰지 않는다"** — 비전문가용이라 GWT를 **의도적으로 배제**.
- `easyproduct-suite`의 handoff 자료: 시나리오·수용 기준은 **spec-kit(개발 단계)이 세울 때** 유스케이스·유저스토리를 참조한다는 전제. 즉 시나리오는 **세트 다음 단계로 미뤄져** 있다.
- 대체물(가장 가까운 것):
  - **유저 스토리** — `easyproduct-doc-builder`가 Stage 1에서, `supporting/user-stories.md`.
  - **유스케이스** — `easyproduct-ia-designer`가 Stage 2에서, `supporting/use-cases.md`(주 흐름·대안·예외 서술, GWT 없음).
  - 둘 다 "**보강자료** — 다른 SSOT를 검증 가능하게 가공, 원본과 다르면 **원본 우선**". 개별 요구의 *예시*이지 흐름의 *축*이 아니다.

### 1.2 파생 관계를 검증하는 전용 주체가 없다 ("누락 점검"만 있고 "어긋남 검증"은 없다)
세트의 정합 장치 셋, 모두 약하다:
1. **Stage 4 완성도 점검**(`easyproduct-suite`) — `easyproduct-doc-builder/references/speckit-fields.md` 체크리스트로 "spec-kit이 뽑을 내용이 세트에 **빠짐없이** 있나"를 조용히 확인. → **누락(completeness)** 점검이지 **어긋남(consistency)** 검증이 아니다.
2. **상호 링크 + back-fill**(Stage 4, 재진입 시 3.7) — 문서 머리에 "이 세트의 다른 문서" 링크, 설계서 빈자리 채움. 고치면 "어긋나지 않게 다시 맞춘다"지만 **사람이 수동으로**, 규칙 기반 자동 검증 아님.
3. **기능 관계도**(`easyproduct-ia-designer` 선택 모듈) — IA에서 도출하는 **읽기 전용 파생물**. "IA가 바뀌면 다시 도출". → 세트에서 **"파생물은 원본에서 재생성"을 명시한 유일한 장치**. 하지만 시나리오엔 적용 안 됨(시나리오가 없으니).

### 1.3 결론
"기획 → 시나리오 → (IA·데이터·화면·유스케이스·정책) → 테스트"의 **축과 그 정합 검증**은 **현재 세트에 존재하지 않는 신규 역할**이다. 이 명세가 그 공백을 메운다.

## 2. 근거 방법론 (왜 시나리오가 축이어야 하는가)

| 방법론 | 시나리오의 위상 | 세트에 주는 함의 |
|---|---|---|
| **Use Case 2.0** (Jacobson) | use-case slice = 요구·설계·테스트·계획을 잇는 **수직 슬라이스**, "개발을 구동" | 시나리오가 IA·화면을 **끌어낼 수 있다**(드라이버) |
| **Specification by Example** (Adzic) | 예제(시나리오)가 **명세이자 테스트인 단일 소스**, living documentation | 확정 시나리오는 그대로 **검증 자산** |
| **BDD / Gherkin** | Given/When/Then = 인수 기준 = 자동화 테스트, **요구→검증 추적성** | 시나리오 형식을 **GWT + 추적 링크**로 |

## 3. 시나리오의 위상 정의 (반드시 스킬에 명문화할 원칙)

1. **시나리오는 "흐름의 축"이되 "결정의 SSOT"가 아니다.** 결정(정책·화면·데이터 구조)의 원본은 여전히 기획서·IA·정책서·데이터 모델. 시나리오는 그것들을 **ID로 꿰는 실** + **테스트의 씨앗**.
2. **두 거버넌스 규칙** (이 둘이 SSOT를 지키는 핵심):
   - **결정 이관 규칙**: "시나리오(특히 스케치)에서 나온 결정은 반드시 원본 문서(IA·기획서·정책·데이터)로 이관한다. 시나리오에만 남기지 않는다."
   - **링크-온리 규칙**: "확정 시나리오는 원본 내용을 베끼지 않고 **링크(ID 참조)**만 한다. 원본이 바뀌면 시나리오를 재도출한다." (기능 관계도·JSON 스냅샷과 동일한 읽기 전용 파생 원칙)
3. **spec-kit 인계 시 시나리오는 "보강자료"에서 "인수 시나리오"로 승격**한다(SSOT 구분표에 명시).

---

## 4. 생애주기 (언제·누가 — 스킬/스테이지 매핑)

```
① 스케치(탐색)   → 기획 직후. 흐름으로 IA·화면을 끌어냄. 자유·반복. (임시, 폐기 가능)
② 이관(확정)     → 스케치에서 나온 결정을 IA·기획서·정책·데이터로. (기존 스킬들이 SSOT 확정)
③ 파생(정리)     → 화면 설계 후. 확정본을 참조해 재도출. supporting/scenarios/ (읽기 전용 파생물)
④ 승격(테스트)   → spec-kit 인계 시 Given/When/Then 인수 시나리오로.
      +
⑤ 검증(정합)     → Stage 4에서 시나리오↔파생 링크의 존재·정합을 점검(신설 역할).
```

**중요한 설계 판단**: 시나리오는 세트에 **두 번** 등장한다 — 앞의 **스케치(드라이버)**와 뒤의 **파생물(검증)**. 같은 문서가 단계에 따라 역할이 바뀐다. 상태 배지(`탐색`/`확정`/`파생`)로 구분한다.

---

## 5. 표준 형식·산출물 규격

- **위치**: `supporting/scenarios/scenario-{도메인}-{이름}.md` (유스케이스·유저스토리 옆).
- **ID 체계**: 시나리오 `SCN.{도메인}.{이름}`, 스텝은 그 아래 순번. FEAT·데이터 변수·정책 ID와 **겹치지 않는 접두어**.
- **형식**: Given/When/Then + **스텝별 추적 링크**. 예:
  ```
  ## SCN.org.selfServeOnboarding  [상태: 파생]
  - 목적 / 행위자 / 선행조건
  - Given 로그인한 사용자가 조직이 없다        → [FEAT.auth.signup] [account.type]
  - When  "조직 만들기"를 한다                 → [FEAT.org.create] [organization, membership.role]
  - Then  본인이 관리자인 조직이 생성된다      → [membership.role=관리자]
  - When  무료 한도를 넘겨 검토 실행을 시도    → [결제 게이트] [POL.payment.publicSector]
  - Then  결제/견적 경로로 유도                → [FEAT.billing.*] [FEAT.sales.quote]
  - 추적표: 이 시나리오가 닿는 FEAT / 데이터 / 정책 / 유스케이스 ID 목록
  - 원본 링크: 각 결정의 SSOT 문서 위치(베끼지 않고 가리키기만)
  ```
- **비전문가 조화**(중요, `doc-builder`의 "GWT 안 쓴다"와 충돌 해결):
  - **스케치 단계(①)**: 자연어 흐름 허용(GWT 강요 안 함) — 기존 비전문가 원칙 유지.
  - **파생·승격 단계(③④)**: GWT 구조 + 추적 링크로 정형화(개발/테스트가 쓰므로).
  - 즉 "GWT를 언제 쓰나"를 **단계로 분리**해 doc-builder 원칙과 모순되지 않게 한다.

## 6. 스킬별 변경 지시 (구체)

### 6.1 `easyproduct-suite` (오케스트레이터) — 변경 대부분이 여기
- **Stage 신설 A — 시나리오 스케치**: Stage 1(기획) 직후, Stage 2(IA) **전 또는 병행**에 배치. 기획서의 핵심 흐름을 시나리오 슬라이스로 스케치해 IA·화면 도출을 이끈다. 기본 옵트인(무거우므로), 단 "흐름이 복잡한 서비스"면 권장.
- **Stage 신설 B — 시나리오 파생(정리)**: Stage 3.8-C(기능 관계도) 옆에 배치. 화면 설계까지 끝난 확정본에서 시나리오를 **읽기 전용 파생물**로 재도출.
- **Stage 4 확장 — 정합·추적 점검(핵심)**: 기존 "완성도 점검"에 다음을 **추가**:
  - 각 시나리오 스텝의 참조 ID(FEAT·데이터 변수·정책)가 **실제 문서에 존재**하는지.
  - 시나리오가 가리키는 결정이 **원본과 어긋나지 않는지**(링크-온리 위반·낡은 서술 탐지).
  - 원본이 바뀌었으면 **재도출 필요**를 표시.
- **색인·인계장 반영**: `00-index.md`에 시나리오 항목 추가(참조용 파생물). `speckit-handoff` SSOT 구분표에 "시나리오 = 보강자료 → spec-kit 인수 시나리오로 승격" 한 줄.

### 6.2 신설 vs 모듈 — 두 안 (택1, §9에서 결정)
- **(a) 전담 스킬 `easyproduct-scenario` 신설**: 스케치·파생·검증을 담당. 유스케이스/유저스토리와 역할 경계를 명확히(예시 vs 흐름축).
- **(b) 기존 스킬에 모듈로 분산**: 스케치=`doc-builder` 선택 모듈, 파생·검증=`ia-designer`/`suite`. 신규 스킬 없이.
- 권장: **(a)** — 시나리오는 여러 문서를 가로지르는 횡단 관심사라 전담 스킬이 경계가 깔끔하다.

### 6.3 `easyproduct-doc-builder`
- "GWT 안 쓴다" 원칙을 **"스케치 단계에 한함"**으로 한정 수정. 파생/승격 단계 GWT는 시나리오 스킬 소관.
- 유저 스토리와 시나리오의 관계 한 줄: 유저 스토리=단일 요구의 가치 서술, 시나리오=여러 요구를 잇는 흐름.

### 6.4 `easyproduct-ia-designer`
- 유스케이스와 시나리오의 관계 정리: 유스케이스=기능 단위 흐름(주/대안/예외), 시나리오=**여러 기능·화면을 관통하는 사용자 여정**(더 큰 단위). 중복 서술 금지, 시나리오는 유스케이스를 **링크**.
- 기능 관계도와 시나리오는 **같은 읽기 전용 파생 원칙**을 공유함을 명시.

### 6.5 `easyproduct-screen-design`
- "시나리오가 있으면 화면 도출의 **드라이버**로 삼는다"는 한 줄 추가(흐름 → 화면). 단 화면의 SSOT는 여전히 화면 설계서 글 정의.

### 6.6 `easyproduct-policy-legal` · `easyproduct-data-model`
- 변경 최소. 시나리오가 정책 ID·데이터 변수를 **참조**하므로, 정책·데이터에 **안정적 ID**가 있어야 함을 확인(정책 항목에 `POL.*` 같은 참조 가능한 ID 부여 권장).

### 6.7 handoff / spec-kit 인계
- 시나리오를 **Given/When/Then 인수 시나리오**로 넘기는 절차 추가. spec-kit의 spec.md "Acceptance Scenarios"에 그대로 매핑됨을 안내.

## 7. 검증 역할 상세 (누가 관계를 검증하나 — 신설 핵심)

**Stage 4의 "시나리오 정합·추적 점검"** 알고리즘(사람이 읽어도, 에이전트가 자동화해도 되는 절차):
1. **링크 존재 검사**: 각 시나리오 스텝이 참조하는 `FEAT.*`·데이터 변수·`POL.*`·`UC.*` ID가 IA·데이터 모델·정책·유스케이스에 **실재**하는가? (죽은 링크 탐지)
2. **정합 검사**: 시나리오가 서술한 흐름/규칙이 원본 결정과 **모순되지 않는가?** (예: 시나리오는 "승인 게이트 있음"인데 IA는 "셀프서비스"면 불일치 → 시나리오 재도출)
3. **커버리지 검사**(선택): 핵심 FEAT/유스케이스가 **어떤 시나리오에도 안 닿으면** 흐름 공백 후보로 표시.
4. **재도출 트리거**: 원본이 바뀌면 해당 시나리오를 `파생` 상태에서 재생성. (링크-온리라 재생성 비용 낮음.)
- 이 점검 결과는 사용자에게 조용히(누락·불일치가 있을 때만) 보고. 기존 완성도 점검과 같은 톤.

---

## 8. 이 개정으로 얻는 것 (수용 기준)
- 세트가 **흐름의 축**을 산출한다(지금은 없음).
- 문서 간 **어긋남을 검증하는 주체**가 생긴다(지금은 누락 점검만).
- 시나리오가 **화면 설계를 드라이브**할 수 있고(선택), **spec-kit 테스트로 승격**된다.
- SSOT 원칙은 유지된다(결정 이관·링크-온리 규칙).
- 비전문가 원칙과 충돌하지 않는다(GWT는 파생·승격 단계에서만).
