# easyproduct — 비개발자를 위한 개발 착수 전 문서 자동화 Claude Skills

[![License: Apache 2.0](https://img.shields.io/badge/License-Apache%202.0-blue.svg)](https://github.com/doocubii/easyproduct/blob/main/LICENSE)
Repository: [github.com/doocubii/easyproduct](https://github.com/doocubii/easyproduct)

개발 지식이 전혀 없는 사람도 **대화만으로** 아이디어 하나를 완전한 개발 착수 전 문서 세트로
완성할 수 있게 돕는 [Claude Skills](https://docs.claude.com) 모음입니다. 오케스트레이터
스킬(`easyproduct-suite`)이 7개의 전문 스킬을 순서·시점에 맞게 조율해, 기획서부터 화면 설계서·시나리오까지
한 흐름으로 만들어 줍니다. 모든 산출물은 **한국어**로 작성됩니다.

> 이 저장소는 [GitHub Spec Kit](https://github.com/github/spec-kit)(spec-kit) 자체가 아닙니다.
> spec-kit은 "명세를 근거로 AI가 코드를 구현하는" 별도의 오픈소스 도구(MIT 라이선스)이고,
> 이 저장소의 스킬들은 **spec-kit을 쓰기 전 단계**, 즉 "무엇을 만들지"를 비개발자도 정리할 수
> 있게 돕는 문서화 도구입니다. 마지막 단계에서 만든 문서 세트를 spec-kit에 넘길 수 있는
> 인계 안내문을 생성해 주지만, spec-kit 설치·사용법은 이 저장소의 범위 밖입니다.

---

## 왜 만들었나

`spec-kit` 같은 Spec-Driven Development(SDD) 도구는 강력하지만, **"무엇을 만들지"를 정리한
명세가 먼저 있어야** 제대로 작동합니다. 기획서, 화면 구조, 데이터 모델, 디자인 방향, 정책·약관
같은 문서를 처음부터 다 갖추고 시작하는 비개발자는 많지 않습니다. 이 스킬 세트는 그 간극을
메웁니다 — "이런 서비스를 만들고 싶다"는 한 문장에서 시작해, 개발에 바로 넘길 수 있는 문서
세트가 나올 때까지 대화로 이끕니다.

**비개발자가 만들지만, 결과물은 개발자·소프트웨어도 받습니다.** 대화의 주인공은 비개발자지만,
나오는 산출물(데이터 모델·화면 명세·정책, 그리고 `0.3.0`에서 더해진 **기계 판독 층** — 아래 "소프트웨어·하네스가 읽고 점검하는 층" 참고)은
**개발자와 소프트웨어가 그대로 소비**할 수 있는 형태입니다. spec-kit 같은 SDD 도구로 넘기는 것이
기본 경로지만, **spec-kit 없이도** 이 문서들의 스키마·점검기·계약서만으로 하네스 엔지니어링(기계
판독 스펙 + 검증)의 기반을 도입할 수 있습니다 — 완성된 하네스를 주는 것이 아니라, **하네스가 소비할
재료와 참조 구현**을 제공합니다.

## 무엇을 만드나

기본 세트(자동으로 포함):
- **기획서 · 설계서** — 무엇을·왜 만드는지, 어떻게 만들지
- **화면·기능 구조(IA)** — 어떤 화면·기능이 있는지 (프론트 + 관리자), 기능마다 고유 ID 부여
- **데이터 모델** — 서비스가 다루는 정보의 그룹·필드 (각 그룹 표 아래에 소프트웨어가 바로 읽는 JSON 미러 포함)
- **디자인 컨셉** — 색·글꼴·컴포넌트 스타일 3안 + 미리보기 이미지
- **정책서 · 이용약관 · 개인정보처리방침** — 운영 규칙 + 법적 문서 초안(전문가 검토 전제)
- **유저 스토리 · 유스케이스** — 수용 기준·시나리오 (보강 자료)

옵트인(원할 때만):
- **화면 설계서** — 화면마다 컴포넌트·배치·문구까지 고정한 구현 명세(사용자 앱/관리자 백오피스
  별도 프로젝트로 분리), 재사용 UI 컴포넌트 인벤토리 포함
- **시나리오** — 여러 화면·기능·정책을 관통하는 사용자 여정(흐름의 축). 각 스텝이 닿는
  기능·데이터·정책을 ID로 링크해 문서 간 정합·추적을 점검하고, spec-kit 인수 시나리오로 승격
- **기능 관계도** — 프론트↔백오피스 기능이 서로 어떻게 트리거하는지 도식화
- **spec-kit 인계 안내문** — 완성된 문서 세트를 spec-kit(또는 다른 SDD 도구)에 넘기기 위한 요약

## 포함된 스킬

| 스킬 | 역할 | 단독 실행 |
|---|---|---|
| [`easyproduct-suite`](./skills/easyproduct-suite/) | 아래 7개를 순서·시점에 맞게 조율하는 오케스트레이터 | — |
| [`easyproduct-doc-builder`](./skills/easyproduct-doc-builder/) | 기획서 + 설계서 작성 | 가능 |
| [`easyproduct-ia-designer`](./skills/easyproduct-ia-designer/) | 화면·기능 구조(IA), 기능 ID 체계, 네비게이션 설계 원칙 | 가능 |
| [`easyproduct-data-model`](./skills/easyproduct-data-model/) | 데이터 모델(정보 그룹·필드) 생성·수정·조회 | 가능 |
| [`easyproduct-design-concept`](./skills/easyproduct-design-concept/) | 디자인 컨셉 3안 + 미리보기 생성 | 가능 |
| [`easyproduct-policy-legal`](./skills/easyproduct-policy-legal/) | 정책서 + 이용약관·개인정보처리방침 초안 | 가능 |
| [`easyproduct-screen-design`](./skills/easyproduct-screen-design/) | 화면별 상세 설계서(구현 명세 수준) | 가능 |
| [`easyproduct-scenario`](./skills/easyproduct-scenario/) | 사용자 여정 시나리오(흐름의 축) 작성 + 문서 간 정합·추적 점검 | 가능 |

각 스킬은 **독립적으로도 동작**하도록 설계되어 있습니다 — 서로의 내부 파일을 직접 참조하지
않고, 필요한 재료(기획서, IA, 데이터 모델 등)를 입력으로 받아 동작합니다. `easyproduct-suite`는
이 스킬들을 순서대로 호출하고 결과를 이어붙이는 조율자일 뿐, 자체 로직을 스킬 안에 중복 구현하지
않습니다.

## 어떻게 동작하나 (전체 흐름)

```
기획서·설계서 → 화면 구조(IA) → 데이터 모델 → 디자인 컨셉 → 정책·약관
        ↓
   [게이트] 디자인 확정 + 문서 전체 검토 + 필요하면 개별 문서만 재수정
        ↓ (옵트인으로 질문)
   화면 설계서 (사용자 앱 → 필요시 관리자 백오피스 별도) → 기능 관계도 → 시나리오(파생)
        ↓ (Stage 4: 시나리오가 있으면 문서 간 정합·추적 점검)
        ↓ (옵트인으로 질문)
   spec-kit 인계 안내문
```

> 시나리오는 반대로 **기획 직후 스케치**해 화면·기능 도출을 이끄는 드라이버로도 쓸 수 있습니다(옵트인).
> 이때 스케치에서 나온 결정은 반드시 원본 문서(IA·정책 등)로 옮긴 뒤 진행합니다.

- **모드 두 가지**: 가이드 모드(단계마다 확인받으며 진행) / 자동 모드(기획서 확정과 디자인
  확정, 두 게이트만 지키고 나머지는 합리적으로 판단해 무중단 진행).
- **이미 만든 문서가 있으면** 있는 것은 재사용하고 없는 것만 채웁니다. 예전 구조로 만든 문서는
  "버전업 업그레이드" 절차로 최신 구조에 맞춰 gap만 채웁니다(처음부터 다시 만들지 않음).

## 설치 방법

**claude.ai**: Settings → Capabilities에서 각 스킬의 `.skill` 파일을 업로드합니다. 세트를
한 번에 쓰려면 `easyproduct-suite`를 반드시 포함하고, 나머지 7개도 함께 설치해야 오케스트레이터가
호출할 수 있습니다.

> `.skill` 파일은 이 저장소에 포함돼 있지 않습니다 — 각 스킬 폴더(`skills/easyproduct-*`)를
> 통째로 zip으로 묶은 뒤 확장자를 `.zip` → `.skill`로 바꾸면 업로드용 파일이 됩니다.

**Claude Code**: 이 저장소를 클론한 뒤 각 스킬 폴더를 전역 스킬 폴더 또는 프로젝트 전용 폴더에
복사합니다. Claude Code는 실제 파일시스템 위에서 동작하므로, 스킬이 만든 문서는 작업 중인 프로젝트
폴더에 그대로 파일로 저장되고 세션이 끝나도 남습니다.

- **전역 스킬 폴더** — macOS/Linux는 `~/.claude/skills/`, Windows는 `%USERPROFILE%\.claude\skills\`
  (= `C:\Users\<사용자명>\.claude\skills\`). 어느 프로젝트에서든 쓰입니다.
- **프로젝트 전용** — 작업 폴더 안의 `.claude/skills/`. 그 프로젝트에서만 쓰입니다.

**간편 설치 스크립트 (권장):** 저장소 루트의 설치 스크립트가 8개 스킬을 한 번에 복사합니다.
`.claude/skills` 폴더가 없으면 만들어 주고, 이미 있는 스킬은 최신 내용으로 갱신합니다.

```bash
git clone https://github.com/doocubii/easyproduct.git
cd easyproduct

# macOS / Linux / Git Bash
./install.sh                      # 대화형: "전역(~/.claude)에 설치할까요? [Y/n]"
                                  #   Y/엔터 → 전역(~/.claude/skills)
                                  #   n      → 프로젝트 폴더 경로를 물어 <폴더>/.claude/skills 에 설치
./install.sh -y                   # 컨펌 없이 전역에 설치
./install.sh /path/to/project     # 컨펌 없이 특정 폴더에 설치 → <폴더>/.claude/skills ('.' = 현재 폴더)
```

```powershell
# Windows PowerShell
.\install.ps1                            # 대화형: "전역(~\.claude)에 설치할까요? [Y/n]" (n이면 폴더 경로 입력)
.\install.ps1 -Yes                       # 컨펌 없이 전역에 설치
.\install.ps1 -Base C:\path\to\project   # 컨펌 없이 특정 폴더에 설치 → <폴더>\.claude\skills
```

아래는 스크립트 없이 **손으로 복사**하는 방법입니다.

**macOS / Linux (bash):**

```bash
git clone https://github.com/doocubii/easyproduct.git
# 각 스킬 폴더(easyproduct-suite, easyproduct-doc-builder 등)를 아래 위치로 복사
cp -r easyproduct/skills/<스킬-폴더명> ~/.claude/skills/       # 전역 설치
# 또는  cp -r easyproduct/skills/<스킬-폴더명> ./.claude/skills/   # 프로젝트 전용 설치
```

**Windows (PowerShell):**

```powershell
git clone https://github.com/doocubii/easyproduct.git
# 전역 스킬 폴더가 없으면 한 번 만든다
New-Item -ItemType Directory -Force "$env:USERPROFILE\.claude\skills" | Out-Null
# 각 스킬 폴더를 전역에 복사 (스킬마다 반복, 또는 skills\* 로 전체 복사)
Copy-Item -Recurse -Force ".\easyproduct\skills\<스킬-폴더명>" "$env:USERPROFILE\.claude\skills\"
# 또는 프로젝트 전용으로 복사
Copy-Item -Recurse -Force ".\easyproduct\skills\<스킬-폴더명>" ".\.claude\skills\"

# 8개 스킬을 전역에 한 번에 복사하려면:
Copy-Item -Recurse -Force ".\easyproduct\skills\*" "$env:USERPROFILE\.claude\skills\"
```

> 세트로 쓰려면 `easyproduct-suite`를 포함해 8개 폴더를 모두 복사하세요. 복사 후 새 Claude Code
> 세션을 시작하면 스킬이 인식됩니다.

## 사용 예시

아래는 실제로 나눌 법한 대화를 재구성한 예시입니다. 스킬 이름이나 명령어를 몰라도, 그냥
평소 말투로 아이디어를 던지는 것으로 시작합니다.

### 예시 1 — 가이드 모드: 처음부터 차근차근

```
사용자: 동네 카페·식당 사장님이 마감 세일 상품을 등록하면, 근처 사람들이 예약하고
       픽업해가는 앱을 만들고 싶어요.

Claude: 좋은 아이디어네요! 개발 착수 전에 필요한 문서들을 대화로 함께 정리해 볼게요.
       기술 용어는 몰라도 괜찮습니다.

       먼저 진행 방식을 정할게요 — 제가 단계마다 여쭤보고 확인받으며 차근차근 갈까요,
       아니면 제가 합리적으로 판단해서 빠르게 초안부터 쭉 만들고 나중에 고치는 쪽이 편하실까요?

사용자: 처음이라 차근차근 갈게요.

Claude: 네, 그렇게 할게요. 몇 가지만 여쭤볼게요.

       이 서비스를 한 문장으로 표현하면, "누구를 위한 무엇"이라고 할 수 있을까요?

사용자: 마감 임박 재고를 버리기 아까운 소상공인과, 그걸 싸게 사고 싶은 동네 주민을
       이어주는 앱이요.

Claude: 좋습니다. 그럼 사용자가 이 앱을 켰을 때 가장 먼저, 가장 자주 하게 될 행동은
       뭘까요?

사용자: 내 주변에 지금 올라온 마감 할인 상품을 둘러보는 거요.

Claude: 그다음으로 꼭 필요한 행동도 있을까요? (예: 예약, 픽업 확인 같은 것들이요)

사용자: 예약하고 결제하고, 나중에 가게 가서 픽업 확인하는 거요.

Claude: 정리하면 핵심 흐름은 "둘러보기 → 예약·결제 → 픽업"이네요. 이 세 가지만 되면
       일단 서비스로서 쓸 만할까요, 아니면 다른 게 더 있어야 할까요?

사용자: 그 정도면 될 것 같아요.

Claude: [기획서 초안의 나머지 항목들도 같은 방식으로 몇 개 더 여쭤본 뒤]

       이 정도면 기획서 핵심이 다 모인 것 같아요. 정리해서 보여드릴게요 —

       ── 기획서 요약 ──
       한 줄 정체성: 마감 할인 재고를 동네 주민과 예약·픽업으로 잇는 위치 기반 앱
       핵심 행동: ① 주변 할인 둘러보기 ② 예약·결제 ③ 픽업 확인
       성공 기준: [함께 정한 기준]
       가정: [사용자가 명시하지 않아 임의로 채운 항목들]

       이대로 맞을까요? 고치고 싶은 부분 있으세요?

사용자: 네 맞아요, 진행해주세요.

Claude: 기획서를 확정했습니다. 이어서 설계서를 만들고, 화면 구조(IA)로 넘어갈게요.
       [이후 화면 구조 → 데이터 모델 → 디자인 컨셉 3안 제시 → 정책·약관까지
       비슷한 방식으로 대화가 이어집니다]
```

### 예시 2 — 자동 모드: 빠르게 초안부터

```
사용자: 프리랜서 디자이너와 소상공인을 이어주는 매칭 플랫폼을 만들려고 해요.
       질문은 최소한으로 하고 알아서 초안부터 쭉 만들어주세요.
```

### 예시 3 — 재진입: 이미 만든 문서 업그레이드

```
사용자: [예전에 만든 IA 문서와 화면 설계서 파일 첨부]
       예전에 만든 문서들인데, 최신 버전으로 업그레이드해줘.

Claude: 네, 처음부터 다시 만들지 않고 지금 버전 구조와 비교해서 빠진 부분만 채울게요.

       IA 문서를 확인해 보니 각 기능에 고유 ID(예: FEAT.member.search)가 아직 없어서
       추가했습니다. 화면 설계서에는 "로그인 필요 여부" 필드가 빠져 있어서, 각 화면을
       보고 판단해 채워 넣었습니다 — 예: 마이페이지는 "필수", 상품 목록은 "무관".

       기존에 이미 있던 내용(화면 구성, 데이터 정의 등)은 전혀 건드리지 않았습니다.
       화면 12개 중 로그인 필수 3개, 무관(콘텐츠 분리) 2개, 무관(동일) 7개로 정리됐습니다.
       확인해 보시겠어요?
```

이처럼 스킬 이름이나 옵션을 몰라도, **자연스러운 문장으로 요청하면 알아서 적절한 스킬과 모드를
선택**해 진행합니다. 가이드/자동 모드 선택, 옵트인 기능(화면 설계서·spec-kit 인계) 여부는
언제든 대화 중에 바꿀 수 있습니다.

## 산출물 폴더 구조

```
[프로젝트명]-docs/
├── 00-index.md          ← 문서 세트 지도 (스냅샷, SSOT 아님, .gitignore 대상)
├── .gitignore
├── ssot/                ← 결정이 사는 곳 — 기획서·설계서·IA·데이터모델·디자인컨셉·정책·약관 등
├── supporting/          ← 보강 자료 — 유저 스토리·유스케이스·시나리오(scenarios/)
├── reference/           ← 파생이지만 보존 — 기능 관계도·디자인 3안 비교
├── screens/{user,backoffice}/  ← 화면 설계서(도메인별 파일 + 공통 UI 정의)
└── temp/                ← 확인 후 버려도 되는 것 — 화면 목업 HTML (.gitignore 대상)
```

각 문서가 **SSOT(원본)**인지, 다른 문서를 검증 가능한 형태로 가공한 **보강 자료**인지, 아니면
**참조용 파생물**인지는 문서 안에, 그리고 spec-kit 인계 안내문 안에 표로 명시됩니다. 문서끼리
내용이 어긋나 보이면 이 표를 기준으로 판단하면 됩니다.

## 명명 규칙 (문서 간 정확한 참조를 위해)

- **기능/화면 ID**: `FEAT.도메인.기능명[.화면종류]` — 예: `FEAT.member.search.list`. IA가
  확정하고, 화면 설계서는 이 ID로 정확히 인용합니다(이름만으로는 "찾기" 같은 동명이인이 헷갈릴
  수 있어 도입).
- **UI 컴포넌트 ID**: `UI.카테고리.컴포넌트명[.variant]`(재사용 컴포넌트) 또는
  `UI.FEAT.도메인.기능명.컴포넌트명`(특정 화면 전용 컴포넌트).
- **데이터 변수**: `그룹.필드`(예: `user.nickname`) — 데이터 모델 문서가 유일한 원본입니다.
- **정책 규칙 ID**: `POL.도메인.규칙명`(예: `POL.payment.refund`) — 정책서가 유일한 원본입니다.
- **유스케이스 ID**: `UC.도메인.목표`(예: `UC.meeting.create`) — IA(유스케이스)가 원본이며, 시나리오가 목표를 ID로 참조합니다.
- **시나리오 ID**: `SCN.도메인.이름`(예: `SCN.deal.reservePickup`), 스텝은 그 아래 순번 — 시나리오가 여러 기능·정책·데이터를 관통하는 여정을 식별합니다.
- **참조 표기 `이름(ID)`**: 위 ID로 링크를 걸 때는 **사람이 읽는 이름과 ID를 함께** 적습니다 —
  예: `메인 홈(FEAT.home.landing)`, `환불·취소 기준(POL.payment.refund)`. ID만 있으면 가독성이 떨어지고 이름만 있으면 참조가
  깨지므로 둘 다 둡니다. 이 표기는 **새로 만들거나 수정하는 문서에만** 적용하며, 기존 문서가 이 형식이 아니어도 잘못된 것은 아닙니다.

## 소프트웨어·하네스가 읽고 점검하는 층 (`0.3.0` 추가)

이 스킬들이 만드는 문서는 이제 **사람이 읽는 층 + 소프트웨어가 읽는 층**을 함께 담습니다. 사람은
표·산문을 읽고, 하네스·CI 같은 소프트웨어는 아래 기계 층을 읽어 **스펙과 문서 간 참조를 규약으로
조회·점검**할 수 있습니다. 사람용 문서 모양은 그대로이고, 기계 층은 문서 안/옆에 조용히 얹힙니다.

- **frontmatter** (문서 맨 앞 메타) — `doc_type`·`version`·`ssot`·`machine.*`로 "이게 무슨 문서이고,
  기계용 블록을 어디서·어떤 계약으로 찾는지"를 선언합니다.
- **태그된 기계 블록** — 사람용 표·산문을 옮긴 JSON 블록(예: 데이터 모델 `datamodel.group`,
  IA `ia.features`, 정책서 `policy.rules`, 화면 설계 `screendesign.screens`, 디자인 `design.tokens`,
  시나리오 `scenario.trace`). **사람용 표현이 원본(SSOT)이고 이 블록은 그 미러**입니다.
- **JSON Schema** — 각 문서 옆 `schemas/*.v1.schema.json`에 검증 계약이 함께 나옵니다(버전을 파일명에
  박아, `v1`로 쓴 문서는 언제나 `v1`으로 검증).
- **크로스도큐먼트 참조 라우팅** — 위 "명명 규칙"의 접두사(`FEAT./DATA./POL./UI./SCN.`)가 곧 "어느
  등기부에서 조회할지"의 타입 표시라, 문서 수십 개에 흩어진 참조가 실제로 존재하는지
  (끊긴 참조 = 죽은 링크) 소프트웨어가 자동 확인합니다.

**점검기와 개발 가이드.** `easyproduct-suite`는 문서 세트를 점검할 때 딸린 무의존 점검기
[`skills/easyproduct-suite/scripts/check-docs.mjs`](./skills/easyproduct-suite/scripts/check-docs.mjs)(Node,
npm 불필요)로 **스키마 검증 + 크로스도큐먼트 참조 무결성**을 확인합니다. 직접 점검기·CI를 만들려면
필드 의미·`doc_type`/anchor 레지스트리·참조 라우팅·점검 알고리즘을 규약으로 정리한
[`references/checker-guide.md`](./skills/easyproduct-suite/references/checker-guide.md)를 보세요. (점검기
소스는 프로젝트로 복사되지 않으며, 필요하면 요청 시 제공됩니다.)

**spec-kit 없이도 됩니다.** 이 기계 층은 spec-kit과 독립입니다 — 완성된 문서 세트를 spec-kit(SDD)에
넘겨도 되고, 넘기지 않고 이 **스키마·점검기·계약서(checker-guide)**만으로 자체 하네스·CI를 붙여
스펙을 검증해도 됩니다. spec-kit은 여러 다운스트림 중 하나일 뿐, 전제 조건이 아닙니다.

> 옛 문서도 각 스킬의 **업그레이드(gap-fill)**로 이 기계 층을 덧입힐 수 있습니다 — 표·산문은 그대로
> 두고 frontmatter·블록·스키마만 채웁니다.

## 로드맵 — 아직 없는 것

- **백엔드 관련 스킬은 아직 없습니다.** API 명세, 서버 아키텍처, 인프라 구성 같은 순수 백엔드
  설계 문서를 만드는 스킬은 향후 추가할 계획입니다. 현재 스킬 세트는 기획·화면·데이터 모델·
  디자인·정책 등 **프론트엔드/제품 정의 중심**으로 구성되어 있습니다(데이터 모델은 다루지만
  API 설계까지는 다루지 않습니다).
- 기여하고 싶으시다면 이슈로 제안해 주세요.

## 이 문서들은 어떻게 개발로 이어지나

완성된 문서 세트는 spec-kit 같은 Spec-Driven Development 도구에 그대로 넘길 수 있습니다.
`easyproduct-suite`가 만드는 **spec-kit 인계 안내문**은 spec-kit의 특정 명령어를 지정하지
않습니다 — 어떤 문서가 있고 어디에 있는지, 무엇이 SSOT인지만 정리해 넘기고, 실제 진행 방식은
spec-kit(및 그 위에서 동작하는 AI 에이전트)이 판단하도록 합니다.
